Library de procedure

Conceituação

Library de procedure é um conjunto de comandos agrupados de modo a efetuar funções afins, a serem incluídos na PROCEDURE DIVISION daqueles programas ou sub-rotinas que a utilizem como recurso.

Utilização

A library de procedure é utilizada na PROCEDURE DIVISION de sub-rotinas e programas host,on-line e batch.

Para um programa ou sub-rotina que utiliza uma library de procedure, deve-se declarar, na tela de cadastramento de recursos, o código da mesma com complemento "P", indicando que ela será incorporada na PROCEDURE DIVISION ou "M", indicando que, quando o programa ou sub-rotina for gerado, não será gerado um copy desta library, pois este copy já deve ter sido feito através de um dos comandos de módulo COBOL, EDIT ou EXCEC. Neste caso, o cadastramento da librarycomo recurso do programa ou sub-rotina será útil nas consultas a referências cruzadas de tal library.

Quando um programa ou sub-rotina declara uma libraryde procedure como recurso, com complemento "P", o gerador de programas do SADS gera um comando COPY na área de work, para incluir as variáveis criadas pelo SADS e um comando COPY, ao final da PROCEDURE DIVISION, contendo os comandos gerados.

As variáveis a serem incluídas na área de work do programa, referentes à library de procedure,são criadas automaticamente pelo próprio SADS, de acordo com a necessidade dos comandos definidos no módulo de cadastramento da library.

As libraries de procedure para utilização em hostsdevem ser construídas de forma diferente, pois elas não são incluídas somente ao final do host, mas em alguns pontos específicos indicados junto ao código da library,respectivamente, nos campos LIBRARY/USO, na definição do host. Estes pontos específicos são os seguintes:

"I" (Início) - Nos procedimentos iniciais do host, após o change/call das sub-rotinas e antes da abertura do banco.

"B" (Banco) - Nos procedimentos iniciais do host,após a abertura do banco.

"T" (Tela) - Nos procedimentos do host, após a rotina de leitura de tela menu.

"A" (Antes) - Nos procedimentos do host, antes do disparo dos aplicativos.

"D" (Depois) - Nos procedimentos do host, após o aplicativo devolver o controle para o host.

"E" (Envio) - Nos procedimentos do host, antes deste enviar a mensagem para o COMS.

"F" (Final) - Nos procedimentos finais do host.

Ao incorporar umalibrary de procedure em um destes locais específicos, deve-se atentar para a lógica da rotina na qual haverá a inserção, pois não podem haver labels que sejam incompatíveis com a library.

Fatores a serem observados na criação de library de procedure apenas definidas no repositório de dados e não geradas completamente pelo SADS:

As libraries que forem declaradas em programas aplicativos necessitam de labels de declaração de início e fim de parágrafos;

Caso o sistema não utilize sub-rotinas globais, os comandos CALL, definidos pelo usuário em libraries de procedure do host que efetuam chamadas às sub-rotinas declaradas como recursos deste, deverão estar definidos da seguinte forma:

CALL "sssSnn OF sssSnnL" USING sSnnmm-sssSnn.

A alteração em relação à forma convencional de chamada foi acrescido o caractere "L" ao final do código da sub-rotina, devido ao fato dos changes dos nomes das sub-rotinas terem sido efetuados no host da seguinte forma:

CHANGE ATTRIBUTE TITLE OF "sssSnnL" TO GBL-CHANGE.

Os comandos que compõem a library devem tratar variáveis que já estejam declaradas na área de trabalho dos programas que a utilizam. Pode-se utilizar libraries de work para definição destas variáveis;

A library deve executar procedimentos genéricos a um grupo de programas e/ou sub-rotinas para justificar sua criação como library, e não como módulo de programa.

Padronização

Para identificar uma library de procedure são necessários: o código do sistema + a constante de library de procedure "L" + um sequencial alfanumérico.

"sssIxx" onde:

"sss" é o código alfabético que identifica o sistema;

"I" é a constante de library de procedure e

"xx" é o sequencial alfanumérico da library de procedure.

Cadastramento

Para cadastrar uma libraryde procedure, é necessário que seu sistema esteja cadastrado no repositório de dados do SADS.

Roteiro

Para se definir uma library de procedure no repositório de dados do SADS e torná-la disponível para utilização, as seguintes funções devem ser executadas:

Definição de library de procedure;

Cadastramento de módulo de processamento de library de procedure;

Edição Cobol;

Geração de library de procedure;

Liberação de library de procedure para produção.

A seguir, cada uma dessas funções será explicada detalhadamente.

Definição de library de procedure

Para que uma library de procedure possa ser utilizada como recurso pelos programas e sub-rotinas,é necessário que ela esteja cadastrada no repositório do SADS.

Como a tela de definição de library de procedure é utilizada também para definir áreas de comunicação de sub-rotinas, registros do dataset de uso geral e áreas de trabalho, existem alguns campos que não devem ser preenchidos no momento da sua definição. Estes campos serão citados a seguir, na explicação dos campos da tela.

CAMPOS DA TELA

Elemento- Código da library de procedure no padrão SADS (sssLxx).

Geracao completa pelo SADS (S/N) - Controle de geração da library de procedure, diferenciando libraries de procedure criadas no CANDE das geradas pelo SADS.

Pode assumir os seguintes valores:

"S" - Indicador de library de procedure completamente gerada pelo SADS;

"N" - Indicador de library definida no repositório de dados do SADS e criada manualmente no CANDE.

Este campo será utilizado quando, no momento da compilação de um programa ou sub-rotina que declare a library de procedurecomo recurso, o campo "COMPILAR MESMO QUE OS RECURSOS ESTEJAM DESATUALIZADOS" for preenchido com o valor "NAO". Neste caso, se a library de procedure tiver sido definida com geração completa pelo SADS, será verificado o seu status de geração. Caso a library esteja com status "A gerar", o programa ou sub-rotina não poderá ser compilado.

Nome - Descrição resumida da library de procedure.

O nome tem o objetivo de identificar e explicar resumidamente a que se refere a library de procedure.

Registros - Este campo não deve ser preenchido para cadastramento da library de procedure.

Tipo - Este campo deve ser preenchido com "P" (Procedure).

Lib. de PROCEDURE p/ uso em:

Sub-rotina - Indicador de uso da libraryde procedure em sub-rotinas.

On-line - Indicador de uso da library de procedure em programas on-line.

Batch - Indicador de uso da library de procedure em programas batch.

Uma library de procedure só pode ter comandos compatíveis com o tipo de lógica onde for utilizada. Por exemplo, se uma library de procedure for utilizada tanto em programas on-line quanto em programas batch, esta library de procedure não deve utilizar comandos de módulo válidos apenas para programas on-line, evitando assim que ocorra erro quando ela for inserida em programas batch.Esta verificação será feita no cadastramento do módulo de processamento da library de procedure e no cadastramento dos recursos do programa que a utilizar.

Operacao - Operação efetuada sobre a definição da library de procedure.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de módulo de processamento de library de procedure

Uma library de procedure é composta por um único módulo, que será sempre o módulo linear 001.

Para cada um dos comandos de módulos a serem cadastrados, os campos a seguir são passíveis de serem informados. A obrigatoriedade ou não de se preencher um campo no momento de cadastramento de um determinado comando e a explicação detalhada de como preenchê-lo encontra-se no tópico "Comandos de módulo", no item "Programa", nesta seção.

CAMPOS DA TELA

SEQ. - Sequência do comando dentro do módulo.

COMANDO/TIPO - Código do comando de módulo que será utilizado pela library de procedure.

Caso necessário, deve-se fazer a complementação do comando, informando seu tipo.

Todos os comandos de módulos disponíveis, bem como seus tipos, estão descritos detalhadamente no tópico "Comandos de módulo", no item "Programa", nesta seção.

ORIGEM - Código do elemento que será utilizado pela library de procedure como origem (entrada) para o comando informado.

É necessário que o elemento utilizado na origem tenha sido cadastrado como recurso para o programa em que a library for incluída, exceto quando o elemento for o nome de um dos comandos DO, EDIT, EXCEC ou a área de  work do próprio programa, que deve ser indicada nos comandos de módulo pelo próprio código do programa (ssspxx).

É permitido também informar o literal "NENHUM" (para os comandos de módulo FIND e TAB), indicando que nenhuma movimentação deve ser efetuada para a chave de acesso do dataset ou tabela informado no campo DESTINO.

DESTINO - Código do elemento que será utilizado pela library de procedure como destino (saída) para o comando informado.

Para o comando de módulo "DO", poderá ser informado o código da tela na qual existam ocorrências. Desta forma, a rotina será executada o número de vezes correspondente ao número de ocorrências definido para a primeira tabela cadastrada na estrutura da tela informada.

Para o comando de módulo "DO", o campo DESTINO e N. VEZES são mutuamente exclusivos, somente um dos dois deve ser informado.

É necessário que o elemento utilizado no destino tenha sido cadastrado como recurso para o programa em que a library for incluída, exceto quando o elemento for a área de  work do próprio programa, que deve ser indicada nos comandos de módulo pelo próprio código do programa (ssspxx).

SET - Código da estrutura DMSII ( set, subset ou access) de acesso ao  dataset ou código da chave secundária de acesso a uma tabela, quando o  dataset ou tabela forem utilizados pelo comando de módulo.

Caso o set não seja informado para acesso a um dataset, será gerado então acesso direto ao dataset. Neste caso, o campo ATRIBUTO não deve ser preenchido. Esta situação é similar ao se informar o código do set e NENHUM no campo ATRIBUTO.

Este campo também pode indicar, para os comandos de módulo READ e START, caso informado "S", que o acesso ao arquivo será feito via chave secundária.

ATRIBUTO - Código de um atributo ou campo cadastrado no repositório de dados do SADS.

Pode assumir os seguintes valores:

"<Código do último atributo para acesso a uma estrutura DMSII>" - Para acesso parcial a uma estrutura DMSII (set, subset ou access), isto é, para acesso até este atributo limite. Utilizado nos comandos que utilizam dataset.

" " - Para que o acesso à estrutura DMSII seja com chave completa, isto é, com qualificação de todos os atributos da chave. Utilizado nos comandos que utilizam dataset.

"NENHUM" - Para que o acesso à estrutura DMSII seja feito sem considerar os campos da chave. Utilizado nos comandos que utilizam dataset.

"<Código do atributo de uma fórmula>"- Para identificação do atributo nos comandos de geração de fórmula.

INDICE - Indicador de indexação de ocorrências nas estruturas origem e destino.

Pode assumir os seguintes valores:

"O" - Quando se desejar indexar as ocorrências do elemento definido na origem;

"D" - Quando se desejar indexar as ocorrências do elemento definido no destino;

"A" - Quando se desejar indexar as ocorrências de ambos os elementos definidos na origem e no destino.

Caso este campo não seja informado, o SADS assumirá que não se deseja indexar as ocorrências tanto na origem como no destino.

O campo INDICE poderá ser utilizado em comandos de módulo de movimentação de dados - MVE, MOVE e MVEFOR - ou em comandos de módulo que efetuam a movimentação de dados implícita - FIND, LOCK, ALTERA etc.

O SADS irá gerar a movimentação dos itens correspondentes - de mesmo mnemônico - do elemento origem para o destino de acordo com a informação cadastrada no campo INDICE e de acordo com a localização do comando de módulo dentro do programa, qual seja:

  1. Fora de qualquer loop;
  2. Interno a um loop normalmente gerado pelo SADS, cujo número de vezes para execução é contabilizado pelo índice IND-GER;
  3. Interno a um loop definido pelo usuário através do comando de módulo DO, cujo número de vezes para execução é contabilizado pelo índice "IND-nnnmm" - onde "nnn" é o número do módulo e "mm", o número do DO;
  4. Interno a ambos os loops, do gerador do SADS e do usuário.

Sendo assim, dependendo da localização na library de procedure onde o comando será gerado, o SADS determinará quais itens serão movidos, quais sejam:

  • Apenas os itens que não são indexados por nenhum índice (S);
  • Apenas os itens pertencentes a tabela unidimensional, que devem ser indexados por um índice (I1);
  • Apenas os itens pertencentes a tabelas bidimensionais, que devem ser indexados por dois índices (I2).

Uma tabela a fim de que possa ser percebido, de acordo com o INDICE e localização do comando, quais itens serão movidos do elemento origem para o elemento destino, se encontra no item "Programa", neste seção.

N.VEZES OU N.FORMULA - Número de vezes que será executado o perform a ser gerado pelo comando de módulo DO ou número da fórmula do atributo a ser calculada.

Este campo deverá ser informado somente para o comando de módulo DO e para comandos de módulo que utilizam fórmula.

Na geração do fonte Cobol, o comando DO será transformado em comando perform.

Para o comando de módulo DO, pode assumir os seguintes valores:

"99" - Para indicar que o comando gerado será um Perform until eof:

PERFORM nnnmm-00-<nome com. DO>THRU nnnmm-99-EXIT

UNTIL EOF-nnnmm EQUAL HIGH-VALUES;

sendo que, no código da chave de fim de arquivo "EOF-nnnmm", "nnn" é o número do módulo e "mm", o número do DO.

Neste caso, o campo "EOF-nnnmm" será composto no tratamento de exceção notfound do comando de acesso cadastrado entre o comando DO e o END, se houver.

De "02" a "98" - Para indicar que o comando gerado será um Perform varying de acordo com o índice indicado no campo N.VEZES ou N.FORMULA:

PERFORM nnnnn-00-<nome com. DO > THRU nnnnn-99-EXIT

VARYING IND-nnnmm FROM 1 BY 1 UNTIL

IND-nnnmm GREATER "numero de vezes";

sendo que, no código do índice "IND-nnnmm", "nnn" é o número do módulo e "mm", o número do DO.

Caso este campo não seja informado no comando de módulo DO, o SADS irá gerar um perform simples da seguinte forma:

PERFORM nnnnn-00-<nome com. DO> THRU nnnnn-99-EXIT;

Para comandos que utilizam fórmula, pode assumir os seguintes valores:

De "01" a "99" - Para indicar o número da fórmula do atributo que será utilizada pelo programa.

Linha livre - Linha para detalhamento do comando informado no campo COMANDO/TIPO.

Os comandos de módulo que utilizam a linha livre são: COB (COBOL), LAB (LABEL), EXCEPT, COPY, COMENT, MSG e MSGIND.

Na linha livre devem ser utilizados prefixos, sufixos, pontos e outras referências específicas para o comando em questão, pois esta linha será gerada na library tal como foi cadastrada.

Para os comandos de módulo de edição - COB, EDIT e EXCEC - foi criada a constante "SADSM", a fim de facilitar na determinação dos desvios (GO TO), índices e chaves de fim (EOF).

Para se utilizar a constante, ao invés de se definir o número do parágrafo para os desvios, índices ou chaves de fim (EOF), deve-se definir a constante "SADSM". Deste modo, na geração da library de procedure, a constante será substituída pelo número do parágrafo correspondente.

OPERACAO - Operação a ser efetuada sobre o cadastramento do módulo.

Pode assumir os seguintes valores:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

CONTINUA (S/N) - Indicador de continuidade de operação.

Pode assumir os seguintes valores:

"S" - Para que seja exibida a próxima página de operação sobre o módulo;

"N" - Para encerrar a operação sobre o módulo.

Neste caso, o campo PROXIMA PAGINA deve estar preenchido com "99".

PROXIMA PAGINA - Número da próxima página a ser exibida ou "99" para encerrar a operação sobre o módulo.

O cadastramento de módulos segue a lógica de atualização paginada e possui todas as características deste tipo de transação, conforme descrito na Seção I, "Conceitos preliminares para a utilização do SADS, item "Paginador de entrada de dados".

A seguir, é apresentado um exemplo de cadastramento de módulo de processamento de library de procedure:

Edição Cobol

O SADS possui uma função para edição de comandos quaisquer em Cobol e comandos para tratamento de exceção.

Através desta função, torna-se possível ao usuário definir uma sequência de comandos a serem inseridos, tal e qual cadastrados, no fonte da library de procedure.

Podem existir determinados comandos da library de procedure que não são gerados da maneira como se deseja pelos comandos de módulo oferecidos pelo SADS. Podem também existir tratamentos de exceção gerados pelo SADS que não atendem à necessidade específica da aplicação. Desta forma, torna-se necessário incluir diretamente no fonte da library de procedure comandos em Cobol que atendam a essa necessidade ou que substituam o tratamento de exceção normalmente gerado.

A inclusão de comandos em Cobol no fonte da library de procedure pode ser feita de duas formas:

Cadastrando cada comando em Cobol a ser inserido na library de procedure através do comando de módulo COB, ou

Cadastrando todos os comandos em Cobol através de um único comando de módulo, que é o EDIT.

Já a substituição do tratamento padrão de exceção do SADS pode ser feita também de duas formas:

Substituindo todo o parágrafo a ser gerado, incluindo nele, assim, o novo tratamento desejado, ou

O mais conveniente, cadastrando o comando de módulo EXCEC após o comando que executa o acesso ao dataset ou ao arquivo convencional.

Através do cadastramento no módulo dos comandos EDIT ou EXCEC, que são identificados por um nome, define-se o local da library de procedure onde será inserido o trecho com comandos Cobol ou qual será o comando de acesso anterior cujo tratamento de exceção será substituído, respectivamente.

Após a finalização do cadastramento do módulo dos comandos EDIT ou EXCEC, o SADS apresenta uma tela para cadastramento do comando, dispondo de 21 linhas para codificação da sequência de comandos em Cobol.

Caso necessário, outras telas poderão ser encadeadas para continuação da codificação.

Para edição de comandos em Cobol, é necessário que a definição da library de procedure e os recursos do programa que terá a library de procedure inserida estejam cadastrados no repositório de dados do SADS. Além disto, no módulo, deve ter sido definido algum comando EDIT ou EXCEC.

O uso dos comandos de módulos EDIT e EXCEC estão descritos detalhadamente no tópico "Cadastramento de edição Cobol", no item "Programa", nesta seção.

Geração de library de procedure

O SADS possui uma função encarregada de gerar, no usercode informado, um arquivo contendo o fonte Cobol da library de procedure especificada. A geração deste fonte é baseada nas informações cadastradas para a library de procedure no repositório de dados do SADS.

Qualquer alteração no cadastramento da library de procedure implica em uma nova geração da mesma , sem que seja necessário alterá-la via Cande.

A geração da library de procedure é utilizada quando se deseja criar o fonte de uma library de procedure a partir de seu cadastramento no SADS ou quando se deseja atualizar o fonte da library de procedure existente no usercode, a partir das alterações efetuadas em seu cadastramento.

As libraries de procedure cadastradas em versões anteriores à 3.3 não podem ser geradas pelo SADS. Isto porque, naquelas versões, não era possível o cadastramento de módulos de libraries de procedure e estas eram codificadas totalmente no Cande. Caso a geração fosse permitida, esta codificação efetuada no Cande seria perdida. Porém, mesmo para estas libraries de procedure cadastradas anteriormente à versão 3.3, caso haja alguma manutenção em seus cadastramentos, o SADS passará a permitir as suas gerações.

Através da função de geração do SADS, é possível gerar todas as library de procedure do sistema completamente, sem a necessidade de complementá-las ou alterá-las via Cande.

No momento da geração, são incorporadas advertências ao fonte da library de procedure sempre quando o SADS percebe alguma incoerência, incompatibilidade ou falta de completeza no fonte da library de procedure a ser gerada. Ao final do mesmo, é informado o número de advertências incluídas.

Assim sendo, é desejável que estas advertências sejam sempre verificadas, a fim de evitar possíveis erros de lógica e processamento. As advertências são incluídas quando são detectadas, por exemplo, as seguintes situações:

Chaves de acesso a estruturas DMS não compostas;

Chaves de acesso a estruturas DMS compostas parcialmente;

Não movimentação de atributos entre estruturas, no caso de comando de módulo "MOVE";

Não movimentação de atributos da tela menu para a área conversacional, no caso de lógica "AP".

Para gerar uma library de procedure, basta transmitir os argumentos do menu de operação GERLIBPRO, cujos campos devem ser preenchidos conforme indicado a seguir.

CAMPOS DA TELA

Library - Código da library de procedure que se deseja gerar, no padrão SADS (sssLxx).

Ambiente - Ambiente no qual a library de procedure deverá ser gerada.

Podem ser informados os seguintes ambientes:

"T" - Teste;

"P" - Produção.

Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Usercode - Usercode de análise.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

São válidos para esta geração os usercodes de análise definidos na tabela SBPM5002. A definição dos usercodes de análise está descrita na Seção V, "Parametrizações do SADS".

Gerar mesmo que ja exista o fonte - Indicador de sobreposição do fonte da library de procedure.

Pode assumir os seguintes valores:

"S" - Para gerar o fonte independentemente da existência de outro de mesmo código no usercode informado;

"N" - Para gerar o fonte somente se não existir uma library de procedure com o mesmo código no usercode informado.

Liberação de library de procedure para produção

A liberação da library de procedure para produção consiste em gerar uma cópia da mesma no usercode de produção.

Só é possível liberar libraries que tenham sido criadas no ambiente de teste.

A liberação da library de procedure para produção não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos da tela "LIBLIBPRO", cujos campos estão explicados a seguir:

CAMPOS DA TELA

Library de Procedure - Código da library de procedure no padrão SADS (sssLxx).

Usercode - Usercode de análise.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Para usercodes de análise, são definidos os ambientes ( usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercode. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS", neste manual.

Se na definição do usercode de análise, na tabela SBPM5002, estiver indicado que as libraries de produção ficarão residentes no usercode de objetos em produção, ao executar suas liberações, aquelas que estão em teste serão mantidas e versões idênticas serão criadas para o usercode de produção. Se as libraries de produção ficarem residentes sem  usercode, ao executar suas liberações, aquelas que estão em teste serão removidas e outras idênticas serão criadas em *, ou seja, sem usercode, para atender aos dois ambientes (teste e produção).

Funções de apoio

Específicas de libraries de procedure

O SADS oferece as seguintes funções de apoio que são específicas do elemento library de procedure:

Duplicação de library de procedure;

Ressequenciamento de módulo;

Exportação e importação de edit;

Consulta fonte de library de procedure;

-Impressão de cadastramento de library de procedure.

A seguir, será explicada detalhadamente cada uma dessas funções.

Duplicação de library de procedure

Para duplicar library de procedure é necessário que:

A library de procedure origem esteja cadastrada no repositório de dados do SADS;

A library de procedure destino não esteja cadastrada.

Para duplicar library de procedure, basta transmitir os argumentos do menu de operação "DUPLIBPRO", cujos campos devem ser preenchidos conforme explicação a seguir:

CAMPOS DA TELA

Library origem - Código da library de procedure a ser duplicada, no padrão SADS (sssLxx).

Library destino - Código da library de procedure que irá assumir o novo cadastramento, no padrão SADS (sssLxx).

A transação de duplicação de library de procedure será executada imediatamente, sem apresentação de tela de processamento e, ao seu término, será enviada uma mensagem confirmando a duplicação.

Ressequenciamento de módulo

O SADS possibilita que os comandos definidos no cadastramento de módulo sejam ressequenciados, ou seja, tenham alteradas suas sequências dentro do módulo, de modo que todas elas tenham um número múltiplo de 10 (dez).

Para utilizar esta operação, o módulo a ser ressequenciado já deve ter sido cadastrado no repositório de dados do SADS.

Para ressequenciar um módulo, basta transmitir o argumento do menu de operação RESMODLIB, cujo campo deve ser preenchido conforme explicação a seguir:

CAMPOS DA TELA

Library - Código da library de procedure que se deseja ressequenciar, no padrão SADS (sssLxx).

Esta transação será executada imediatamente, sem apresentação de tela de processamento e, ao seu término, será enviada uma mensagem confirmando o ressequenciamento.

Exportação e importação de edit

O SADS possibilita exportar comandos de módulo EDIT para o Cande e importar arquivos de comandos em Cobol para que se tornem comandos EDIT. Ao efetuar uma exportação de um EDIT no Cande, será gerado um arquivo de nome "EDIT/ sssennmmm/xxxxxxxxx", onde:

"sssenn" é o código da library de procedure;

"mmm" é o número do módulo ao qual o EDIT pertence, no caso de library de procedure é 001;

"xxxxxxxxx" é o nome do EDIT.

Este arquivo pode então ser alterado e depois importado pelo SADS, permitindo, com estas operações, que sejam utilizadas as facilidades de edição do Cande.

Pode-se também criar o arquivo diretamente no Cande, desde que ele tenha este padrão de nome, para que ele seja importado pelo SADS.

Sempre quando for efetuada uma importação de comando EDIT do Cande, a library de procedure que teve o EDIT importado ficará marcada e o programa que a utilizar só poderá ser compilado caso ela seja regerada. Isto para garantir que os comandos do EDIT sejam devidamente gerados na library de procedure após sua importação para o SADS.

Para exportar ou importar um comando EDIT, basta transmitir os argumentos do menu de operação "TRAEDILIB", cujos campos devem ser preenchidos conforme explicação a seguir:

CAMPOS DA TELA

Operacao - Operação a ser efetuada sobre o comando EDIT.

Pode assumir os seguintes valores:

"IMP" - Importação do arquivo no Cande para o comando EDIT do SADS;

"EXP" - Exportação do comando EDIT do SADS para um arquivo no Cande;

Library - Código da library de procedure a qual o EDIT pertence.

Edit - Nome do comando EDIT.

Usercode - Usercode onde reside o arquivo a ser importado/exportado.

O usercode a ser informado deve estar habilitado ao usuário como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

EXPORTAR MESMO QUE JA EXISTA O EDIT? (S/N) - Indicador de sobreposição de arquivos no Cande, na operação de exportação.

Pode assumir os seguintes valores:

"S" - Para que seja gerado o arquivo no Cande contendo os comandos do EDIT, mesmo que já exista um arquivo neste usercode com mesmo nome. O arquivo anterior será sobreposto.

"N" - Para que seja não seja gerado o arquivo no Cande contendo os comandos do EDIT, caso já exista um arquivo neste usercode com mesmo nome.

Consulta fonte de library de procedure

O SADS permite, sem precisar ativar a janela Cande, consultar o fonte de libraries de procedure geradas. A consulta ao fonte pode ser exibida a partir da primeira linha do fonte ou a partir de um label, que deve ser informado com o tamanho máximo de 17 caracteres. Desta forma, a consulta será exibida a partir da primeira linha da library de procedure que começar com o conjunto de caracteres informado.

Para consultar o fonte de uma library de procedure, é necessário que a mesma já tenha sido gerada no usercode informado.

Para consultar o fonte de uma library de procedure, basta transmitir os argumentos do menu de operação "CONFONLIB", cujos campos devem ser preenchidos conforme explicação a seguir:

CAMPOS DA TELA

Programa/library - Código da library de procedure cujo fonte será consultado, no padrão SADS (sssLxx).

Usercode - Usercode onde reside o fonte da library de procedure.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Ambiente - Ambiente no qual a library de procedure foi gerada:

Podem ser informados os seguintes ambientes:

"T" - Teste;

"P" - Produção.

Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Label/sequencia - Label ou sequência a partir do qual será apresentado o fonte da library de procedure;

A consulta ao fonte da library de procedure é do tipo paginada, contendo as informações de cabeçalho, como o código e o nome da library de procedure, e, a seguir, o seu fonte.

Em cada linha do fonte é exibido o conteúdo dos seis primeiros bytes do MARKID (colunas 73 a 78).

No fonte gerado, o MARKID irá conter a informação "MOD001" ou "<<Nome da edição Cobol/exceção>>", caso a linha correspondente tenha sido gerada a partir de edição Cobol - comando EDIT - ou através de um tratamento de exceção - comando EXEC.

Impressão de cadastramento de library de procedure

O SADS oferece uma função de geração de um relatório impresso com todas as informações do cadastramento de uma library de procedure: definição e cadastramento de módulos.

Essa operação tem por objetivo facilitar a manutenção do cadastramento de uma library de procedure, através da visualização em papel impresso de todas as informações constantes neste cadastramento, para posterior alteração.

Para imprimir o cadastramento de uma library de procedure, basta transmitir os argumentos do menu de operação "IMPCADLIB", cujos campos devem ser preenchidos conforme explicação a seguir:

CAMPOS DA TELA

Library - Código da library de procedure

Pode assumir os seguintes valores:

"sss" - Para imprimir o cadastramento de todas as libraries de procedure do sistema;

"sssLxx" - Para imprimir o cadastramento de library de procedure específica.

Impressora - Código da impressora.

Pode assumir os seguintes valores:

"<Código da impressora remota>" - Para imprimir o manual na impressora remota informada (esta impressora deve estar cadastrada na rede de terminais);

"PRINTER" - Para imprimir o manual na impressora central, no formulário de 132 posições;

"PRINxx" - Para imprimir o manual na impressora central em formulário especial, onde "xx" é um formid definido na instalação.

As impressoras remotas podem ser cadastradas, com um formid correspondente, na tabela SBPM5026. Todas as vezes que for disparada uma impressão pelo SADS, esta tabela é consultada, e caso a impressora esteja cadastrada, o relatório será impresso com o seu formid correspondente. A tabela SBPM5026 está descrita na Seção V, "Parametrizações do SADS", neste manual.

Caso este campo não seja informado, será assumida a impressora default cadastrada para o usuário no Sistema de Segurança. Caso esta não exista cadastrada, será então obrigatória a informação do código da impressora no momento do disparo da impressão.

A impressão será executada imediatamente, sem apresentação de tela de processamento e, ao seu término, será enviada uma mensagem confirmando a impressão.
