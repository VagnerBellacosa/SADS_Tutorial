Área de trabalho

Conceituação

Área de trabalho é um espaço reservado para declaração de campos e atributos a serem incluídos na WORKING-STORAGE SECTION daqueles programas e sub-rotinas que a utilizem como recurso.

Uma área de trabalho é, portanto, uma estrutura composta de atributos e/ou campos, como mostra o exemplo a seguir:

01 WSS-DATA

05 WSS-DIA PIC 9(02).

05 WSS-MES PIC 9(02).

05 WSS-ANO PIC 9(02).

01 WSS-TABELA-EMPRESA OCCURS 3 TIMES.

05 WSS-EMPCOD PIC X(03).

05 WSS-EMPNOME PIC X(20).

A definição de uma área de trabalho dá origem a uma library que, em tempo de compilação, será incorporada aos programas e sub-rotinas.

Normalmente são declarados na área de trabalho aqueles atributos ou campos que serão utilizados em vários programas e/ou sub-rotinas. Isto evita que os mesmos tenham de ser declarados individualmente em cada um destes programas e sub-rotinas.

Padronização

Para identificar uma área de trabalho são necessários: o código do sistema + a constante de área de trabalho "L" + indicador de 2 (dois) caracteres - sendo o primeiro alfanumérico e o segundo, numérico.

Utilização

A área de trabalho é utilizada na WORKING-STORAGE SECTION de sub-rotinas e programas host, on-line e batch.

Nos programas on-line, as áreas de trabalho podem assumir outras características, ou seja, podem ser utilizadas como área conversacional ou área global.

A área conversacional é uma estrutura de atributos e/ou campos, cujo objetivo é armazenar e transferir dados entre transações efetuadas em uma mesma estação.

Área global de comunicação, por sua vez, é uma área comum entre os programas on-line e o programa host que os gerencia, podendo ser utilizada para transferir informações entre suas transações.

Portanto, a diferença básica entre estas duas áreas é que, enquanto as informações armazenadas na área conversacional ficam disponíveis para todas as transações efetuadas na mesma estação, as armazenadas na área global ficam disponíveis para as transações gerenciadas pelo mesmo host, enquanto este estiver ativo.

Cumpre ressaltar que, estas áreas estarão disponíveis para utilização desde que o aplicativo tenha sido preparado para tal.

Para utilização de uma área conversacional em um aplicativo, além da sua definição e geração, deve-se atentar para a execução dos seguintes passos:

  1. Definir a área de trabalho no repositório de dados do SADS com tipo igual a "C" (conversacional);
  2. Definir o tamanho da área conversacional com um acréscimo de, no mínimo, 14 bytes no seu tamanho. Estes 14 bytes a mais, necessários na área conversacional, são utilizados pelo gerador de programas do SADS para armazenar o código da transação e a data/hora do header do registro processado;
  3. Indicar, nos menus do aplicativo, as transações que irão utilizar área conversacional. Para isto, basta assinalar, nas respectivas transações, o campo ACV com "S". Sempre que forem solicitadas transações que não indicam utilização de área conversacional, o host a inicializará com nulls (zero binário);
  4. Definir a área como recurso dos programas que irão utilizá-la, com complemento igual a "C". Este procedimento determina que a área de trabalho será utilizada como área conversacional nos mesmos;
  5. Relacionar a área com o(s) host(s) do aplicativo, através do campo LIBRARY PARA AREA CONVERSACIONAL, na tela de definição do(s) mesmo(s). Este procedimento é necessário para o processamento de informações da área conversacional em alguma library de procedure especificada neste(s) host(s).

No momento da geração da library da área conversacional, caso o tamanho total definido para a mesma for menor que o definido para o host, será gerado um FILLER complementar, no final da library, com a diferença dos tamanhos dos mesmos.

Após a execução dos procedimentos citados, mais geração, compilação e binder dos elementos envolvidos - libraries e programas -, a área conversacional estará disponível para ser utilizada.

Quando uma estação é desconectada, o conteúdo da área conversacional é excluído.

Para utilização de uma área de trabalho global nos aplicativos, além da sua definição e geração, deve-se atentar para a execução dos seguintes passos:

  1. Definir a área de trabalho no repositório de dados do SADS com tipo igual a "G" (global);
  2. Relacionar a área ao host, através do campo LIBRARIES / USO, na tela de definição do mesmo; informando seu código no padrão SADS e sua utilização igual a "W" (work);
  3. Definir a área como recurso dos programas que irão utilizá-la, com complemento "W". Este procedimento determina que a área de trabalho será utilizada como área de WORKING-STORAGE SECTION dos mesmos.

Após a execução dos procedimentos citados, mais geração, compilação e binder dos elementos envolvidos, a área de trabalho estará disponível para ser utilizada como área global.

Uma área de trabalho (tipo = "E", "C" ou "G"), desde que não tenha sido definida a sua estrutura, pode ser transformada em uma library de procedure (tipo = "P"). Também, uma library de procedure, desde que não tenha módulos cadastrados, pode ser transformada em uma library de área de trabalho (tipo = "E", "C" ou "G").

As libraries de área de trabalho são agregadas aos programas através do comando COPY.

Exemplo:

COPY "SX/Y/PL/01".

Este comando será gerado no host desta forma, mesmo quando as libraries forem utilizadas em programas on-line como área global. Porém, nos programas aplicativos, o comando gerado será o seguinte:

COPY "ss/s/PL/xn FROM 100100 THRU 199000.

Exemplo:

01 YL0101-SXYL01 GLOBAL.

  COPY "SX/Y/PL/01" FROM 100100 THRU 199000.

Cadastramento

Para efetuar o cadastramento da área de trabalho, é necessário que o sistema, para o qual ela será definida, esteja cadastrado no repositório de dados do SADS.

Roteiro

As funções utilizadas para definir e tornar disponível uma área de trabalho são as seguintes:

 Definição de área de trabalho;

 Cadastramento de estrutura de registro de área de trabalho;

 Geração de library de área de trabalho;

 Liberação de library de área de trabalho para produção.

Definição de área de trabalho

Utilizando a mesma tela de definição de área de trabalho, pode-se também registros de dataset de uso geral e libraries de procedure. Por isso, existem nela alguns campos que não devem ser preenchidos no momento da definição da área de trabalho.

CAMPOS DA TELA

 Elemento - Código da área de trabalho no padrão SADS (sssLxn).

 Geração completa pelo SADS (S/N) - Controle de geração da área de trabalho, diferenciando as áreas de trabalho criadas no CANDE das geradas pelo SADS.

Pode assumir os seguintes valores:

"S" - Indicador de área de trabalho gerada pelo SADS;

"N" - Indicador de área de trabalho definida no repositório de dados do SADS e criada manualmente no CANDE.

Este campo será utilizado quando uma compilação de programa ou sub-rotina que declare a área de trabalho como recurso, estiver com o campo COMPILAR MESMO QUE OS RECURSOS NÃO ESTEJAM ATUALIZADOS preenchido com o valor "NAO". Neste caso, se a área de trabalho contiver o valor "S", será verificado o seu status de geração.

 Nome - Descrição resumida da área de trabalho.

O nome da área de trabalho tem o objetivo de identificar e explicar resumidamente a que se refere a área de trabalho.

 Registros - Quantidade de registros que a área de trabalho possui.

Cada registro da área de trabalho é um conjunto de atributos e/ou campos subordinados a um único nível 01 do Cobol. As libraries de área conversacional são consideradas exceções, já que são geradas a partir do nível 05.

Este campo não deve ser informado para área de trabalho.

 Tipo - Tipo de utilização da área de trabalho.

Os seguintes tipos podem ser informados:

"E" - Para utilização em WORKING-STORAGE SECTION de programas e sub-rotinas, ou para registro do dataset de uso geral SBGB02;

"C" - Para utilização da área de trabalho como área conversacional;

"G" - Para utilização como área global em programas on-line;

"P" - Para utilização em library de procedure.

Uma área de trabalho (tipo = "E", "C" ou "G"), desde que não tenha sido definida a sua estrutura, pode ser transformada em uma library de procedure (tipo = "P"). Também, uma library de procedure, desde que não tenha módulos cadastrados, pode ser transformada em uma library de área de trabalho (tipo = "E", "C" ou "G").

 Lib. de PROCEDURE p/ uso em:

Sub-rotina (S/N) - Este campo não deve ser informado para área de trabalho.

On-line (S/N) - Este campo não deve ser informado para área de trabalho.

Batch (S/N) - Este campo não deve ser informado para área de trabalho.

 Operacao - Operação efetuada sobre a definição da área de trabalho.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de estrutura de registro de área de trabalho

Uma área de trabalho pode conter um ou mais registros.

Cada registro, por sua vez, é um conjunto de atributos e/ou campos, que pode ou não redefinir um outro da mesma área de trabalho.

Para cadastrar a estrutura de um registro da área de trabalho, é necessário que a mesma já esteja definida no repositório de dados do SADS.

O registro "00" pode ser definido em uma área de trabalho, porém não será permitido o cadastramento de sua estrutura.

Nos grupos de campos que ocorrem várias vezes na tela, deverão ser definidos os atributos que fazem parte da estrutura do registro da área de trabalho e, se houver, suas particularidades.

No contexto da área de trabalho, entenda-se atributo como sendo atributo propriamente dito, campo ou sinônimo.

CAMPOS DA TELA

 ELEMENTO - Código da área de trabalho no padrão SADS (sssLxn);

 REGISTRO - Número do registro a ser cadastrado;

Ao incluir uma estrutura, este número não precisa ser informado.

 REDEFINE REGISTRO - Número do registro a ser redefinido.

Se o campo REDEFINE REGISTRO não for informado, na geração da library, os registros serão gerados com layouts independentes.

 NOME - Descrição resumida do registro em cadastramento.

 SEQ - Número sequencial para posicionamento do atributo dentro da estrutura.

O primeiro número apresentado no campo SEQ é igual a "0010" e os demais acrescidos de 10.

Para incluir um atributo entre dois outros já existentes, é necessário preencher uma linha que estiver em branco e alterar o número sequencial desta linha para um valor entre os números já informados. O SADS reorganizará automaticamente a estrutura. Na próxima operação, os atributos já estarão na nova ordem.

 ATRIBUTO - Mnemônico de um atributo ou campo.

 SUF - Sufixo para diferenciar o atributo, caso ele apareça mais de uma vez na mesma estrutura.

 NIV - Nível de estruturação do atributo, análogo ao nível do Cobol.

O nível inicial é o 10. Se for necessário criar agrupamentos de atributos, deve-se utilizar múltiplos de 5 (15, 20 e assim por diante), de forma semelhante ao Cobol. O nível 77 pode ser utilizado, análogo ao nível 77 do Cobol.

NO SADS não é permitida a utilização do nível 88.

 NDEF - Nível de definição do atributo.

Os valores possíveis são os seguintes:

"S" (Sistema) - Para atributo definido no sistema informado no campo SIS ou para atributo pertencente ao mesmo sistema da área de trabalho;

"G" (Geral) - Para atributo pré-definido no repositório de dados do SADS que terá utilização genérica pelos sistemas aplicativos;

"P" (Particular) - Para atributo utilizado unicamente neste registro da área de trabalho e, portanto, não cadastrado no repositório de dados do SADS.

Neste caso, o formato, tamanho, indicador de sinal e decimais do atributo têm de ser obrigatoriamente informados.

Não pode ser informado NDEF = "P" caso tenha sido parametrizado na tabela SBPM5000 que não é aceitável atributos/campos particulares nas estruturas SADS.

Maiores detalhes a respeito desta parametrização, deve-se consultar a Seção V, "Parametrizações do SADS", item "Atributo", subitem "Redefinição em estruturas".

"R" (Redefinição) - Para atributo que redefine o anterior.

O atributo com NDEF = "R" será interpretado como item grupo e, portanto, para que a redefinição seja válida, ele deve conter itens elementares.

Caso NDEF não seja informado, o gerador de libraries do SADS interpretará o atributo como sendo um item grupo.

 SIS - Código do sistema para o qual o atributo foi definido.

O campo SIS deve ser indicado somente quando o campo NDEF estiver preenchido com "S" e o atributo pertencer a um sistema diferente do sistema da área de trabalho.

 COMP - Indicador de atributo compactado ou binário.

Os valores possíveis são os seguintes:

"S" (Compactado) - Para atributos numéricos que devam ser gerados com o "COMP" do Cobol, deve-se informar "S";

"B" (Binary) - Para atributos numéricos que devam ser gerados com formato binary.

 FORM - Formato do atributo.

Os valores possíveis são os seguintes:

"A" (Alfabético);

"N" (Numérico);

"D" (Display ou alfanumérico);

"R" (Real);

"B" (Boolean).

Este campo não pode ser informado caso tenha sido parametrizado na tabela SBPM5000 que os atributos/campos não podem ter seus formatos redefinidos nas estruturas nas quais são incluídos.

Maiores detalhes a respeito desta parametrização, deve-se consultar a Seção V, "Parametrizações do SADS", item "Atributo", subitem "Redefinição em estruturas".

 TAM - Tamanho do atributo.

Este campo não pode ser informado caso tenha sido parametrizado na tabela SBPM5000 que os atributos/campos não podem ter seus tamanhos redefinidos nas estruturas nas quais são incluídos.

Maiores detalhes a respeito desta parametrização, deve-se consultar a Seção V, "Parametrizações do SADS", item "Atributo", subitem "Redefinição em estruturas".

 SIN - Indicador de atributo sinalizado.

Os valores possíveis são os seguintes:

"S" - Para atributo númerico sinalizado;

"N" - Para atributo não sinalizado;

"J" - Para atributo a ser definido com a cláusula "JUST RIGHT".

 DEC - Quantidade de casas decimais que o atributo numérico possui.

Os campos FORM, TAM, SIN e DEC devem ser informados para atributos/campos particulares ou para asssumir alguma destas características de forma diferente daquela que se encontra em sua definição.

 CLAS - Este campo não é utilizado para o cadastramento da estrutura da área de trabalho.

 OCCURS - Semelhante à cláusula "OCCURS" das estruturas do Cobol.

Pode ser utilizado somente para definir itens de grupo com ocorrências.

 CONTINUA(S/N) - Indicador de continuidade de operação.

As possibilidades para o preenchimento deste campo são as seguintes:

"S" - Para prosseguir no cadastramento da mesma estrutura;

"N" - Para encerrar o cadastramento da estrutura.

 OPERACAO - Operação efetuada sobre a estrutura da área de trabalho.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Geração de library de área de trabalho

Na geração de library de área de trabalho, é criado um arquivo fonte Cobol, com o layout da área de trabalho, que será incorporado aos programas aplicativos que a utilizarem.

A library da área de trabalho é gerada a partir da estrutura de dados de todos os registros da mesma.

Portanto, para gerar uma library, é necessário que a área de trabalho e as estruturas de seus registros estejam cadastradas no repositório de dados do SADS.

A library contém os atributos e campos que compõem a área de trabalho. A seguir, são apresentados dois exemplos de library de área de trabalho.

Para a área "SXYL01 - Área conversacional do sistema SXY", é gerada a library:

05 YL0101-SXYL01.

10 YL0101-EMPCOD PIC 9(003).
10 YL0101-FILCOD PIC 9(004).
10 Yl0101-FILEND PIC X(040).

05 YL0102-SXYL01 REDEFINE YL0101-SXYL01.

10 YL0102-LOGCOD PIC 9(003) COMP.
10 YL0102-ARTCOD PIC 9(003) COMP.
10 YL0102-LOJCGC PIC 9(014) COMP.
10 FILLER PIC X(037).

Para a área "SXYL05 - Área de trabalho do sistema SXY", a library gerada é:

01 YL0501-SXYL05.

10 YL0501-EMPCOD PIC 9(003).
10 YL0501-FILCOD PIC 9(004).
10 Yl0501-FILEND PIC X(040).

01 YL0502-SXYL05 REDEFINE YL0501-SXYL05.

10 YL0502-LOJCOD PIC 9(003) COMP.
10 YL0502-ARTCOD PIC 9(003) COMP.
10 YL0502-LOJCGC PIC 9(014) COMP.
10 FILLER PIC X(037)

Os prefixos dos campos de library de área de trabalho têm a seguinte formação:

Terceira letra do sistema ("Y", no exemplo) + a letra "L" (que identifica uma área de trabalho) + o sequencial da área ("01" e "05", no exemplo) + o número do registro ("01" e "02", no exemplo) + "-" (constante) + o mnemônico do atributo ou campo.

Deve-se atentar para o fato de que as libraries de área conversacional são geradas a partir do nível "05" e as demais, a partir do nível "01".

A library da área de trabalho é gerada no usercode definido para o ambiente escolhido, sendo possível gerá-la sem usercode (em *) para o ambiente de produção. A especificação de onde a library deverá residir encontra-se descrita na tabela SBPM5002, que se encontra descrita na seção V, "Parametrizações do SADS", neste manual.

A geração de library de área de trabalho não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu de operação "GERLIBARE", cujos campos estão explicados a seguir:

CAMPOS DA TELA

 Area de Trabalho - Código da área de trabalho no padrão SADS (sssLxn).

 Ambiente - Ambiente para o qual a library será gerada:

"T" (Teste);

"P" (Produção).

Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Quando uma library é gerada em ambiente de teste, o acesso à mesma será feito somente nas compilações para este ambiente. Porém, se a geração for executada para produção, ficando residente sem usercode (dependendo do cadastramento do seu usercode de análise na tabela SBPM5002), o acesso à mesma será feito nas compilações para ambos os ambientes.

 Usercode - Usercode de análise.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Para usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS", neste manual.

Liberação de library de área de trabalho para produção

A liberação de library de área de trabalho para produção consiste na criação de uma cópia da library no usercode de produção e, dependendo da parametrização do usercode de análise na tabela SBPM5002, na exclusão da cópia do usercode de teste.

Só é possível liberar libraries que tenham sido geradas no ambiente de teste.

A liberação de library de área de trabalho para produção não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos da tela "LIBLIBARE", cujos campos estão explicados a seguir:

CAMPOS DA TELA

 Area de Trabalho - Código da área de trabalho no padrão SADS (sssLxn).

 Usercode - Usercode de análise.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Para usercodes de análise são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita neste manual, na Seção V, "Parametrizações do SADS", . neste manual.

Se na definição do seu usercode de análise, na tabela SBPM5002, estiver indicado que as libraries de produção ficarão residentes no usercode de objetos em produção, ao executar suas liberações, aquelas que estão em teste serão mantidas e versões idênticas serão criadas para o usercode de produção. Se as libraries de produção ficarem residentes sem usercode, ao executar suas liberações, aquelas que estão em teste serão removidas e outras idênticas serão criadas em *, ou seja, sem usercode, para atender aos dois ambientes (teste e produção).

Funções de apoio

Específicas de área de trabalho

Tratamento de área conversacional

O SADS permite manipular as informações existentes, num dado momento, na área conversacional, ou seja, permite incluir, alterar, excluir ou consultar as informações armazenadas em uma determinada estação na sua área conversacional.

CAMPOS DA TELA

Em cada página serão mostrados 240 caracteres, dispostos em três blocos de quatro linhas cada, sendo que, a primeira linha exibe o caractere, a segunda e a terceira, o caractere hexadecimal correspondente, e a quarta linha, a posição do byte no registro.

 TAM.REG - Este campo vem preenchido com o número de bytes que o registro contém.

 POS.INIC - Campo para indicar, em operações de alteração, a posição do primeiro byte que será alterado.

 NUM.POS - Quantidade de posições que serão alteradas (máximo de 80 para caracteres e de 40 para hexadecimais).

A partir da posição informada no campo POS.INIC, serão alteradas uma quantidade de posições equivalente ao número informado no campo NUM.POS.

 CHA/HEX - Tipo do caractere informado para alteração.

Os valores possíveis são os seguintes:

"H" - Para caracteres hexadecimais;

"C" - Para caracteres que não são hexadecimais.

 CONTINUA(M/P/N) - Indicador de continuação de operação.

Os valores possíveis são os seguintes:

"M" (Mesma) - Para exibir a mesma página, ou seja, a mesma posição relativa do registro mas com os novos caracteres, caso tenham sido alterados;

"P" (Próxima) - Para exibir a próxima página com os 240 bytesseguintes;

"N" (Não continuar) - Para retornar ao menu de escolha "ARE".

O último campo é uma linha de 80 posições utilizada para informar os caracteres a serem inseridos ou substituídos a partir da posição inicial informada no campo POS.INI.

A alteração é feita de 80 em 80 caracteres ou de 40 em 40 hexadecimais, no máximo. Serão alterados tantos bytes quanto o número informado no campo NUM.POS.
