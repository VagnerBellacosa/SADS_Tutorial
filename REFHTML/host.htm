Host

Conceituação

Host é um programa on-line que desempenha dois papéis fundamentais, durante o desenvolvimento e funcionamento dos sistemas aplicativos, a saber:

 Numa primeira fase, gerencia a prototipação de transações on-line de um sistema quando este ainda está no projeto lógico, ou seja, quando não existem programas codificados e a base de dados ainda não foi construída. A partir de menus, telas e regras previamente cadastradas, o host simula a execução do sistema, dando a idéia de como será sua operação após o desenvolvimento. Nesta fase, ele é denominado de host lógico.

 Na etapa de processamento do sistema, após a fase de prototipação, executa funções genéricas, recebe e distribui as transações diretamente para os respectivos programas. Estes programas, que especificamente processam transações on-line, são denominados "programas chamados" e seus objetos são agregados ao host através do utilitário binder da Unisys. Nesta fase, ele é denominado de host físico.

Programas chamados são aqueles que executam os procedimentos relacionados às transações on-line de um sistema, cuja execução não pode ser processada isoladamente, sendo, por isso, necessária uma prévia agregação ao programa host através do binder.

Fonte de um programa host

Os hosts são totalmente gerados pelo SADS, portanto, as complementações nos seus fontes através de módulos são desnecessárias. Entretanto, os tratamentos específicos nos hosts podem ser feitos através de libraries de work e de procedure.

Devido ao fato de serem totalmente gerados pelo SADS, os fontes de programas host possuem um padrão único de codificação.

Para efetuar manutenções no fonte de um programa host, basta alterar o seu cadastramento no repositório de dados do SADS e gerá-lo novamente.

O fonte de um host contém comandos responsáveis por executar funções genéricas dos sistemas, garantindo, assim, a uniformidade dos comandos e, consequentemente, uma boa performance dos mesmos.

Nomenclaturas adotadas para programas hosts

Existem algumas nomenclaturas adotadas pelo SADS referentes aos hosts. São elas:

 Host lógico - Host utilizado na etapa de prototipação de sistema on-line, quando este ainda não possui programas e a base de dados ainda não foi construída. A partir de menus, telas e regras previamente cadastradas, o host simula a execução do sistema, fornecendo uma idéia de como será a sua operação após o desenvolvimento.

 Host físico - Host que processa as transações após o desenvolvimento e implantação do sistema. Neste caso, o host, além de simular a operação, gerencia as transações que realmente fazem acesso às bases de dados do sistema.

 Host primário - Host encarregado de controlar e disparar todas as transações pertencentes aos menus a ele relacionados. É aconselhável que sejam agregadas ao host primário apenas as transações menos pesadas do sistema, ou seja, aquelas que não consomem muito tempo de processamento. Desta forma, as demais ficariam agregadas ao host secundário, que, por sua vez, é disparado pelo host primário. Isto quer dizer que, mesmo as transações processadas pelo host secundário devem ser definidas no primário.

As transações gerenciadas diretamente pelo host primário estão associadas aos menus de código "sssUnn".

 Host secundário - Host disparado pelo primário com o objetivo de gerenciar as transações mais pesadas do sistema, ou seja, aquelas que consomem maior tempo de processamento. Deste modo, o processamento do host primário é aliviado, evitando, assim, a concorrência das transações demoradas com as rápidas, minimizando enfileiramentos e garantindo um bom tempo de resposta ao usuário final.

As transações gerenciadas pelo host secundário estão associadas aos menus de código "sssULn".

Padronização

O código padrão que identifica um programa host no repositório de dados do SADS é o seguinte:

 "sssNnn" onde:

"sss" é o código alfabético que identifica o sistema:

"N" é a constante de programa on-line e host e

"nn" é o indicador numérico (valores válidos de 00 a 10).

O padrão de codificação de um programa host é o mesmo dos programas on-line. O que os diferencia é o indicador numérico, que, para o host, varia de 00 a 10.

Utilização

O host, tal como citado em sua definição, é responsável por gerenciar as transações de um sistema, tanto na sua fase de prototipação como na de desenvolvimento.

Para melhor explicar como os hosts são utilizados, serão expostos os seguintes tópicos:

Funções genéricas efetuadas pelo host;

Área global de comunicação do host.

Funções genéricas efetuadas pelo host

O host trata várias funções genéricas, evitando, assim, que estas tenham de ser definidas e codificadas nos programas aplicativos.

As principais funções tratadas pelo host são as seguintes:

Segurança de acesso aos dados: esta validação é efetuada de acordo com informações do logon;

Abertura de banco de dados: somente os hosts, em transações on-line, abrem bancos de dados;

Recuperação de banco de dados: tratamento padrão de exclusão de informações do restart dataset;

Tratamento de menus e submenus: montagem de data, hora, sigla de empresa, ambiente do logon do usuário e campos específicos de output nestas telas e processamento das opções;

Tratamento de opções genéricas: as opções 94 a 99 são reservadas e tratadas exclusivamente pelos hosts;

Comunicação com o COMS: a recepção e o envio de mensagens em transações on-line geralmente são executadas pelos hosts.

Controle da área conversacional;

Navegação entre menus: tratamento de menus de escolha e de operação;

Funções de help: preparação e disparo do help on-line do SADS;

Consistência de menu e transação: interface com a consistência geral do SADS nas funções de preparação, disparo e apresentação de mensagens de erro;

Comunicação entre diferentes programas: disparo de transações de outros hosts, inclusive com controle de retorno;

Tratamento de mensagens de erros: leitura e apresentação de mensagens de erro tabeladas;

Disparo dos paginadores de entrada e de saída;

Leitura de telas, em sistemas convencionais, e composição de dados, em sistemas cooperativos.

A seguir, cada uma destas funções são explicadas separadamente.

Segurança de acesso aos dados

Para toda transação que é requisitada, através da integração com o Sistema de Segurança, o host verifica se o acesso é permitido ao usuário.

Além disso, o host dispara as transações de logon e logoff quando as mesmas forem solicitadas pelo usuário.

Quando há cancelamento de programa em produção, o host desabilita automaticamente a transação causadora do cancelamento. Neste caso, a transação só poderá voltar a ser processada se houver nova habilitação no Sistema de Segurança. A finalidade deste procedimento é evitar o disparo sucessivo da transação causadora do cancelamento sem antes corrigir o problema que o gerou.

Abertura de banco de dados

Os bancos de dados e todos os datasets aos quais serão feitos acessos pelos programas chamados têm de estar declarados na base de dados cadastrada para ser utilizada pelo host. Nas rotinas on-line, somente os hosts abrem bancos de dados. Portanto, quando "entra no ar", o host processa a abertura dos bancos e os mantém abertos enquanto ele estiver ativo.

Recuperação de banco de dados

A recuperação de banco de dados pode ser sincronizada com o COMS, de acordo com o critério definido pela instalação no ato da implantação do SADS.

Não havendo recuperação sincronizada, o host solicita ao usuário o reprocessamento da transação que estava sendo executada quando ocorreu o abort. Neste caso, o host remove também as informações do restart dataset referentes a ele, sem fazer nenhum tratamento das mesmas.

Tratamento de menus e submenus

O host trata todas as telas de um sistema: menus, submenus e telas de processamento.

Existem as seguintes possibilidades de tratamento das mensagens enviadas ao host através destas telas:

Mensagem originada de um de seus menus ou submenus de opção.

Neste caso, o host determina, através do número da opção informada no mesmo, qual programa chamado deverá ser disparado ou qual submenu deverá ser apresentado.

Mensagem originada de uma tela de processamento:

Neste caso, o host entrega a mensagem ao programa chamado, relativo à transação, a fim de que ele a processe.

Quando o programa chamado concluir a operação sobre a informação originada da tela de processamento, o controle é retornado ao host.

A resposta da operação para o usuário final será enviada pelo host de acordo com o que foi indicado pelo programa chamado nos campos GBL-ERRO e GBL-STATUS da área global de comunicação.

Tratamento de opções genéricas

Algumas opções, que podem ser disparadas de qualquer um dos menus de opção de sistemas desenvolvidos com o SADS, são tratadas diretamente pelos hosts, sem a necessidade de estarem relacionadas ao mesmo no repositório de dados do SADS.

Estas opções, denominadas opções genéricas, executam as seguintes funções:

Opção 94 - Retorno ao menu hierarquicamente superior da navegação executada via SOS01;

Opção 95 - Integração com o Sistema de Controle de Processamento (SPO);

Opção 96 - Acesso ao menu com detalhamento de opções;

Opção 97 - Ativação da tela de explicação dos argumentos do menu;

Opção 98 - Ativação do logon;

Opção 99 - Ativação do logoff.

A seguir, serão explicadas cada uma destas opções separadamente.

-Opção 94 - Retorno ao menu hierarquicamente superior (nível IV) da navegação via SOS01.

A transação SOS01 propicia a navegação entre os menus dos sistemas de forma hierarquizada, auxiliando o usuário final a encontrar o código correto da transação a ser disparada, para utilização de um determinado sistema.

O SOS01 controla a navegação entre os níveis I, II, III e IV. O nível V já é controlado pelo host do sistema. A partir de algum menu (nível V) ou submenu (nível VI) de um sistema, deve-se utilizar a opção 94 para retornar ao menu de nível IV e passar o controle da navegação para o SOS01.

-Opção 95 - Integração com o Sistema de Controle de Processamento (SPO).

Esta opção tem o objetivo de realizar a integração entre os sistemas aplicativos e o Sistema de Controle de Processamento (SPO). Através desta opção, pode-se selecionar qual serviço será executado dentre aqueles que se encontram cadastrados no SPO e para os quais o usuário está autorizado.

-Opção 96 - Acesso ao menu com detalhamento de opções.

Quando um sistema é utilizado por um número reduzido de usuários e estes já conhecem amplamente todas as transações dos menus, é dispensável que as telas de menu lhes sejam completamente apresentadas. Para este tipo de usuário, basta que os campos de argumentos e opção estejam disponíveis para que eles possam disparar a operação desejada.

Considerando que os menus são as telas mais utilizadas em um sistema, pois é a partir deles que qualquer operação do sistema tem início, caso eles pudessem ser sempre resumidos, haveria um ganho real considerável de tráfego de caracteres na linha.

A opção 96, oferecida pelo SADS, é um recurso que torna possível a simplificação da tela menu a fim de reduzir o tráfego da linha, sem que, no entanto, o acesso ao seu desenho completo seja perdido.

Para isso, o analista deverá construir o desenho resumido do menu para ser utilizado pelos usuários e conservar o desenho completo do mesmo, que deverá ficar disponível para consultas.

Para utilizar a opção 96 como recurso de redução de tráfego de linha, os seguintes procedimentos devem ser efetuados:

a) Duplicação da tela menu de código sssTnn00 para o novo código sssTnn96;

b) Simplificação do desenho da tela menu de código sssTnn00 para conter apenas o trancode, data/hora, ambiente de logon do usuário, campos de argumento, opção e campo de transmissão;

c) Liberação dos desenhos de ambas as telas para o ambiente onde o sistema será executado: teste e/ou produção;

d) Geração da library da tela de código sssTnn00;

e) Compilação do host e programas chamados;

f) Execução do binder para o host.

-Opção 97 - Ativação da tela de explicação dos argumentos do menu.

Através desta opção o usuário tem acesso à tela de explicação sobre o preenchimento dos argumentos necessários ao disparo de uma determinada transação do menu ou submenu.

Para acionar a tela de explicação dos argumentos do menu, o usuário deverá informar o número da opção que requer esclarecimentos no primeiro campo disponível da tela menu e 97 no campo OPCAO. O host apresentará a tela de explicação dos argumentos referentes à opção somente se o seu desenho estiver cadastrado no repositório de dados.

Para que este recurso oferecido pelo SADS possa ser utilizado, alguns procedimentos devem ter sido efetuados, quais sejam:

a) Cadastramento do desenho da tela de explicação dos argumentos da opção no repositório de dados.

Esta tela deverá possuir o código sssEnnmm, onde "nn" representa o código do menu e "mm", o número da opção.

Exemplo: A tela explicativa da opção 03 do menu ABC01 deve possuir o código ABCE0103.

b) Liberação do desenho da tela de explicação para o ambiente onde o sistema será executado: teste ou produção.

Através desta opção pode-se, também, obter descrições de telas e funções, dependendo da situação em que a opção e a função de help forem requisitadas.

-Opção 98 - Ativação do logon.

Esta opção é utilizada quando o usuário deseja efetuar o logon para utilização do SADS e/ou de sistemas desenvolvidos no ambiente SADS.

Quando esta opção é selecionada a partir de um menu de opção de qualquer sistema desenvolvido com o SADS, o host deste dispara o Sistema de Segurança, que, por sua vez, apresenta uma tela onde o usuário deverá informar seu número ou código e sua senha, dentre outros dados.

Caso algum usuário esteja com um logon ativo na estação, o Sistema de Segurança fará o logoff do mesmo e o logon do novo usuário.

Após o logon, será exibida uma tela, mensagem ou contexto, ou será executado um programa, de acordo com definição de resposta após logon cadastrada para o usuário ou de acordo com a parametrização adotada para a instalação ou estação (esta parametrização é definida na tabela SBPM6004).

Detalhes a respeito da resposta enviada após o logon estão descritos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações", e na Seção V, "Parametrizações do SADS", no item "Resposta após o logon/logoff", neste manual.

-Opção 99 - Ativação do logoff.

Esta opção é utilizada quando o usuário deseja efetuar o logoff no ambiente SADS.

Após o logoff, será exibida uma tela, mensagem ou contexto, ou será executado um programa, de acordo com a definição de resposta após o logon cadastrada para o usuário ou de acordo com a parametrização adotada para a instalação ou estação (esta parametrização é definida na tabela SBPM6004).

Maiores detalhes a respeito da resposta após o logoff podem ser consultados na Seção VI, "Sistema de Segurança", item "Usuários e habilitações" e na Seção V, "Parametrizações do SADS", no item "Resposta após o logon/logoff", neste manual.

Comunicação com o COMS

O host é o programa do sistema responsável por estabelecer a interface com o COMS.

Ele recebe, através do COMS, a mensagem de um de seus menus e a entrega para o programa chamado que processará a transação. Após o processamento, o controle é retornado ao host, que envia a resposta da transação, através do COMS, para a estação.

Controle da área conversacional

A área conversacional é uma área de memória reservada para armazenamento e transferência de informações entre as diversas transações de uma mesma estação.

A área conversacional, para que possa ser utilizada pelas transações de um sistema, deve ser declarada no host do mesmo. Deste modo, o host a manterá atualizada.

Cumpre ressaltar que o host sempre limpará a área conversacional quando for processada uma transação que não a utilize.

Para definir que uma transação irá utilizar a área conversacional, deve-se informar "S" para a transação, no campo ACV, na tela de cadastramento de menu.

Navegação entre menus

O SADS dispõe de um método amigável de navegação, orientado por menus, que facilita ainda mais a utilização do sistema.

O host é, pois, o elemento responsável por permitir esta navegação entre os vários menus de escolha, de operação e telas de processamento de um sistema desenvolvido através do SADS.

Detalhes a respeito de como cadastrar e utilizar a navegação podem ser consultados na Seção III, "Funções de apoio", no item "Navegação", neste manual.

Funções de help

A ajuda on-line ou help é uma opção oferecida pelo SADS, e ativada pelo host, para esclarecimento de dúvidas ou de erros ocorridos, apoiando o usuário durante a utilização do sistema.

Caso o help seja solicitado logo após a detecção de um erro pela consistência, serão apresentadas as validades do atributo relativo ao erro.

A função de help é utilizada, também, para se obter maiores informações a respeito de uma tela de explicação de argumentos, para menus de opção.

Para utilizar a função de help, basta transmitir o trancode HELP.

Consistência de menu e transação

A validade dos dados de entrada de programas on-line é feita pela consistência geral de forma interpretativa e em tempo de execução.

O host pode disparar a consistência geral para telas de processamento e menus. Para que este disparo seja efetuado, deve-se indicá-lo especificamente, para cada transação, no cadastramento dos menus.

É aconselhável que a consistência da transação seja disparada pelo host, ao invés de pelo programa chamado, para se obter melhor performance do sistema.

Para definir o disparo da consistência da transação e do menu pelo host, deve-se informar "S" nos campos TRANSACAO e MENU, respectivamente, na tela de cadastramento do menu.

Comunicação entre diferentes programas

O host possui funções responsáveis por controlar vários tipos de comunicação entre programas, quais sejam:

Comunicação entre programas chamados vinculados ao mesmo host em um mesmo menu

Esta comunicação ocorre quando, após finalizado um determinado programa, for necessário o encadeamento de outro programa que esteja agregado ao mesmo host e no mesmo menu, para finalizar o processamento do primeiro.

Este tipo de comunicação torna-se possível de duas formas distintas:

a) Em um programa com lógica de consulta simples, através da alteração do desenho de sua tela de consulta, incluindo os argumentos do menu.

Para propiciar este encadeamento, deve-se, então, incluir no desenho e na estrutura da tela de consulta todos os argumentos do menu (como campos de input) , inclusive o campo OPCAO. Desta forma, a tela de consulta irá se comportar tal e qual a tela menu, e poderá ser disparado qualquer programa deste menu.

Deve-se também cadastrar a consistência para esta tela de consulta, que será executada antes do encadeamento do próximo programa.

b) Em um programa com lógica qualquer, através da composição dos campos da área global GBL-STATUS e GBL-INDPROGRAMA, dos campos de argumento da tela menu do segundo programa e da área conversacional;

Para indicar ao host que um programa chamado encadeará outro em uma mesma transação, o primeiro programa deverá conter os seguintes comandos:

MOVE 6 TO GBL-STATUS

MOVE <número da opção referente ao programa a ser encadeado> TO GBL-INDPROGRAMA.

Se necessário, deve-se compor os campos da área conversacional e os de input (argumentos) da tela menu, que serão utilizados pelo programa encadeado. Em certos casos deve-se compor, inclusive, o código da transação.

Quando um encadeamento via GBL-STATUS = 6 for utilizado, o programa encadeado deverá zerar o GBL-STATUS pois, como o endereço contido em GBL-INDPROGRAMA é dele, o host pode voltar a dispará-lo, causando um loop.

Esta forma de encadeamento permite que o BEGIN-TRANSACTION seja executado no programa encadeador e o END-TRANSACTION no programa encadeado. Desta forma, caso haja cancelamento, as transações executadas nos dois programas serão desfeitas.

O host não dispara consistência geral para o programa encadeado. Assim, na definição da opção no menu, deve-se indicar que ela não possui consistência na transação.

Caso haja necessidade da execução da consistência geral para a opção encadeada, ela deve ser disparada de um módulo de processamento do programa encadeado, através dos seguintes comandos:

PERFORM 93000-00-CONSISTENCIA THRU 93000-99-EXIT.

IF SSC-QTDERROS GREATER 0

MOVE GBL-TELA-ERROS TO MENSAGEM

GO TO 99000-99-EXIT-PROGRAM

ELSE

MOVE SSC-AREACOM TO MENSAGEM.

Mesmo que a consistência geral não seja executada, o campo MENSAGEM será movido para o campo SSC-AREACOM, para evitar problemas no encadeamento.

Comunicação entre programas de diferentes menus

Esta comunicação ocorre quando, após finalizado um determinado programa, for necessário o encadeamento de um segundo programa que não pertença ao mesmo menu do primeiro.

Este tipo de comunicação torna-se possível de duas formas distintas:

a) Em um programa com lógica de consulta paginada, através da alteração do desenho de sua tela de consulta, incluindo os argumentos do menu, necessários ao encadeamento do segundo programa.

Esta forma de encadeamento está detalhadamente descrita no tópico "Desvio de transações através do paginador de saída de dados", item "Utilitários on-line", na seção III, neste manual.

b) Em um programa com lógica qualquer, através do disparo via COMS, caracterizando, portanto, duas transações distintas.

Este tipo de comunicação torna-se possível através do campos da área global GBL-STATUS e dos campos de argumento da tela menu do segundo programa.

Para determinar ao host que um programa encadeará um segundo programa, o primeiro programa deverá conter os seguintes comandos:

MOVE 9 TO GBL-STATUS.

Além disso, o programa deverá compor os seguintes campos:

 Todos os campos de input (argumentos) da tela menu, necessários ao disparo do segundo programa, inclusive o código da transação, pois ele será utilizado pelo COMS para o endereçamento.

 O campo H-TAMANHO de acordo com o tamanho total dos campos de input da tela menu da transação a ser encadeada.

Se necessário, pode-se compor os campos da área conversacional, para serem utilizados pelo programa encadeado.

Transferência de transação

Este tipo de comunicação ocorre quando uma opção do host for, na verdade, processada por um programa pertencente a um outro sistema ou a um outro host. Deste modo, quando a opção for requisitada, o host repassa os argumentos informados para serem processados pelo programa responsável.

Esta transferência é executada através do COMS, caracterizando, portanto, duas transações distintas.

Este tipo de comunicação é automática e executável a partir da especificação da opção no cadastramento do menu de onde ela será disparada.

Detalhes sobre o processo de transferência entre programas estão descritos no item "Menu", nesta seção.

Tratamento de mensagens de erros

Quando o programa chamado concluir seu processamento específico, o controle será devolvido ao host.

A resposta da operação para o usuário final será enviada pelo host de acordo com o que foi indicado pelo programa chamado nos campos GBL-ERRO e GBL-STATUS da área global de comunicação.

Maiores detalhes sobre como utilizar e preencher estes campos podem ser consultados no tópico "Área global de comunicação", neste mesmo item.

Disparo dos paginadores de entrada e de saída

O host é responsável por disparar os paginadores de entrada (STP05) e de saída de dados (STP03) do SADS.

Na inclusão ou entrada de dados, os programas chamados preparam as telas onde os dados serão informados e devolvem o controle ao host. Este, por sua vez, dispara o paginador de entrada que se encarrega de gerenciar a paginação das telas montadas e de disparar a consistência dos dados. Ao encerrar a paginação, o controle é repassado ao programa chamado que irá efetuar o devido tratamento dos dados.

Na consulta ou saída de dados, os programas chamados preparam todas as informações a serem apresentadas ao usuário e devolvem o controle para o host. Este, por sua vez, dispara o paginador de saída, que se encarrega de gerenciar a paginação das telas montadas.

Leitura de telas

No cadastramento do menu pode-se especificar o número da tela a ser utilizada por cada opção disponível no mesmo.

No caso de haver especificação de número de tela, o host efetuará a leitura da mesma antes de entregar o controle ao programa chamado responsável por processar a transação.

A utilização desta facilidade evita que o programa chamado tenha de fazer um novo link com a sub-rotina de leitura de telas, melhorando consideravelmente a performance do sistema.

Área global de comunicação do host

A comunicação entre o host e seus programas chamados é feita através da área global de comunicação, que é incorporada aos mesmos através de library.

Conteúdo da área global de comunicação

A área global de comunicação é composta pelas seguintes subáreas:

 Database Section;

 Áreas de controle do SADS;

 Área conversacional;

 Área de envio e recebimento de mensagens;

 Área de comunicação do host com os programas chamados.

Segue a descrição de cada uma destas áreas.

DATABASE SECTION

Esta área é reservada à declaração das estruturas DMSII, banco de dados e datasets, utilizados pelo host e programas chamados. A área declarada para o host, portanto, será exatamente a mesma utilizada pelos programas chamados.

Áreas de controle do SADS

São áreas incorporadas na WORKING-STORAGE SECTION e que são utilizadas para comunicação do host com seus programas chamados.

Estas áreas são utilizadas em processamentos genéricos, tais como: leitura de telas, disparo da consistência geral, segurança de acesso aos dados, recuperação DMSII e comunicação com o COMS.

Área conversacional

A área conversacional é uma área comum ao host e a seus programas chamados.

Cada sistema pode definir sua própria área conversacional no repositório de dados do SADS e gerar sua library para utilização pelos programas aplicativos.

O host mantém a área conversacional disponível, íntegra e atualizada para as transações que fazem uso dela, de modo a permitir a transferência de dados entre transações de uma mesma estação.

Área de envio e recebimento de mensagens

Esta área é utilizada pelo host para transferir as mensagens recebidas para o programa responsável pela execução e enviar as telas e mensagens de resposta para o terminal.

Esta área é redefinida pelas libraries das telas nos programas chamados.

Campos para comunicação do programa chamado com o host

Estes campos são utilizados pelos programas chamados para transferir informações ao host, no momento em que o processamento é retornado ao mesmo.

Estes campos, tal como apresentados na library, são exibidos a seguir:

 05 GBL-MENSAGEM - Mensagem a ser enviada pelo host no rodapé da tela.

 10 GBL-MENSINI PIC X (60). - Parte inicial da mensagem.

 10 GBL-MENSFIM PIC X (20). - Parte final da mensagem.

 05 GBL-ERRO PIC 9 (01) COMP. - Indica a forma como a resposta será processada pelo host.

 05 GBL-STATUS PIC 9 (01) COMP. - Indica o tipo de processamento a ser efetuado no host.

 05 GBL-INDPROGRAMA PIC 9 (03) COMP. - Número da opção a ser disparada pelo host,em encadeamento de programas.

 05 GBL-TBLCOD PIC X (08). - Tabela de mensagem de erros.

 05 GBL-MENSCOD PIC X (05). - Código da mensagem de erro.

Segue uma explicação detalhada de cada um destes campos:

a) GBL-MENSAGEM:

Este campo contém a mensagem a ser enviada pelo host para o rodapé da tela.

O programa chamado, quando desejar enviar uma mensagem qualquer ao usuário (mensagem de erro, confirmação de operação, dentre outras), deve ter o campo GBL-MENSAGEM preenchido com a mensagem desejada. Porém, o campo GBL-ERRO deve ter um valor diferente de zero e ser preenchido de acordo com o processamento a ser efetuado pelo host para a exibição da mesma.

Os valores a serem informados no campo GBL-ERRO estão apresentados numa tabela a seguir, em explicação específica reservada para o mesmo.

b) GBL-MENSINI:

Parte inicial da mensagem a ser composta pelo programa chamado. Utilizada quando houver necessidade de divisão da mesma.

c) GBL-MENSFIM:

Parte final da mensagem a ser composta pelo programa chamado. Utilizada quando houver necessidade de divisão da mesma.

Este campo também pode ser utilizado quando a mensagem a ser apresentada pertencer a uma tabela de erro. Neste caso, esta informação tem de ser composta pelo programa chamado.

Exemplo: O programa chamado ABCN75 deseja enviar a mensagem de erro de código N7503, da tabela de erros ABCM5099, detectada no argumento da tela menu CT01I-EMPCOD. Para que isso seja possível, ao final da sua execução, ele deverá conter os seguintes comandos:

MOVE 5 TO GBL-ERRO

MOVE "N7503" TO GBL-MENSCOD

MOVE "ABCM5099" TO GBL-TBLCOD

MOVE CT01I-EMPCOD TO GBL-MENSFIM.

Neste caso, a mensagem exibida será a "N7503" da tabela ABCM5099, concatenada com o valor transmitido no campo CT01I-EMPCOD.

d) GBL-ERRO:

Este campo é utilizado para que o programa indique o tipo de apresentação da mensagem que o host irá efetuar.

Os processamentos possíveis de serem efetuados no host estão apresentados na tabela a seguir:

Este campo, de acordo com o código informado nos programas chamados, pode indicar os seguintes processamentos a serem efetuados pelo host:

f) GBL-INDPROGRAMA:

Número da opção relativa ao programa aplicativo que se deseja encadear, nos casos de encadeamento de programas numa mesma transação (utilização do GBL-STATUS = 6).

g) GBL-TBLCOD:

Código da tabela de mensagens a qual será executado acesso para apresentação da mensagem, cujo código está contido no campo GBL-MENSCOD. Esta informação será utilizada, também, pelo help on-line.

Este código de tabela deve ser informado no programa chamado para GBL-ERRO 5 e 6.

Caso uma tabela com código imediatamente anterior possua um item (mensagem) com o mesmo código, ela também será exibida.

h) GBL-MENSCOD:

Código da mensagem que será apresentada. Esta mensagem tem de estar descrita na tabela cujo código estiver contido no campo GBL-TBLCOD.

Este código de mensagem deve ser informado no programa chamado para GBL-ERRO 5 e 6.

Cadastramento

O processo de definição do host exige o cadastramento prévio do sistema para o qual ele será definido no repositório de dados do SADS.

No caso de definição de host físico (que não seja protótipo), com acesso a bancos de dados, o cadastramento do host só será possível se a base de dados a ser por ele utilizada já estiver cadastrada.

Se o programa host utilizar tabelas, sub-rotinas, arquivos Port, libraries de work ou de procedure, estes também deverão estar previamente cadastrados no repositório de dados do SADS.

Definição de host

CAMPOS DA TELA

 Host - Código do host no padrão SADS (sssNnn).

 Nome - Descrição resumida do host.

O nome tem a função de explicar e identificar resumidamente a que se refere o host.

Base de dados - Código da base de dados a ser utilizada pelo host, no padrão SADS (sssB00xx).

 Bancos (I/U) - Códigos dos bancos de dados que serão utilizados pelo host, no padrão SADS (sssBxx), e critérios de abertura dos mesmos.

Os critérios de abertura que podem ser utilizados são:

 "I" (Inquiry) - Para banco de dados que será utilizado pelo host somente para consulta dos dados nele armazenados;

 "U" (Update) - Para banco de dados que será utilizado pelo host para atualização dos dados nele armazenados.

A informação de quais são os bancos de dados utilizados pelo host é opcional, pois este não trabalha necessariamente com banco de dados.

Além disso, os bancos de dados somente poderão ser informados caso a base de dados declarada pelo host possua o código "sssB0000", indicando que a base de dados é genérica a todo o sistema. Neste caso, para elementos (hosts, programas batch e sub-rotinas) que referenciam a base de dados genérica é possível utilizar apenas dois bancos, sendo que apenas um deles pode ser utilizado como update.

 "OPCAO" nos menus (P/U) - Indicador da localização do campo para seleção das opções nas telas de menu do sistema.

Os valores possíveis são os seguintes:

 "P" (Primeiro) - Para que a OPCAO seja o primeiro campo da tela após o código de transação;

 "U" (Último) - Para que o campo OPCAO seja o último da tela.

Sugere-se que a opção seja definida como o último campo dos menus, pois os módulos do SADS assim foram desenvolvidos e o usuário poderá utilizá-los simultaneamente com os menus do sistema.

 Linhas p/ mensagens - Quantidade de linhas de mensagens de erro que serão tratadas pelo programa host.

Sugere-se definir duas linhas de mensagens de erro para as telas.

 Compilador - Nome do compilador a ser utilizado na compilação do programa host.

Deverá ser utilizada alguma das versões do compilador Cobol. A sugestão é que se utilize o compilador COBOL74, pois no momento hosts somente são gerados em COBOL74.

 Area conv:

 Tam - Quantidade de bytes ocupados pela área conversacional do host.

O tamanho da área conversacional deve ser um múltiplo de 6 (seis) e indicará o tamanho máximo da área conversacional a ser utilizada por todos os programas do host.

O tamanho da área conversacional utilizada pelo host deve ser sempre acrescido de 14, que é o tamanho da área necessária às consistências geradas automaticamente para programas de atualização. Nesta área são armazenadas a transação e data e hora da última atualização do registro processado.

Para hosts que não utilizam área conversacional, deve ser informado 0 (zero) neste campo.

A área conversacional é utilizada para armazenar e transferir dados entre as transações de uma mesma estação.

Caso se deseje utilizar área conversacional, o tamanho mínimo que pode ser informado é de 18 bytes.

 Library - Código da library da área conversacional a ser utilizada pelo host, no padrão SADS (sssLnn).

 Port File - Código do arquivo port file utilizado pelo host, no padrão SADS (sssAxn). (Opcional)

O arquivo port file é utilizado para comunicação entre programas. Para declará-lo no host ele tem de estar previamente cadastrado no repositório de dados do SADS.

Caso ele seja definido no host, os programas chamados podem tratá-lo como global.

 Sub-rotinas - Códigos das sub-rotinas a serem chamadas pelo programa host, informados no padrão SADS (sssSxx). (Opcional)

A declaração de sub-rotinas a serem disparadas pelo host só será necessária caso o sistema não utilize sub-rotinas globais. Caso contrário, esta declaração é dispensável, pois o host já efetua normalmente as chamadas a todas as sub-rotinas relacionadas aos seus programas.

As sub-rotinas que devem ser declaradas no host são aquelas utilizadas com frequência pelos programas chamados associados ao host e que, se declaradas como temporárias, permanecerão ativas enquanto o host também estiver ativo.

As chamadas às sub-rotinas normalmente não são executadas pelo host, a não ser em casos excepcionais.

O host, para colocar uma sub-rotina no ar executa um disparo passando o valor "9" no primeiro byte da área de comunicação. Assim, a sub-rotina deve tratar esta informação, não executando, neste caso, nenhum processamento.

 Arquivos - Códigos dos arquivos a serem utilizadas pelo host, no padrão SADS (sssAxnnn). (Opcional).

Os arquivos poderão ser utilizadas pelo host através de um procedimento especial contido em alguma library de procedure a ser incorporada ao mesmo. Ao declarar arquivos para um host, suas libraries serão invocadas. Os arquivos declarados no host têm de estar previamente cadastrados no repositório de dados do SADS.

 Tabelas - Códigos das tabelas a serem utilizadas pelo host, no padrão SADS (sssMnnnn). (Opcional).

As tabelas poderão ser utilizadas pelo host através de um procedimento especial contido em alguma library de procedure a ser incorporada ao mesmo. Ao declarar tabelas para um host, suas libraries serão invocadas. As tabelas declaradas no host têm de estar previamente cadastradas no repositório de dados do SADS.

 Libraries / Uso (W/I/B/T/A/D/E/F) - Código da library a ser incorporada ao host e código do local onde a mesma será incluída. (Opcionais)

Os códigos de locais onde as libraries podem ser incluídas são:

 W" (Work) - Para que a library informada seja incorporada na Working-Storage Section do host;

 "I" (Início) - Para que a library informada seja incorporada nos procedimentos iniciais do host, após o change/call das sub-rotinas, antes da abertura do banco;

 "B" (Banco) - Para que a library informada seja incorporada nos procedimentos iniciais do host, após a abertura do banco;

Definição de menu

Existem três tipos de classificações para os menus: menus de escolha e de operação - que não são utilizados na definição e geração de hosts - e menus de opção, que são os menus tratados pelos hosts para disparo de transações.

Os menus de escolha e de operação são apresentados no item "Navegação" da Seção III, "Funções de apoio", neste manual.

Menus de opção são aqueles efetivamente tratados pelos hosts e processados pelos programas.

Um menu de opção é composto de:

a) Uma tela para apresentação das transações disponíveis;

b) Uma definição no repositório de dados, onde serão especificadas as características do processamento a ser executado pelo host para cada transação.

As telas dos menus de opção não são apresentadas pelo módulo de navegação do SADS.

O padrão para codificação de menu é "sssUxn".

As transações definidas em um menu podem ser tratadas por um outro host, que é chamado host secundário, que somente pode ser disparado através do primário.

Portanto, dependendo de qual host tratará o menu, pode-se classificá-lo dentre dois tipos, quais sejam:

-Menu primário: menu de opção onde estão especificadas todas as transações que serão tratadas tanto pelo host primário quanto pelo secundário.

-Menu secundário: menu de opção onde estão especificados apenas os submenus e as transações que serão tratadas pelo host secundário.

Todos os detalhes referentes ao elemento menu, incluindo a sua definição, estão descritos no item "Menu", nesta seção.

Geração de programa host

Através do SADS, o programa host é gerado em linguagem Cobol a partir de suas especificações cadastradas no repositório de dados, não sendo necessário, de modo geral, efetuar qualquer complementação nos seus fontes.

Para efetuar manutenções no fonte de um programa host, basta alterar o seu cadastramento no repositório de dados do SADS e gerá-lo novamente.

O fonte de um host contém comandos responsáveis por executar funções genéricas dos sistema, garantindo, assim, a uniformidade e boa performance dos aplicativos.

Para a geração, é necessário que o host, seus menus e sua base de dados, caso exista, já estejam definidos no repositório de dados do SADS.

Geração de library de sub-rotinas globais

Para que fosse possível às sub-rotinas dos aplicativos serem utilizadas como globais, foi necessário criar uma função de geração de library de sub-rotinas globais. Através desta função, é gerado um arquivo com os comandos de CHANGE e CALL das sub-rotinas utilizadas nos programas aplicativos, para posteriormente ser incorporado ao host e ao Programa chamador de sub-rotinas, que juntos propiciam esta utilização global.

A função de geração de library de sub-rotinas globais também possibilita gerar o Programa chamador de sub-rotinas, que será incorporado ao host através do binder.

O padrão para o código da library de sub-rotinas globais a ser gerada é o seguinte:

"ss/s/NI/nn" onde:

 "ss/s" é o código alfabético que identifica o sistema;

 "NI" é a constante de library de sub-rotinas globais e

 "nn" é o identificador numérico do host (de 00 a 10).

O padrão para o código do Programa Chamador a ser gerado é o seguinte:

"ss/s/NF/nn/SISTSUB" onde:

 "ss/s/NF/nn" é o código do fonte do host e,

 "SISTSUB" é a constante de sub-rotinas globais.

Todos os detalhes referentes à conceituação e esquema de funcionamento das sub-rotinas globais, incluindo o Programa chamador de sub-rotinas, encontram-se descritos nesta Seção, no item "Sub-rotinas".

A geração de library de sub-rotinas globais não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos da tela "LIBSUBGBL", cujos campos estão explicados a seguir:

CAMPOS DA TELA

 Host - Código do host no padrão SADS (sssNnn).

 Ambiente - Ambiente na qual a library de sub-rotinas globais será gerada.

Os ambientes possíveis são:

"T" - Para gerar a library de sub-rotinas globais para o ambiente de teste.

Neste caso, ela será utilizada por programas compilados para teste.

"P" - Para gerar a library de sub-rotinas globais para o ambiente de produção.

Neste caso, ela será utilizada por programas compilados para produção.

Se o ambiente de produção para geração de libraries não especificar usercode (usercode *), as libraries serão utilizadas simultaneamente para compilações para teste e produção.

Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

 Usercode - Usercode de análise.

Para usercodes de análise, são definidos os elementos (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na seção V, "Parametrizações do SADS", neste manual.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Liberação de library de sub-rotinas globais para produção

Para a utilização das sub-rotinas dos aplicativos como globais no ambiente de produção, foi necessário que se criasse uma função de liberação de library de sub-rotinas globais.

Através desta função, será possível disponibilizar a library de sub-rotinas globais anteriormente gerada para os programas aplicativos a serem compilados para produção.

Todos os detalhes referentes à conceituação e esquema de funcionamento das sub-rotinas globais, incluindo o Programa chamador de sub-rotinas, encontram-se descritos nesta Seção, no item "Sub-rotinas".

Para liberar uma library para produção, é necessário que a mesma tenha sido gerada para o ambiente de teste.

A liberação de library de sub-rotinas globais não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos na tela "LIBLIBGBL", cujos campos estão explicados a seguir:

CAMPOS DA TELA

 Host - Código do host no padrão SADS (sssNnn).

 Usercode - Usercode de análise.

Para usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS", neste manual.

Se na parametrização do usercode informado houver indicação para que libraries de produção fiquem residentes no usercode de produção, ao liberar library para produção, a versão de teste será mantida e uma idêntica será criada no usercode de produção. Caso contrário, a library de teste será removida e outra será gerada sem usercode (em *) para utilização pelos programas compilados em ambos os ambientes.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".
