Introdução

Termos técnicos da Internet

Para facilitar o entendimento deste, que foi desenvolvido para atender a um "novo" ambiente, segue uma explicação sucinta e simplificada de alguns termos técnicos que aparecem com frequência no mundo Internet.

As explicações de tais termos foram obtidas a partir de glossários mais completos, que compilam relações de termos técnicos, siglas, nomes de instituições, etc. do mundo Internet. Estes glossários completos, por sua vez, foram obtidos a partir dos sites de dois dos mais importantes agentes da Internet no Brasil: a RNP - Rede Nacional de Pesquisa (www.ci.rnp.br) - e a EMBRATEL (www.embratel.net.br ).

A seguir estão apresentados tais termos:

 Browser

O browser é um cliente para extração de informação em um Web Server. Tipicamente, um browser será um programa, em um computador pessoal, que acessará, via Internet ou Intranet, um servidor (isto é, um programa que atende a demanda de clientes remotos) contendo informações de interesse amplo.

 Cookies

Mecanismo geral que a conexão do lado servidor pode usar a fim de armazenar e recuperar informações do lado cliente. Um servidor, quando retorna um objeto HTTP para o cliente, pode também enviar uma informação que o cliente deverá armazenar. Dados a respeito do cliente fazem parte desta informação. Após isto, qualquer requisição HTTP requisitada pelo cliente, irá incluir a transmissão de tal informação de volta ao servidor.

 HTML

Acrônimo de Hypertext Markup Language, é a linguagem padrão para escrever páginas de documentos Web, que possibilita preparar documentos com gráficos e links para outros documentos, para visualização em browsers.

 Internet
  1. Com inicial maiúscula, significa a "rede das redes", originalmente criada nos EUA, que se tornou uma associação mundial de redes interligadas, que utilizam protocolos da família TCP/IP. A Internet provê transferência de arquivos, login remoto, correio eletrônico, news e outros serviços;
  2. Com inicial minúscula, genericamente, significa uma coleção de redes locais e/ou de longa distancia, interligadas por pontes, roteadores e/ou gateways.

A Internet utiliza um modelo de computação denominado Cliente/Servidor, que permite o uso eficiente dos recursos disponíveis em cada máquina.

As aplicações deste modelo reduzem o tráfego na rede através da otimização na troca de dados entre clientes e servidores. O servidor cuida do armazenamento e da alteração de grandes coleções de dados, controla o acesso à informação e processa pedidos, enquanto o cliente (um PC, por exemplo) se encarrega de fazer o que for necessário para que a aparência e a interação com a informação seja a melhor possível.

 Intranet

É o termo descritivo usado para a implentação das tecnologias da Internet dentro de uma corporação, ao invés de uma conexão externa com a Internet. Essa implementação proporciona, de forma transparente, a disponibilização de imensos recursos informativos de uma organização para todos os seus computadores, com um custo e tempo mínimos de implementação.

 Servidor
  1. No modelo cliente/servidor, é o programa responsável pelo atendimento a determinado serviço solicitado por um cliente. Serviços como archie, Gopher, WAIS e WWW são providos por servidores;
  2. Referindo-se a equipamento, o servidor é um sistema que provê recursos tais como armazenamento de dados, impressão e acesso dial-up para usuários de uma rede de computadores.

 Site [instalação]
  1. Uma instituição, onde computadores são instalados e operados;
  2. Um nó Internet.

 TCP/IP

Acrônimo de Transmission Control Protocol/Internet Protocol, é a família de protocolos para a comunicação de dados inter-redes, originalmente proposta para a Advanced Research Products Agency (ARPA). Hoje é um padrão de fato para inter-redes abertas, e seu uso é amplamente difundido dentro e fora dos EUA.

 WWW

World Wide Web, ou Web. Meta-rede, baseada em hipertextos, que integra diversos serviços Internet, através de uma interface que possibilita o acesso a informações multimídia.

Apresentação

O Servidor Web é um módulo do SADS que visa disponibilizar os sistemas aplicativos que executam em uma máquina Unisys no ambiente World Wide Web.

Através dele, as transações dos aplicativos poderão ser acessadas por browsers que são padrões de mercado, tais como o Comunicator <$FComunicator é marca registrada da Netscape Communications Corporation.> e o Internet Explorer <$Finternet Explorer é marca registrada da Microsoft Corporation.>, em uma Intranet, ou mesmo na própria Internet.

Este servidor irá transformar, automaticamente, as aplicações atuais com telas caractere, disparadas através do COMS, em documentos HTML, para Web browsers.

Desta forma, os dados já armazenados no mainframe poderão ser acessados e atualizados através de browsers.

Como se percebe, através da utilização do Servidor Web, será possível a utilização de um equipamento Unisys - Série A ou Clear Path - como máquina servidora Internet/Intranet. Ela irá receber as requisições oriundas dos browsers e irá atendê-las, enviando documentos HTML que serão resultados de conversões automáticas de telas de transações de aplicativos, que processam em uma janela direta do COMS.

As informações a serem disponibilizadas no Servidor Web já se encontram no Unisys e os sistemas aplicativos já estão disponíveis para acessá-las automaticamente através da Intranet/Internet. Desta forma, é desnecessária a redundância destas informações e a criação de novos sistemas em uma nova plataforma.

Como a plataforma Unisys já é dominada tecnicamente, o treinamento requerido para viabilizar uma conexão com a Internet com sucesso é mínimo. Isto levando em consideração que o Servidor Web é de fácil instalação e não requer nenhum treinamento específico para controle e administração.

A equipe técnica, já habilitada a trabalhar com o mainframe, poderá continuar o desenvolvimento de novos aplicativos, agora contando com uma nova forma de acesso dos mesmos pelos usuários.

O Servidor Web irá permitir que homepages, filmes, processos JAVA, figuras, sons etc. sejam armazenados e disponibilizados na plataforma Unisys, aproveitando a disponibilidade de espaço de armazenamento e a rapidez de recuperação dos dados.

O Servidor permitirá também que, através do campo de endereço do próprio browser, seja disparada qualquer mensagem a fim de ser tratada pelo COMS, tal como: "?ON SADS", "?WRU", outros trancodes etc. Ele irá entregar devidamente tal mensagem ao COMS, para que ele a processe, e irá devolver a resposta também em páginas HTML.

Pré-requisitos

Para o perfeito funcionamento do Servidor Web, os seguintes pré-requisitos devem ser satisfeitos:

Os micros que irão acessar o Servidor Web deverão possuir um software de browser instalado. Além disso, é necessário que tal software seja de uma versão atualizada, a fim de poder usufruir de todas as implementações em código JavaScript criadas automaticamente nas páginas HTML.

  1. Desta forma, caso o browser utilizado seja o Navigator, é necessário que sua versão seja a 3.0 ou superior. Caso seja o Explorer, a versão deve ser 3.0 ou superior. Demais browsers não foram testados, visto serem pouco utilizados.
  2. Deve haver o protocolo TCP-IP instalado, tanto no micro com browser, quanto no Unisys.
  3. Para que uma página HTML que é um template de resposta de uma transação, possa ser corretamente enviada pelo Servidor Web, ela deve ter o limite máximo de 64 Kbytes.

Instalação

O Servidor Web foi implementado através dos seguintes objetos:

  • Programa principal SI/W/NC/00;
  • Library SI/W/SC/01:
  • Library SI/W/SC/02 e
  • Library SI/W/SC/03: Para formatação dos dados a serem transferidos entre os sistemas aplicativos e o Servidor Web.

Para o correto funcionamento do produto, o objeto do programa SI/W/NC/00 deve ser definido como um MCS no IDC ou NDL.

O program que irá disparar este objeto, deve possuir as seguintes configurações no Utility do COMS:

  1. Seu objeto deve estar sob * (asterisco);
  2. O usercode no qual ele será executado deve possuir obrigatoriamente o atributo "COMSCONTROL";
  3. O nome do program é livre, porém, por questão de padronização, sugere-se SIW00;
  4. Os parâmetros MINCOPIES e MAXCOPIES devem ser definidos com "1", a fim de garantir que apenas uma e sempre uma cópia do programa esteja no ar em uma mesma máquina.

Configurações

Parametrizações

Os parâmetros necessários ao funcionamento do Servidor Web devem ser definidos nas tabelas SBPM5077 - Parâmetros gerais para funcionamento do Servidor Web - e SBPM5078 - Extensões dos arquivos a serem tratados pelo Servidor Web.

A seguir, será exibida uma explicação sobre cada um dos parâmetros a serem cadastrados:

Porta de comunicação TCP/IP com os Browsers (Tabela SBPM5077)

 CODIGO = Deve-se informar um dos seguintes valores:

"PORTA"  Para configuração da porta a ser utilizada pelo Servidor Web;

"ADM1"- Para configuração da porta a ser utilizada pelo Administ. do Servidor Web; ou

"HTTP21"  Para configuração de uma segunda porta para utilização pelo Servidor Web, caso seja necessária.

 PARAMETRO = Número da porta.

A porta principal, por convenção, é a 80.

Máximo de conexões simultâneas permitidas para comunicação TCP/IP com os Browsers (Tabela SBPM5077)

 CODIGO = Deve-se informar um dos seguintes valores:

"MAXPOR" - Para configuração do número máximo de conexões simultâneas que poderão ser estabelecidas pelo Servidor Web utilizando a porta definida no parâmetro PORTA;

"ADM2" - Para configuração do número máximo de conexões simultâneas que poderão ser utilizadas pelo Administrador do Servidor Web, utilizando a porta definida no parâmetro ADM1; ou

"HTTP22"  Para configuração do número máximo de conexões simultâneas que poderão ser utilizadas através da segunda porta definida para o Servidor Web (Porta definida no parâmetro HTTP21).

 PARAMETRO = Informar o valor correspondente ao número máximo de conexões simultâneas na respectiva porta. Sugere-se 20.

 Número de conectores livres para comunicação TCP/IP com os Browsers (Tabela SBPM5077)

 CODIGO = Deve-se informar um dos seguintes valores:

"NUMPOR"  Para configuração do número de conectores sempre disponíveis na porta definida no parâmetro "PORTA";

"ADM3"  Para configuração do número de conectores sempre disponíveis na porta definida no parâmetro "ADM1"; ou

"HTTP23"  Para configuração do número de conectores sempre disponíveis na porta definida no parâmetro "HTTP21".

 PARAMETRO = Informar o valor correspondente ao número de conectores livres na respectiva porta.

Sugere-se 5. O máximo permitido é 30.

Permissão para gravação de cookies nas máquinas clientes (Tabela SBPM5077)

 CODIGO = Deve-se informar um dos seguintes valores:

"COOKIE"  Para configurar a permissão de se gravar cookies nas máquinas clientes, quando se utiliza a porta definida no parâmetro "PORTA";

"ADM4"  Para configurar a permissão de se gravar cookies nas máquinas clientes, quando se utiliza a porta definida no parâmetro "ADM1"; ou

"HTTP4"  Para configurar a permissão de se gravar cookies nas máquinas clientes, quando se utiliza a porta definida no parâmetro "HTTP1".

 PARAMETRO = Indicador de permissão de gravação de cookies nas máquinas clientes, que pode ser:

"0"- Não é permitido ao Servidor gravar cookies nas máquinas clientes; ou

"1"- O Servidor poderá gravar cookies nas máquinas clientes.

Para que o Servidor Web possa fazer o controle das sessões de trabalho das diversas estações conectadas, ele irá enviar cookies nas máquinas clientes. Porém, caso a recepção de um cookie seja impedida através desta parametrização, não será possível manter os dados correntes da estação criada no mainframe entre uma requisição e outra, tais como: código e senha do logon, janela corrente, usercode e password, etc.

Nível do log a ser gerado na execução do Servidor Web (Tabela SBPM5077)

 CODIGO = "NIVLOG"

 PARAMETRO = Identificador numérico do nível do log a ser gerado.

Quanto maior o nível do log, mais informações serão armazenadas.

Diretório raiz dos arquivos a serem acessados pelo Servidor Web (Tabela SBPM5077)

 CODIGO = "WEBDIR"

 PARAMETRO = Nome do diretório onde residirá todos os arquivos a serem acessados pelo Servidor Web.

Sugere-se que se informe "WEB_DIR".

Na parametrização do diretório dos arquivos, pode-se informar também seu usercode, conforme o seguinte exemplo:

(AMBWEB)WEB_DIR

Neste caso, quando se informar o usercode, ou o objeto do Servidor - SI/W/NC/00  ou tal usercode devem ser privilegiados, para evitar o problema de não se encontrar os arquivos a serem acessados, quando estes forem privados.

Os discos dos arquivos acessados serão os mesmos definidos na family do usercode do program SIW00 no Utility.

Prefixo dos códigos das estações a serem criadas (Tabela SBPM5077)

 CODIGO = "WEBSTA"

 PARAMETRO = Prefixo que irá compor o código das estações a serem criadas pelo Servidor Web.

O código completo de uma estação terá o seguinte padrão:

<parâmetro WEBSTA>/<endereçoIP>

Caso seja desejável definir uma configuração específica para todas as estações a serem criadas pelo Servidor Web, tal como window e user defaults, deve-se então cadastrar a estação <parâmetro WEBSTA>/DEFAULT no Utility contendo a configuração desejada. Desta forma, todas as estações criadas pelo Servidor Web irão assumir esta configuração cadastrada.

Regra de formação do código da estação (Tabela SBPM5077)

 CODIGO = "STP"

 PARAMETRO = Indicação de código diferenciado das estações a serem criadas, que pode ser:

"0" - Caso o código da estação seja criado conforme os padrões do Servidor Web:

<parâmetro WEBSTA>/endereçoIP;

onde WEBSTA é um prefixo para as estações, também parametrizado na tabela SBPM5077.

"1" - Caso a estação seja criada com o seguinte código :

<parâmetro WEBSTA>sequencial/endereçoIP;

onde WEBSTA é um prefixo para as estações, também parametrizado na tabela SBPM5077.

"2" - Caso a estação seja criada com o seguinte código :

<parâmetro WEBSTA>sequencial;

onde WEBSTA é um prefixo para as estações, também parametrizado na tabela SBPM5077.

Salva das telas dos aplicativos que foram convertidas automaticamente (Tabela SBPM5077)

 CODIGO = "SVHTML"

 PARAMETRO = Indicador de salva de páginas HTML.

Antes do Servidor Web enviar uma tela do aplicativo para a estação cliente, ele a converte automaticamente para documento HTML. Porém, ele só efetuará esta conversão caso este documento não exista gravado no mainframe, no diretório raiz. O documento HTML já existente no mainframe tanto pode ter sido criado através de uma conversão anterior do Servidor Web, quanto elaborado manualmente.

O Servidor Web irá salvar o documento HTML gerado através da conversão caso, na tabela SBPM5077, CODIGO = "SVHTML", o campo PARAMETRO esteja preenchido com "S".

O documento HTML será salvo no mainframe com o seguinte title:

(usercode)diretório/tela on pack

onde:

 usercode - É o usercode definido para o program SIW00 no Utility (ou o usercode também definido na tabela SBPM5077 - Tabela de Parâmetros do Servidor Web, PARAMETRO = "WEBDIR";

 diretório - É o diretório raiz dos arquivos a serem acessados pelo Servidor Web, definido na tabela SBPM5077 - Tabela de Parâmetros do Servidor Web, PARAMETRO = "WEBDIR";

 tela - Código da tela referente ao arquivo (sssTmmnn);

 pack  Disco primário definido na family do usercode do program SIW00.

Este documento HTML, referente a uma tela do aplicativo, pode ser alterado a fim de tornar sua apresentação mais elaborada e criativa. Para maiores detalhes das alterações possíveis, deve-se consultar o item "Características funcionais", posteriormente descrito nesta seção.

Extensão dos arquivos a serem acessados (Tabela SBPM5077)

Sempre quando é solicitada uma página HTML, o Servidor Web irá verificar, no diretório raiz dos arquivos a serem acessados - parâmetro "WEBDIR" da tabela SBPM5077-, se existe um arquivo com tal código. Se, por exemplo, for solicitado "palestra", será procurado o arquivo "<WEBDIR>/PALESTRA".

Caso exista, o Servidor irá enviá-lo à estação cliente. Caso não exista e o arquivo seja solicitado sem extensão, será então verificado se existe, também no diretório <WEBDIR>, o arquivo com o mesmo código, porém com extensão similar à especificada no parâmetro "EXT" da tabela SBPM5077. Se, por exemplo, nesta tabela estiver cadastrado "HTM", será então verificada a existência do arquivo "<WEBDIR>/PALESTRA.HTM".

Então, para a parametrização da extensão dos arquivos a serem acessados, deve-se cadastrar os seguintes campos na tabela SBPM5077:

 CODIGO = "EXT"

 PARAMETRO = Sugere-se "HTML" ou "HTM".

Timeout das estações (Tabela SBPM5077)

 CODIGO = "TIMOUT"

 PARAMETRO = Tempo máximo, em minutos, que uma estação fica no ar sem ter se comunicado com o Sevidor Web.

 Caso a estação não se comunique com o servidor por um tempo superior ao estabelecido neste parâmetro, a mesma será deletada.

Padrão para o fundo de páginas (Tabela SBPM5077)

 CODIGO = "FUNDO"

 PARAMETRO = Código do arquivo (".GIF", ".JPEG", etc.) que irá compor o fundo das páginas que serão montadas automaticamente pelo Servidor Web.

Permissão para disparo de transações COMS (Tabela SBPM5077)

Qualquer mensagem pode ser disparada através do campo de endereço do próprio browser, a fim de ser tratada pelo COMS, tal como: "?ON SADS", "?WRU", outros trancodes etc. O servidor irá entregar devidamente tal mensagem ao COMS para que ele a processe, devolvendo a resposta em páginas HTML.

Para disparar uma mensagem COMS, a mesma deve ser especificada no campo de endereço do browser no seguinte formato:

endereço/COMS_BIN/mensagem, onde:

 endereço - É o endereço IP da máquina na qual foi instalado o Servidor Web;

 COMS_BIN - É o diretório que identifica as mensagens a serem tratadas pelo COMS;

 mensagem - É a mensagem a ser interpretada pelo COMS.

Como exemplo, em uma determinada máquina fictícia na qual o Servidor Web está instalado, poderia ser disparada a seguinte mensagem:

http//exemplo.com.br/COMS_BIN/SDD03

Para configurar se será permitido disparar uma mensagem COMS através do Servidor Web, o seguinte parâmetro deverá ser determinado:

 CODIGO = "CBIN"

 PARAMETRO = Pode-se informar um dos seguintes valores:

"0"  Para que não seja permitido disparar mensagens COMS através do Servidor Web; ou

"1" - Para permitir o disparo de mensagens COMS através do Servidor Web.

Tamanho da área conversacional (Tabela SBPM5077)

 CODIGO = "CAREA"

 PARAMETRO = Tamanho da área conversacional a ser utilizada pela library SIWS03.

A library SIWS03 é utilizada para formatação dos dados a serem transferidos entre os sistemas aplicativos e o Servidor Web. Dependendo então da necessidade de tal library se comunicar com o Servidor, alguns dados deverão ser salvos na área conversacional, relativos a cada estação, e o tamanho destes dados salvos deve ser definido nesta parametrização.

Posicionamento do identificador da transação nas mensagens (Tabela SBPM5077)

Toda vez que o Servidor recebe uma string de dados, ele necessita reconhecer a tela, para acessar o seu respectivo template, em HTML, que está armazenado no mainframe. Desta forma, é imprescindível que o Servidor saiba reconhecer a exata posição dos identificadores das transações.

Os identificadores das transações, sejam eles trancodes ou não, podem estar posicionados em qualquer parte da mensagem a ser tratada pelo Servidor Web. Eles poderão ser compostos por caracteres não consecutivos do corpo da mensagem.

Para especificar ao Servidor como determinar os identificadores das transações, pode-se escolher entre informar os campos formatados (CAMPO1 e CAMPO2) ou as posições dos caracteres (POS1 e POS2) que deverão compô-los.

Caso a parametrização seja efetuada informando-se os campos (CAMPO1 E CAMPO2), no campo PARAMETRO deve-se informar a sequência do campo no corpo do menu e o número de caracteres a serem extraídos do mesmo.

Caso a parametrização seja efetuada informando-se as posições dos caracteres (POS1 e POS2), no campo PARAMETRO deve-se informar a posição inicial dos caracteres no corpo da mensagem, a iniciar de zero, e o número de caracteres a serem extraídos a partir desta posição.

Para melhor entendimento, suponha a seguinte mensagem a ser transmitida:

Supondo que o identificador da transação desta mensagem seja "ABC01.02", pode-se então parametrizar sua composição, através da indicação do campo, da seguinte forma:

 CODIGO = "CAMPO1"

 PARAMETRO = 1,8

Nesta parametrização foi informado que o campo que irá compor o identificador da transação (CAMPO1) é o primeiro da mensagem, ocupando 8 posições (1,8.

É importante informar que o primeiro campo da mensagem é o de número 1 (um).

Como um segundo exemplo, suponha a seguinte mensagem a ser transmitida:

Também supondo que o identificador da transação desta mensagem seja "ABC01.02", pode-se então parametrizar sua composição, através da indicação de sua posição, da seguinte forma:

 CODIGO = "POS1"

 PARAMETRO = 122,8

Nesta parametrização foi informado que os caracteres a compor o identificador da transação (POS1) ocupam 8 bytes a partir da posição inicial 122.

Concluindo, para a parametrização dos identificadores das transações, deve-se informar os seguintes campos na tabela SBPM5077:

 CODIGO = "CAMPO1", "CAMPO2", "POS1" ou "POS2".

 PARAMETRO =:
    • Se a parametrização for feita através dos campos formatados da mensagem, informar:

<sequência do campo>, <número de bytes do campo>;
    • Se a parametrização for feita através da posição dos caracteres na mensagem, informar:

<posição inicial dos caracteres>, <número de bytes do campo>

Tipos de extensões tratáveis (Tabela SBPM5078)

Na tabela SBPM5078 deverão ser especificados os códigos de extensões dos diversos tipos de arquivos possíveis de serem tratados pelo Servidor Web. Para estas especificações, deverão ser informados os seguintes campos da tabela:

 Extensao = Código da extensão dos arquivos a serem tratados pelo Servidor Web.

Como exemplos, os arquivos podem possuir as seguintes extensões: "JPE", "JPG", "JPEG", "HTM", "HTML", "EXE", "GIF", "MIME", etc.

 Content Type = Tipo do conteúdo armazenado nos arquivos que possuem a extensão especificada.

Os tipos de conteúdos devem ser informados em inglês, de acordo com o padrões já existentes e possíveis de serem tratados pelo Servidor Web.

Relativos a cada um dos exemplos de extensões descritos anteriormente, deveriam ser especificados os seguintes Content Types, respectivamente: "image/jpeg", "image/jpeg", "image/jpeg", "text/html", "text/html", "application/octet-stream", "image/gif", "message/rfc822".

Através dos exemplos expostos, percebe-se que podem ser definidos mais de um código de extensão para o mesmo tipo de conteúdo de arquivos.

Ambiente

Para configuração do ambiente onde deverão ser executados os sistemas aplicativos, recomenda-se:

    1. Definir uma Window no Utility onde serão executados todos os sistemas que terão acesso à WEB. Esta Window não deve ter "Notify Open".
    2. Declarar uma estação com o código <parâmetro WEBSTA>/DEFAULT no Utility.

É conveniente que todas as estações a serem criadas pelo Servidor Web possuam configurações específicas, tal como window e user defaults. Para isto sugere-se então cadastrar uma estação com o código <parâmetro WEBSTA>/DEFAULT no Utility contendo a configuração desejada. Feito este cadastramento, todas as estações criadas pelo Servidor Web irão assumir esta configuração cadastrada.

A estação default deve ser criada com as seguintes características:

2.1 - Default Window: Window recém-criada;

2.2 - Default usercode: Usercode com o mínimo de atribuições necessárias para se acessar os sistemas;

É recomendável que este usercode não esteja habilitado a disparar comandos "?";

2.3 - Window List: Somente a window recém-criada;

2.4 - CloseAction: Permanecer na Window.

 

Sistemas Aplicativos

Para facilitar a disponibilização de um sistema aplicativo residente em um mainframe Unisys para o ambiente Web, através do Servidor Web, pode-se optar entre as duas seguintes formas de desenvolvimento dos mesmos:

  1. Desenvolver o sistema através de uma ferramenta qualquer, ou até mesmo sem utilizar nenhuma ferramenta, porém tratando a entrada e saída de dados de acordo com o protocolo estabelecido pelo Servidor Web; ou
  2. Definir o sistema como cooperativo no SADS e gerá-lo automaticamente.

A seguir serão detalhadas cada uma destas duas formas de se desenvolver um sistema para integração com o Servidor Web.

Sistema não SADS

Quando o sistema não tiver sido gerado de forma cooperativa através do SADS, suas entradas e saídas, que serão integradas ao ambiente Web, deverão ser adaptadas para seguir o protocolo do Servidor.

Entrada

As entradas de dados para o sistema, enviadas pelo Servidor, não têm nada de particular: o programa deverá ser preparado para receber uma string contendo somente os dados de entrada que serão tratados pelo programa. Nenhum desenho de tela será enviado pelo Servidor.

A página HTML para captar os dados via Browser pode possuir formatos e elementos dispostos de forma totalmente criativa. Não é necessário que a página possua os mesmos campos da tela relativa do sistema. Porém, no momento de se enviar os dados ao mainframe via Servidor Web, a página deverá compor uma string contendo somente os dados a serem tratados pelo sistema.

Para informar ao Servidor sobre os campos de dados que deverão compor a string, o último campo a ser tratado no form a ser transmitido deverá ter a seguinte estrutura:

<input type=hidden name=layout value="tam1, tam2, tam3... tamult">, onde:

layout: Nome fixo do campo de informações;

tam1, tam2 ... tamult: Tamanho de cada um dos campos que irão compor a string, inclusive o trancode, caso possua.

Saída

Os dados que saem do sistema e que serão exibidos em uma página HTML devem também ser formatados em uma string de dados, sem atributos de vídeo. Esta string deverá possuir, em seus bytes iniciais, um header para comunicação com o Servidor Web no seguinte padrão:

"&K<código do template HTML(8bytes)><código do contexto(8bytes)><os dados>", onde:

&K  Constante que identifica início de header tratável pelo Servidor;

<código do template HTML> - 8 bytes que informam o código do template HTML que deverá ser acessado pelo Servidor e que deverá ser composto pelos dados enviados pelo sistema. Após esta composição de dados, a página HTML será enviada pelo servidor ao Browser;

<código do contexto> - 8 bytes que informam o código do contexto referente ao template HTML a ser acessado.

<os dados> - Dados fornecidos pelo sistema, provenientes de bases de dados no mainframe, que deverão compor a página HTML a ser apresentada no ambiente Web.

Com as informações acima, o Servidor irá acessar o seguinte arquivo a fim de compor os dados a serem apresentados:

(usercode)diretório/CONTEXTO/contexto/"template.ext"/T on pack

onde:

 usercode - É o usercode definido para o program SIW00 no Utility

 diretório - É o diretório raiz dos arquivos a serem acessados pelo Servidor Web, definido na tabela SBPM5077 - Tabela de Parâmetros do Servidor Web, PARAMETRO = "WEBDIR";

 CONTEXTO - Constante que identifica que o arquivo HTML para substituir uma tela de navegação de um contexto;

 contexto - É o código do contexto informado;

 template  É o código do template informado;

 ext - Extensão do arquivo definida na tabela SBPM5077 - Tabela de Parâmetros do Servidor Web -, PARAMETRO = "EXT";

 pack  Disco primário definido na family do usercode do program SIW00.

 

Sistema SADS

Quando o sistema tiver sido gerado de forma cooperativa através do SADS, durante seu desenvolvimento, nenhum trabalho adicional necessita ser feito para adequar suas entradas e saídas ao padrão estabelecido pelo Servidor Web. Isto porque, pelo simples fato de defini-lo como cooperativo, ele já é gerado totalmente compatível com o Servidor Web.

Geração de páginas HTML

 Geração de página HTML a partir de uma tela específica.

    • As palavras-chaves que não forem encontradas serão ignoradas.
    • Serão considerados apenas os caracteres maiúsculos para localização das palavras-chaves.

Características funcionais

Através da utilização do Servidor Web, torna-se possível a utilização de equipamentos Unisys Série A ou Clear Path como máquinas servidoras Internet/Intranet. Elas irão receber as requisições, oriundas de programas browser  e irão atendê-las, enviando documentos HTML, byte-codes JAVA, ou mesmo transações que processem em janela direta do COMS.

Início de sessão

Ao receber a primeira solicitação de uma sessão, o servidor verificará a existência do arquivo INDEX.HTML e, caso exista, ele será então exibido. Este arquivo pode ser elaborado manualmente de acordo com a necessidade e criatividade da própria instalação. Se este arquivo não existir, o servidor exibirá uma página contendo um índice de todos os arquivos existentes no diretório especificado no parâmetro "WEBDIR" da tabela SBPM5077 - Tabela de Parâmetros do Servidor Web.

Acesso aos arquivos

As telas das transações a serem utilizadas são convertidas automaticamente, sem necessidade de reprogramação, para documentos HTML, antes mesmo de serem enviadas para a estação cliente.

Sempre quando é solicitada uma página HTML, o Servidor Web irá verificar, no diretório raiz dos arquivos a serem acessados - parâmetro "WEBDIR" da tabela SBPM5077-, se existe um arquivo com tal código. Se, por exemplo, for solicitado "palestra", será procurado o arquivo "<WEBDIR>/PALESTRA".

Caso exista, o Servidor irá enviá-lo à estação cliente. Caso não exista e o arquivo seja solicitado sem extensão, será então verificado se existe, também no diretório <WEBDIR>, o arquivo com o mesmo código, porém com extensão similar à especificada no parâmetro "EXT" da tabela SBPM5077. Se, por exemplo, nesta tabela estiver cadastrado "HTM", será então verificada a existência do arquivo "<WEBDIR>/PALESTRA.HTM".

Se este arquivo também não existir, o Servidor Web interpretará o código "palestra" como sendo de um diretório e irá verificar a presença do seguinte arquivo: "<WEBDIR>/PALESTRA/INDEX.HTM".

Não encontrando e ainda assumindo que "palestra" é um diretório, o Servidor montará uma página com a lista de todos os arquivos e diretórios presentes no diretório solicitado. Caso não contenha arquivos e/ou diretórios, o Servidor Web, então, retornará uma mensagem de erro, indicando que o arquivo especificado - "palestra" - não foi encontrado (NO FILE).

Em síntese, para que o Servidor Web encontre e exiba um arquivo, ele percorre os seguintes caminhos:
    1. <WEBDIR>/ARQUIVO
    2. <WEBDIR>/ARQUIVO.<EXT>
    3. <WEBDIR>/ARQUIVO/INDEX.HTM
    4. <WEBDIR>/ARQUIVO/=
    5. NO FILE.

Para que o Servidor Web possa fazer o controle das sessões de trabalho das diversas estações conectadas, ele irá enviar cookies nas máquinas clientes. Porém, como a recepção de um cookie pode ser impedida através de configuração efetuada no browser, é importante salientar que, caso isto ocorra, não será possível manter os dados correntes da estação criada no mainframe entre uma requisição e outra, tais como: código e senha do logon, janela corrente, usercode e password, etc.

Mensagens de erro

Sempre quando for necessário apresentar uma mensagem de erro ao usuário oriunda de um sistema cooperativo, o Servidor Web enviará uma página HTML que possibilitará exibir tal mensagem em um diálogo, impedindo que a página anteriormente preenchida seja perdida.

Porém, se a mensagem de erro for enviada por um sistema não cooperativo ou um sistema não SADS, ela será entregue ao browser como uma página HTML qualquer, e, desta forma, irá se sobrepor àquela já existente. Neste caso, para recuperar a página anterior já preenchida, deve-se utilizar o botão "Back" do browser.

Mensagens para o COMS

Através do campo de endereço do próprio browser, pode ser disparada qualquer mensagem a fim de ser tratada pelo COMS, tal como: "?ON SADS", "?WRU", outros trancodes etc. O servidor irá entregar devidamente tal mensagem ao COMS para que ele a processe, devolvendo a resposta em páginas HTML.

Para disparar uma mensagem COMS, a mesma deve ser especificada no campo de endereço do browser no seguinte formato:

endereço/COMS_BIN/mensagem

onde:

endereço - É o endereço IP da máquina na qual foi instalado o Servidor Web;

COMS_BIN - É o diretório que identifica as mensagens a serem tratadas pelo COMS;

mensagem - É a mensagem a ser interpretada pelo COMS.
