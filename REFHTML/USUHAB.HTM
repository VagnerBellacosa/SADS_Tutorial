Usuários e habilitações

Introdução

O cadastro de usuários do Sistema de Segurança é composto basicamente de quatro tipos de informações, a saber:

Os dados dos usuários, que são informações específicas e que não dependem de aplicativos ou transações aos quais eles podem obter acesso, tais como número, senha, nome, código externo, empresa, área, unidades autorizadas, etc.

As habilitações de aplicativos, que determinam com quais aplicativos os usuários podem trabalhar.

As habilitações de transações, que determinam a quais transações, em cada um dos aplicativos anteriormente habilitados, os usuários podem obter acesso.

As habilitações de elementos, que determinam com quais elementos (valores de campos) os usuários poderão trabalhar em um aplicativo ou lista de elementos.

Para identificação de aplicativos, o Sistema de Segurança segue os padrões explicados na Seção I, "Requisitos e conceitos para utilização do SADS".

Já a codificação de listas de elementos ou elementos não segue nenhum padrão, sendo determinada de acordo com a necessidade e especificação do aplicativo que os utilizará.

Maiores detalhes sobre esta identificação de listas de elementos e elementos estão descritos no tópico "Níveis de Segurança disponíveis", nesta seção.

Responsáveis pelos cadastramentos

As operações de manutenção no cadastro e habilitações dos usuários só podem ser efetuadas pelo administrador de segurança, nível de acesso 99, ou por usuários por ele devidamente habilitados, que devem possuir nível de acesso 98, denominados subadministradores de segurança.

Aos administradores de segurança é permitido executar qualquer função do Sistema de Segurança, desde que esteja habilitado ao aplicativo e transação correspondentes.

Para que um usuário de nível de acesso 98 possa efetuar manutenções no cadastro e nas habilitações de outros usuários, ele deve possuir as seguintes habilitações e atender aos seguintes requisitos:

Todas as habilitações a serem por ele cadastradas ou alteradas devem possuir unidades autorizadas (empresa, estabelecimento e unidade) iguais ou compatíveis às suas, da seguinte forma:

Caso o usuário de nível de acesso 98 possua unidades autorizadas genéricas (iguais a 9's), então ele poderá efetuar manutenções em habilitações definidas em quaisquer unidades autorizadas, excetuando-se aquelas que não possuam unidades autorizadas cadastradas (iguais a brancos ou zeros );

Caso o usuário de nível de acesso 98 possua unidades autorizadas específicas, então ele poderá efetuar manutenções somente nas habilitações definidas nestas mesmas unidades autorizadas;

Caso o usuário de nível de acesso 98 não possua unidades autorizadas cadastradas (iguais
a brancos ou zeros ), então ele poderá efetuar manutenções somente naquelas habilitações
que também não possuírem unidades autorizadas informadas.

As unidades autorizadas do usuário afetado pela operação - alteração, exclusão, duplicação, troca de senha, etc - devem ser compatíveis às suas.

Ele não poderá cadastrar ou alterar o cadastramento de um outro usuário que possua nível de acesso 99 - Administrador de Segurança -, o que somente poderá ser feito por outro usuário de nível de acesso 99.

Ele não poderá alterar seu próprio cadastramento ou suas próprias habilitações.

Para que possa efetuar manutenção no cadastro de um segundo usuário de nível de acesso menor que 98 - alteração, exclusão, duplicação, troca de senha, etc -, ele deverá possuir todos os aplicativos que estiverem habilitados ao usuário definidos como elemento,na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar). A única exceção se faz na operação de liberação de usuário, que o sub-administrador estará habilitado a processá-la independentemente de suas habilitações e das habilitações dos usuários alvos da operação.

Para que possa habilitar elementos a um outro usuário, ele já deve possuir igual habilitação de elemento.

Para que possa habilitar um determinado aplicativo a um segundo usuário, o subadministrador de segurança deverá possuir tal aplicativo habilitado como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar).

Caso o usuário a ser por ele habilitado possua também nível de acesso 98, então este tipo de habilitação deve ser "9" (Habilitar outro usuário 98).

Só um administrador de segurança possui autorização para especificar "9" no tipo de habilitação e, assim mesmo, somente para um usuário 98 e apenas quando a lista de elementos for a "SBT" ou "SBT01".

Como exemplo, para que o usuário de nível de acesso 98 possa autorizar um outro usuário a utilizar o aplicativo "GLH04", ele deve possuir a seguinte habilitação de elemento:

Aplicativo/Lista de Elementos: "SBT" ou "SBT01"

Elemento: "GLH04" ou "GLH" ou "GL" ou "G" ou " " (brancos)

Tipo da habilitação: 2

Para que possa habilitar uma transação de um aplicativo a um segundo usuário, o subadministrador de segurança deverá possuir tal aplicativo habilitado como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar).

Caso o subadministrador também esteja habilitado ao aplicativo, ele deve então possuir habilitação à mesma transação do aplicativo, para que possa habilitá-la a outro usuário.

Para realizar manutenções no cadastro e habilitações de usuários, deve-se utilizar o menu de operação "CADUSU" e nele informar os argumentos de acordo com a operação a ser processada.

Cadastramento de usuário/perfil

Existem dois tipos de usuários no Sistema de Segurança:

O usuário comum, que possui seus dados cadastrais e suas próprias habilitações, específicas para as funções que desempenha e aplicativos/elementos que utiliza; e

O usuário perfil modelo, que congrega características e habilitações comuns a um grupo de usuários.

Pessoas de uma mesma equipe, que comumente trabalham com os mesmos aplicativos, executando funções similares, possuem dados semelhantes em seus cadastramentos e habilitações. Nesta situação, é aconselhável que se crie um perfil modelo, contendo esses dados comuns, e o relacione a todos os usuários pertencentes à equipe, através da operação de vinculação de usuário a perfil.

A vantagem de se criar um perfil modelo é que todas as características e habilitações, comuns aos usuários, são cadastradas uma única vez para o perfil, sendo herdadas por todos os usuários a ele vinculados. Qualquer alteração nestas habilitações passará a valer também para os usuários vinculados, após efetuar suas liberações.

Um usuário pode ser vinculado a um ou mais perfis modelo, de acordo com as funções afins que compartilha com tais perfis.

No momento da vinculação é estipulada uma prioridade para a mesma, a fim de que, havendo coincidência ou discordância de habilitações a serem herdadas pelo usuário, quando o mesmo estiver vinculado a mais de um perfil, seja então respeitada a prioridade destas vinculações.

Todo usuário deve ser submetido à operação de liberação antes que possa efetuar logon e após ter sido finalizado todo o seu cadastramento. Através deste processo é que são sintetizadas suas habilitações, que foram cadastradas especificamente para o mesmo ou foram herdadas dos vários perfis modelo aos quais ele foi porventura vinculado.

A cada vez que é alterado o cadastramento do usuário - unidades autorizadas ou habilitações - ou forem alterados os perfis aos quais ele está vinculado, deve-se então proceder novamente à sua liberação.

Após a primeira liberação do usuário, ele estará apto a efetuar o logon, mesmo se seu cadastramento estiver sendo alterado. É importante salientar que a alteração de sua empresa, estabelecimento, unidade, estação, ambiente ou dias habilitados passará imediatamente a ser considerada assim que processada. Porém, para que suas habilitações passem a ser consideradas, é necessário efetuar novamente sua liberação.

Um usuário vinculado pode ter habilitações próprias de aplicativos e elementos, diferentes daquelas cadastradas para os perfis a ele relacionados. Desta forma, é possível redefinir - restringir ou ampliar - as habilitações que um usuário herda de seus perfis modelo.

Um usuário vinculado pode também ter suas características próprias, tais como ambiente autorizado, empresa, área, etc.

Um usuário vinculado pode ser desligado de qualquer um de seus perfis a qualquer momento. Neste caso, o usuário perde as habilitações que foram herdadas do perfil, através da vinculação. O usuário fica com status "a liberar" e só poderá efetuar novo logon, caso seja processada a sua liberação.

Conforme percebido, um perfil modelo é um usuário abstrato, utilizado apenas para definir características e habilitações a serem herdadas por um grupo de usuários. Ele não efetua logon e nem trabalha em uma sessão SADS. Por este motivo, não possui dados cadastrais, referentes à utilização dos sistemas e controle de logon e senha, quais sejam nível de acesso, indicador de integração do logon com o Marc, validade da senha, indicador de controle de alteração de senha, indicador de dias habilitados para logon e códigos das estações habilitadas para uso.

As operações de manutenção no cadastro de usuário - inclusão, alteração e exclusão -, seja ele usuário comum ou perfil modelo, são disparadas a partir do mesmo menu de operação "CADUSU", bastando informar se o tipo do usuário a ser processado é um usuário comum ("U") ou perfil modelo ("P").

Contudo, as telas de processamento utilizadas em tais operações são diferenciadas para os dois tipos de usuários. Na manutenção de um usuário comum serão requeridos campos referentes à utilização dos sistemas e controle de logon e senha, o que, conforme explicado anteriormente, não são exigidos na manutenção de um usuário perfil modelo.

Inclusão de usuário/perfil

Esta operação propicia a inclusão de um usuário comum ou perfil modelo.

Caso o número do usuário a ser incluído não seja informado, será então gerado um número imediatamente superior ao maior número de usuário já cadastrado no Sistema de Segurança.

Após a operação de inclusão de usuário, será automaticamente encadeada a operação de habilitação de aplicativos. Isto deve-se ao fato de que um usuário só consegue efetuar logon caso o seu cadastramento esteja completo, ou seja, com as especificações de quais aplicativos e transações ele estará autorizado a trabalhar.

Porém, se o usuário recém criado possuir nível de acesso 99, não será encadeada a transação de habilitação de aplicativos, pois o Sistema de Segurança assumirá, automaticamente, que ele estará habilitado a trabalhar com todos os aplicativos e transações.

Todo usuário de nível de acesso 99 está habilitado a executar qualquer operação do Sistema de Segurança, inclusive a inclusão de usuário. Porém, o usuário de nível de acesso 98, para que possa efetuar a inclusão de um usuário, deve atender determinados requisitos, que se encontram relacionados nesta seção, no item "Usuários e habilitações", tópico "Introdução", subtópico "Responsáveis pelos cadastramentos".

Para incluir um usuário comum ou perfil modelo, deve-se utilizar o menu de operação "CADUSU", operação "INC".

CAMPOS DA TELA

Numero - Número do usuário, caso tenha sido especificado o número do usuário a ser incluído.

Nome - Nome do usuário.

Codigo - Código do usuário.

O código do usuário servirá, opcionalmente, para identificar o usuário junto ao Sistema de Segurança. Desta forma, no momento do logon ou em qualquer operação a ser efetuada internamente ao Sistema de Segurança, tanto o número quanto o código do usuário serão aceitos como seus identificadores.

O código e o número devem ser únicos e específicos para cada usuário.

O código do usuário que efetuou o logon fica disponível aos aplicativos através do campo global GBL-USUCODEXT, podendo estes utilizá-lo ou não.

Quando houver integração do logon do usuário com o Marc, seu código externo será obrigatório e deverá ser idêntico ao seu usercode no Marc.

Livre - Dado livre a respeito do usuário.

Este dado deve ser informado de acordo com a necessidade e conveniência de integração com outros aplicativos ou de identificação do usuário.

Este dado poderá ser pesquisado posteriormente por outros aplicativos, através de acesso à sub-rotina SBTS05, que é responsável pelas consultas aos dados dos usuários.

Sigla da Empresa - Sigla da empresa do usuário.

Esta sigla deve ter sido cadastrada anteriormente na tabela SBPM6001 - Tabela de Empresas, conforme descrito na Seção V, "Parametrizações do SADS".

A sigla da empresa do usuário que efetuou o logon fica disponível aos aplicativos através do campo global GBL-EMPSIGLA, podendo estes utilizá-la ou não.

Area - Código da área do usuário.
Este código deve ter sido cadastrado anteriormente na tabela SBPM6006 - Tabela de Áreas, associado à sigla da empresa informada, conforme descrito na Seção V, "Parametrizações do SADS".

Ambiente - Código do ambiente no qual o usuário poderá obter acesso.

Pode assumir os seguintes valores:

"T" - Quando o usuário puder obter acesso somente ao ambiente de teste;

"P" - Quando o usuário puder obter acesso somente ao ambiente de produção;

"A" - Quando o usuário puder obter acesso a ambos os ambientes, teste e produção.

Uma vez determinado um ambiente específico ("P" ou "T") para o usuário, este só terá acesso aos aplicativos habilitados a ele e que estiverem compilados para o ambiente em questão.

O ambiente no qual o usuário efetuou o logon fica disponível aos aplicativos através do campo global GBL-USUAMBI, podendo estes utilizá-lo ou não.

Nivel de acesso - Poder de acesso de um usuário comum, definido para o Sistema de Segurança.

Pode assumir os seguintes valores:

"99" - Nível de acesso que identifica o administrador de segurança.

Um usuário com nível de acesso 99 é implicitamente habilitado a realizar qualquer operação dentro do Sistema de Segurança. Não será então necessário definir para ele habilitações de aplicativos, transações e elementos, pois será automaticamente habilitado a todos os aplicativos (" " - branco) com tipo de habilitação "2" e a todas as transações. Também será habilitado automaticamente a todos os elementos (LISTA DE ELEMENTOS: " " - ELEMENTO: " " - TIPO HAB: 2).

"98" - Nível de acesso que identifica os usuários habilitados, pelo administrador de segurança, a realizar operações no Sistema de Segurança, dentro de limites estabelecidos pelas suas próprias habilitações.

Estes usuários são também denominados subadministradores de segurança.

"97" - Nível de acesso que identifica o preparador de serviços no Sistema de Controle de Processamento (SPO).

Níveis inferiores a "97" - Níveis dos usuários que não podem efetuar manutenções no Sistema de Segurança, podendo ser utilizados de acordo com os critérios dos aplicativos.

Caso o usuário seja perfil modelo, o nível de acesso é um dado não especificável.

Caso o nível de acesso de um usuário comum não seja informado, será assumido nível "00" (zero).

O nível de acesso do usuário que efetuou o logon fica disponível aos aplicativos através do campo global GBL-USUNIVACES, podendo estes utilizá-lo ou não.

O nível de acesso do usuário pode também ser definido por aplicativo. Desta forma, para cada aplicativo habilitado ao usuário, pode-se estabelecer qual o seu nível de acesso específico, que poderá ser testado pelo próprio aplicativo.

Maiores esclarecimentos a respeito dos níveis de acesso dos usuários encontram-se descritos no item "Níveis de segurança disponíveis", nesta seção.

Integrado com o Marc - Indicador de que o logon do usuário será efetuado através de integração com o Marc.

Caso seja definido que o logon do usuário será integrado com o Marc, então não lhe será exigido o preenchimento da tela de habilitação para início de uma sessão SADS. Neste caso o logon será disparado automaticamente pelo próprio Sistema de Segurança, assumindo os seguintes valores para os campos da tela de logon:

O código do usuário será igual ao usercode por ele informado no Marc ;

Através deste código é então identificado pelo SADS/SS o número de usuário;

A senha não será exigida;

O ambiente e as unidades autorizadas serão obtidos do cadastro do usuário, caso sejam específicas, ou, caso contrário, dos defaults cadastrados para o usuário, que, neste caso, para integração com o Marc, obrigatoriamente devem ter sido cadastrados.

Se o usuário possuir logon integrado com o Marc e desejar efetuar logon em um ambiente e unidades diferentes daqueles assumidos, então ele deve utilizar o menu de operação "HAB" para disparar novo logon com os valores desejáveis. Neste caso ele necessita então informar a senha.

Caso o usuário seja perfil modelo, o indicador de logon integrado com o Marc é um dado não especificável.

Validade - Data final para a validade do cadastro do usuário.

Caso tenha sido estabelecida uma data de validade para o usuário, em qualquer data igual ou posterior a esta, o usuário será impedido de efetuar logon e, conseqüentemente, não conseguirá trabalhar com os aplicativos SADS que exigirem segurança.

Após o vencimento do cadastro do usuário, seu status será alterado para "bloqueado".

Caso o usuário seja perfil modelo, a validade é um dado não especificável.

Caso a validade não seja informada para um usuário comum, seu cadastramento terá validade ilimitada.

Alteracao de senha - Código do controle de exigência de troca de senha do usuário.

Pode assumir os seguintes valores:

"0" - Para que seja exigida troca periódica da senha do usuário, após aproximadamente 60 dias de sua criação ou da última troca realizada.

Haverá, ainda, um período de tolerância de mais dez dias para que o usuário efetue a troca de senha, sendo que, findo esse prazo, o usuário ficará impedido de efetuar logon.

Caso isto ocorra, somente o administrador de segurança ou o subadministrador, se devidamente habilitado, poderá trocar sua senha e liberar seu acesso.

"1" - Para que seja permitida a troca de senha pelo usuário, mas não seja exigida.

"2" - Para que não seja permitida a troca de senha pelo usuário.

Caso o usuário seja perfil modelo, o código de alteração de senha é um dado não especificável.

Caso este campo não seja informado para um usuário comum, será assumido o código "0" e será então exigida a troca periódica de sua senha .

Caso o código do controle seja "0" ou "1", o usuário poderá trocar sua senha a qualquer momento, independentemente do Sistema de Segurança exigir ou não tal troca.

Pode ser cadastrado um valor default para o campo "Alteracao de senha", que deverá ser assumido por todos os usuários cadastrados na instalação. Este valor pode ser informado no campo PAR-6, CODIGO igual a SISSEG, da tabela SBPM5000. Caso tenha sido feita esta parametrização, todos os usuários a serem incluídos ou alterados deverão possuir igual valor no campo "Alteração de senha". Detalhes sobre esta parametrização podem ser encontrados na Seção V, "Parametrizações do SADS", item "Segurança, subitem "Alteração de senhas", deste manual.

Dias Habilitados - Indicador de controle de dias nos quais o usuário está autorizado a efetuar logon.

Pode assumir os seguintes valores:

"0" - Sem restrições quanto aos dias nos quais o usuário poderá efetuar logon;

"1" - O usuário não poderá efetuar logon aos sábados e domingos;

"2" - O usuário não poderá efetuar logon aos domingos;

"3" - O usuário não poderá efetuar logon nos feriados;

"4" - O usuário não poderá efetuar logon aos sábados, domingos e feriados;

"5" - O usuário não poderá efetuar logon aos domingos e feriados.

Caso o usuário seja perfil modelo o indicador de controle de dias habilitados é um dado não especificável.

Caso este campo não seja informado para um usuário comum, será assumido que não existem restrições quanto aos dias nos quais o usuário poderá efetuar logon.

Unidades Autorizadas:

As unidades dos usuários - empresa, estabelecimento e unidade - devem ser definidas respeitando-se a hierarquia da instalação, considerando que uma empresa é formada por vários estabelecimentos, que por sua vez possuem várias unidades.

Antes então que se cadastre qualquer usuário, deve-se definir quais os códigos de empresas, estabelecimentos e unidades deverão ser definidos na instalação.

O cadastramento das unidades autorizadas de um usuário irá estabelecer em quais unidades este obterá autorização de acesso.

Sempre que um usuário iniciar uma sessão SADS, serão identificadas as unidades nas quais ele irá trabalhar especificamente na sessão por ele aberta.

Estas unidades ficam disponíveis para utilização pelos aplicativos através dos campos globais GBL-EMPCOD, GBL-ETBCOD e GBL-UNDCOD.

Os aplicativos poderão então utilizar estas unidades para, por exemplo, selecionar os dados a serem exibidos ou manipulados pelo usuário.

Empresa - Código da empresa na qual o usuário terá autorização de acesso.

Pode assumir os seguintes valores:

"999" - Para indicar que o usuário poderá ter acesso a qualquer empresa.

No momento do logon, caso o usuário não possua empresa default cadastrada, será exigido então que ele informe em qual empresa ele deseja obter acesso naquela sessão de trabalho. Caso contrário, será assumida a empresa default.

"nnn" - Para indicar que o usuário tem acesso somente a uma empresa específica.

Podem ser informados valores de "001" a "998" que, caso utilizados, devem estar cadastrados na tabela SBPM6001, conforme descrito na Seção V, "Parametrizações do SADS", item "Empresas".

No momento do logon, não será exigido que o usuário informe a empresa na qual deseja trabalhar, porque a empresa especificada em seu cadastramento sempre será assumida, tendo em vista ser a única na qual o usuário poderá obter acesso.

"000" - Para indicar que o usuário não tem vínculo a nenhuma empresa.

Neste caso, se o usuário solicitar uma transação de um aplicativo que trata as informações de acordo com a empresa na qual o usuário efetuou o logon, então não lhe será permitido, nesta transação, manipular ou consultar nenhuma informação.

No momento do logon, não lhe será exigida a informação da empresa na qual deseja trabalhar, pois será assumida a empresa "000".

Caso o código da empresa não seja informado, será assumida, internamente ao Sistema de Segurança, empresa "000".

Estabelecimento - Código do estabelecimento no qual o usuário terá autorização de acesso.

Pode assumir os seguintes valores:

"9999" - Para indicar que o usuário poderá ter acesso a qualquer estabelecimento.

No momento do logon, caso o usuário não possua estabelecimento default cadastrado, será exigido então que ele informe em qual estabelecimento ele deseja, naquela sessão de trabalho, obter acesso. Caso contrário, será assumido o estabelecimento default.

"nnnn" - Para indicar que o usuário tem acesso somente a um estabelecimento específico.

Podem ser informados valores de "0001" a "9998".

No momento do logon, não será exigido que o usuário informe o estabelecimento no qual deseja trabalhar, porque o estabelecimento especificado em seu cadastramento será sempre assumido, tendo em vista ser o único no qual o usuário poderá obter acesso.

"0000" - Para indicar que o usuário não tem vínculo a nenhum estabelecimento.

Neste caso, se o usuário solicitar uma transação de um aplicativo que trata as informações de acordo com o estabelecimento no qual o usuário efetuou o logon, então, nesta transação, não lhe será permitido manipular ou consultar nenhuma informação.

Não lhe será exigida a informação do estabelecimento no qual deseja trabalhar, no momento do logon, pois será assumido o estabelecimento "0000".

Caso o código do estabelecimento não seja informado, será assumido, internamente ao Sistema de Segurança, estabelecimento "000".

Unidade - Código da unidade na qual o usuário terá autorização de acesso.

Pode assumir os seguintes valores:

"999" - Para indicar que o usuário poderá ter acesso a qualquer unidade.

Caso o usuário não possua unidade default cadastrada, no momento do logon, será exigido então que ele informe em qual unidade ele deseja obter acesso naquela sessão de trabalho. Caso contrário, será assumida a unidade default.

"nnn" - Para indicar que o usuário tem acesso somente a uma unidade específica.

Podem ser informados valores de "001" a "998".

No momento do logon, não será exigido que o usuário informe a unidade na qual deseja trabalhar, pois a unidade especificada em seu cadastramento será sempre assumida, tendo em vista ser a única que o usuário poderá acessar.

"000" - Para indicar que o usuário não tem vínculo a nenhuma unidade.

Neste caso, se o usuário solicitar uma transação de um aplicativo, que trata as informações de acordo com a unidade na qual o usuário efetuou o logon, então, nesta transação, não lhe será permitido manipular ou consultar nenhuma informação.

Não lhe será exigida a informação da unidade na qual deseja trabalhar, no momento do logon, pois será assumida a unidade "000".

Vinculação/desvinculação de usuário a perfis

O usuário designado pelo SADS/SS como perfil modelo é aquele que possui dados comuns a um grupo de usuários.

Pessoas de uma mesma equipe, que comumente trabalham com os mesmos aplicativos, executando funções similares, possuem dados semelhantes em seus cadastramentos e habilitações. Nesta situação, é aconselhável que se crie um perfil modelo, contendo esses dados comuns, e o relacione a todos os usuários pertencentes à equipe, através da operação de vinculação de usuário a perfil.

A vantagem de se criar um perfil modelo é que todas as características e habilitações, comuns aos usuários, são cadastradas uma única vez para o perfil, sendo herdadas por todos os usuários a ele vinculados. Qualquer alteração nestas habilitações passará a valer também para os usuários vinculados, após efetuar suas liberações.

Para vincular um usuário a um perfil modelo, é necessário que o perfil já esteja cadastrado.

Um usuário pode ser vinculado a um ou mais perfis modelo, de acordo com as funções afins que compartilha com tais perfis.

No momento da vinculação é estipulada uma prioridade para a vinculação, a fim de que, havendo coincidência ou discordância nas habilitações a serem herdadas pelo usuário, caso esteja vinculado a mais de um perfil, seja então respeitada a prioridade destas vinculações.

Todo usuário, antes que possa efetuar logon e após ter sido finalizado todo o seu cadastramento, deve ser submetido à operação de liberação. Através deste processo é que são sintetizadas suas habilitações que foram cadastradas especificamente para o mesmo ou que foram herdadas dos vários perfis modelo aos quais ele foi porventura vinculado.

Se houver discordância entre as habilitações do usuário e dos perfis aos quais ele estiver vinculado, a prioridade será :

  1. Habilitações dos aplicativos do usuário; e depois
  2. Habilitações dos aplicativos dos perfis, em ordem da prioridade de cadastramento.

Na liberação do usuário, as discordâncias de unidades autorizadas do usuário e dos vários perfis vinculados serão resolvidas da seguinte forma:

Se o usuário possuir unidades autorizadas específicas ou 0's :

Vai herdar as unidades do usuário e estas não poderão ser informadas na operação de vinculação a perfil, desde que exatamente iguais às unidades do usuário.

Se o usuário possuir unidades autorizadas igual a 9's e o perfil possuir unidades específicas e diferentes de 0's:

Vai herdar as unidades do perfil, porém estas poderão ser informadas na operação de vinculação a perfil.

Se o usuário e o perfil possuírem unidades autorizadas igual a 9's :

As unidades poderão ser informadas na operação de vinculação, caso sejam específicas (nnn). Neste caso, prevalecerão as unidades informadas na vinculação.

Se as unidades não forem informadas no momento da vinculação, os aplicativos serão habilitados com `9's .

Se as unidades autorizadas do usuário forem iguais a 9's e as do perfil forem iguais a 0's:

As unidades poderão ser informadas na operação de vinculação, caso sejam específicas (nnn). Neste caso, a unidade informada será sobreposta em todas as habilitações de aplicativos do perfil.

Se as unidades não forem informadas no momento da vinculação, os aplicativos serão habilitados com `9's (que é a definição do usuário).

Este procedimento será sempre efetuado quando houver incompatibilidade entre as unidades autorizadas do perfil e do usuário.

A cada vez que é alterado o cadastramento do usuário - unidades autorizadas ou habilitações - ou forem alterados os perfis aos quais ele está vinculado, deve-se então proceder à sua liberação.

Um usuário vinculado pode ter habilitações próprias de aplicativos, transações e elementos, diferentes daquelas cadastradas para os perfis a ele relacionados. Desta forma, é possível redefinir - restringir ou ampliar - as habilitações que um usuário herda de seus perfis modelo.

Um usuário vinculado pode ser desligado de qualquer um de seus perfis a qualquer momento.

O usuário a ser vinculado a um perfil pode ou não estar cadastrado. Se ainda não tiver sido cadastrado, será exibida a tela para definição de seus dados próprios e, a seguir, será encadeada a tela para vinculação do usuário aos perfis modelo desejados.

Qualquer alteração efetuada nas habilitações de um perfil modelo será herdada por todos os usuários a ele vinculados.

Após desvinculado, o usuário perde todas as habilitações herdadas do perfil modelo. A partir deste momento, portanto, nenhuma das modificações realizadas no cadastramento do perfil será herdada pelo usuário.

O usuário poderá ser vinculado a vários perfis. Para cada perfil a ele vinculado, haverá a possibilidade de se cadastrar, opcionalmente, unidades autorizadas, tipo da habilitação e nível de acesso, que serão sobrepostos em todos as habilitações de aplicativos do perfil, para aquele usuário, observando as regras da liberação.

Tanto o perfil quanto o usuário poderão possuir listas de habilitações de elementos. A prioridade destas habilitações funciona como a de unidades autorizadas. Se existirem elementos habilitados iguais para o usuário e para os perfis, valerão primeiramente as habilitações de elemento do usuário e, depois, os elementos dos perfis, em ordem de prioridade.

Habilitações de usuário

As habilitações a serem definidas para os usuários do Sistema de Segurança são:

Habilitações de aplicativos: que irão determinar com quais aplicativos os usuários terão autorização para trabalhar;

Habilitações de transações: que irão determinar com quais transações dos aplicativos anteriormente habilitados os usuários poderão obter acesso; e

Habilitações de elementos: que irão determinar com quais elementos (valores de campos) os usuários poderão trabalhar em um aplicativo ou lista de elementos.

As habilitações de elementos não são obrigatórias e somente se deve proceder aos seus cadastramentos para aqueles aplicativos que efetuarem o teste de habilitação de elemento.

Embora seja possível não habilitar aplicativos, transações e elementos ao usuário, no momento de sua criação, é obrigatório que estas habilitações tenham sido definidas em algum momento, para que ele possa trabalhar no ambiente SADS.

Unidades das habilitações

Todas as habilitações dos usuários (aplicativos, transações e elementos), definidas no Sistema de Segurança, são cadastradas em nível de empresa, estabelecimento e unidade.

As unidades autorizadas ao usuário e as unidades nas quais o aplicativo, transação ou elemento será habilitado devem ser sempre compatíveis, da seguinte forma:

Se o usuário estiver autorizado a trabalhar em diferentes empresas (Empresa autorizada = "999"), as habilitações podem ser definidas para todas as empresas ("999") ou para uma empresa específica ("nnn").

Este procedimento também é válido para o estabelecimento e a unidade.

Se um usuário tem acesso a somente uma empresa específica (Empresa autorizada = "nnn"), então só é permitido cadastrar habilitações para esta mesma empresa ("nnn").

Mesmo procedimento se aplica a estabelecimento e unidade autorizados.

Se um usuário tem acesso somente a sistemas que não estão ligados a nenhuma empresa (Empresa autorizada = "000"), então só é permitido habilitar aplicativos, transações e elementos para a empresa "000".

Mesmo procedimento se aplica a estabelecimento e unidade autorizados.

No momento em que o usuário estiver trabalhando em uma sessão SADS, o aplicativo, transação ou elemento só estará habilitado ao usuário caso a empresa, estabelecimento e unidade informados no início da sessão de trabalho sejam compatíveis àqueles especificados na habilitação.

Testes das habilitações

O Sistema de Segurança faz a validação da habilitação de aplicativos segundo os códigos de transação das telas de processamento ou de menus 01 a 09. Para o SADS/SS, portanto, os códigos de transação dos menus de escolha ou de operação não podem ser utilizados para definir as habilitações dos usuários. Porém, apenas os códigos de menu (sssmm) dos menus de escolha e operação devem ser habilitados aos usuários para que eles possam utilizar a navegação.

Habilitação de aplicativos

Através da operação de habilitação de aplicativos é possível determinar com quais aplicativos o usuário está autorizado a trabalhar e de qual maneira ele está autorizado a trabalhar: somente consultar, consultar e atualizar ou não permite trabalhar.

Habilitação de elementos

O Sistema de Segurança permite aos aplicativos validar se um usuário pode ou não executar o acesso a um elemento específico.

O elemento a ser testado depende da necessidade do aplicativo, podendo ser qualquer informação, interna ao mesmo, que se queira validar a segurança de acesso. Daí o elemento não possuir padrão de codificação, podendo conter até 17 posições e em qualquer formato de edição (numérico, alfabético ou alfanumérico).

Utilitários do cadastramento de usuários

Duplicação de usuário/perfil

Esta operação propicia a criação de um usuário com as mesmas características e habilitações de um outro usuário já cadastrado.

Após a duplicação de um usuário, pode-se optar por encadear a tela de alteração do usuário recém-criado, para adequar suas características e/ou habilitações.

O administrador de segurança pode efetuar a duplicação de quaisquer usuários.

Um subadministrador de segurança só poderá efetuar a duplicação, se todos os aplicativos habilitados ao usuário a ser duplicado existirem cadastrados como elementos, com tipo de habilitação igual a "2", na lista de elementos do "SBT" ou "SBT01". Se o usuário a ser duplicado possuir também nível de acesso 98, então este tipo de habilitação deve ser "9", ao invés de "2", e o novo usuário só será habilitado aos aplicativos que estiverem cadastrados para o usuário a ser duplicado com tipo de habilitação "9". Tais habilitações serão incluídas para o novo usuário com tipo de habilitação "2".

Para duplicar um usuário, deve-se utilizar o menu de operação "UTIUSU", operação "DUP".

Alteração de unidades autorizadas

Todas as habilitações dos usuários (aplicativos, transações e elementos), definidas no Sistema de Segurança, são cadastradas em nível de empresa, estabelecimento e unidade.

As unidades autorizadas ao usuário e as unidades nas quais os aplicativos, transações e elementos lhe são habilitados devem ser sempre compatíveis, da seguinte forma:

Se o usuário estiver autorizado a trabalhar em diferentes empresas (Empresa autorizada = "999"), as habilitações podem ser definidas para todas as empresas ("999") ou para uma empresa específica ("nnn").

Este procedimento também é válido para o estabelecimento e a unidade.

Se um usuário tem acesso a uma empresa específica (Empresa autorizada = "nnn"), então só é permitido cadastrar habilitações para esta mesma empresa ("nnn").

Mesmo procedimento se aplica a estabelecimento e unidade autorizados.

Se um usuário tem acesso somente a sistemas que não estão ligados a nenhuma empresa (Empresa autorizada = "000"), então só é permitido habilitar aplicativos, transações e elementos para a empresa "000".

Mesmo procedimento se aplica a estabelecimento e unidade autorizados.

Caso haja a necessidade de se alterar as unidades autorizadas de uma habilitação, as novas unidades devem continuar mantendo compatibilidade com as unidades autorizadas ao usuário. Esta situação pode levar a um impasse: para alterar as unidades da habilitação deve-se anteriormente alterar as unidades autorizadas ao usuário visando manter a compatibilidade. Porém, antes de se alterar as unidades autorizadas, deve-se adequar as unidades das habilitações, para que estas não se tornem incompatíveis.

Para solucionar este impasse, a operação de alteração de unidades autorizadas processa, de uma só vez, a alteração das unidades autorizadas ao usuário e a alteração das unidades das habilitações, evitando-se assim possíveis incompatibilidades.

Para se processar tal operação, devem ser informadas quais são as unidades atuais a serem trocadas e quais as novas unidades.

Também é possível, ao invés de trocar o código das unidades autorizadas, duplicá-las para os novos códigos. Desta forma as habilitações cadastradas nas antigas unidades serão mantidas e serão criadas iguais habilitações nas novas unidades.

Esta operação não será permitida se o número de aplicativos a serem duplicados ultrapassar o limite de 500 por usuário.

Cumpre ressaltar que, na tentativa de executar a operação de troca ou duplicação, as unidades autorizadas ao usuário e as unidades das habilitações dos aplicativos, transações e elementos devem manter sempre a compatibilidade. Desta forma, nenhuma alteração será efetuada caso haja tentativa de se quebrar tal regra.

Respeitando a compatibilidade referida anteriormente, todas as habilitações que forem encontradas cadastradas nas mesmas unidades atuais serão então trocadas ou duplicadas para as novas unidades e, caso as unidades autorizadas do usuário sejam semelhantes às unidades de origem especificadas, então elas também serão trocadas (caso a operação seja de troca).

Resumidamente, os seguintes testes serão efetuados no momento da troca ou duplicação das unidades autorizadas de um usuário:

As novas unidades deverão ser diferentes das unidades atuais;

Um usuário de nível de acesso 98 só poderá processar a operação caso as novas unidades sejam compatíveis às suas;

Se alguma das unidades autorizadas do usuário for igual a 9's ("999"), então a relativa unidade atual deverá ser informada;

Se alguma das unidades autorizadas do usuário for diferente de 9's ("999"), então a relativa unidade atual deverá ser igual à unidade autorizada;

Se uma das unidades autorizadas do usuário for igual a 9's ("999") e a nova unidade for igual a 0's ("000"), então a operação só será processada caso não haja nenhuma habilitação na referida unidade com valor diferente de 9's.

Se todas as unidades autorizadas forem diferentes de 9's, então não é necessário informar unidades atuais, pois serão assumidas as autorizadas. Caso as unidades atuais sejam informadas, elas devem ser iguais às autorizadas.

Se todas as unidades autorizadas forem diferentes de 9's, então não é permitido executar a operação de duplicação, somente troca.

Só será permitido trocar ou duplicar as unidades, caso não exista nenhuma habilitação que, pelo fato de não possuir unidades iguais às informadas, tenha unidades que se tornarão incompatíveis com as novas unidades do usuário.

As unidades das habilitações não serão alteradas caso sejam diferentes das unidades atuais ou já sejam iguais às novas unidades.

Se a operação for de duplicação, então as unidades autorizadas do usuário não serão alteradas.

Se todas as unidades autorizadas do usuário forem iguais a 9's e todas as novas unidades forem diferentes de 0's, então as unidades autorizadas do usuário não serão alteradas.

Se a unidade autorizada for diferente de 9's ou, se a unidade autorizada for igual a 9's e a unidade destino for igual a 0's, então a referida unidade autorizada será alterada.

Unidades Atuais

Novas Unidades

Empresa - Código da empresa na qual o usuário terá autorização de acesso.

Pode assumir os seguintes valores:

"999" - Para indicar que o usuário poderá ter acesso a qualquer empresa.

No momento do logon, caso o usuário não possua empresa default cadastrada, será exigido então que ele informe em qual empresa ele deseja obter acesso naquela sessão de trabalho. Caso contrário, será assumida a empresa default.

"nnn" - Para indicar que o usuário tem acesso somente a uma empresa específica.

Podem ser informados valores de "001" a "998", que devem estar cadastrados na tabela SBPM6001, conforme descrito na Seção V, "Parametrizações do SADS", item "Empresas".

No momento do logon, não será exigido que o usuário informe a empresa na qual deseja trabalhar, porque a empresa especificada em seu cadastramento será sempre assumida, tendo em vista ser a única empresa na qual o usuário poderá acessar.

"000" - Para indicar que o usuário não tem vínculo a nenhuma empresa.

Neste caso, se o usuário solicitar uma transação de um aplicativo que trata as informações de acordo com a empresa na qual o usuário efetuou o logon, então, nesta transação, não lhe será permitido manipular ou consultar nenhuma informação.

No momento do logon, não lhe será exigida a informação da empresa na qual deseja trabalhar, porque será assumida a empresa "000".

Caso o código da empresa não seja informado, será assumido, internamente ao Sistema de Segurança, empresa "000".

Estabelecimento - Código do estabelecimento no qual o usuário tem autorização de acesso.

Pode assumir os seguintes valores:

"9999" - Para indicar que o usuário poderá ter acesso a qualquer estabelecimento.

No momento do logon, caso o usuário não possua estabelecimento default cadastrado, será exigido então que ele informe em qual estabelecimento ele deseja obter acesso naquela sessão de trabalho. Caso contrário, será assumido o estabelecimento default.

"nnnn" - Para indicar que o usuário tem acesso somente a um estabelecimento específico.

Podem ser informados valores de "001" a "9998".

No momento do logon, não será exigido que o usuário informe o estabelecimento no qual deseja trabalhar, porque o estabelecimento especificado em seu cadastramento será sempre assumido, tendo em vista ser o único estabelecimento que o usuário poderá acessar.

"0000" - Para indicar que o usuário não tem vínculo a nenhum estabelecimento.

Neste caso, se o usuário solicitar uma transação de um aplicativo que trata as informações de acordo com o estabelecimento no qual o usuário efetuou o logon, então, nesta transação, não lhe será permitido manipular ou consultar nenhuma informação.

No momento do logon, não lhe será exigida a informação do estabelecimento no qual deseja trabalhar, pois será assumido o estabelecimento "0000".

Caso o código do estabelecimento não seja informado, será assumido, internamente ao Sistema de Segurança, estabelecimento "0000".

Unidade - Código da unidade na qual o usuário terá autorização de acesso.

Pode assumir os seguintes valores:

"999" - Para indicar que o usuário poderá ter acesso a qualquer unidade.

No momento do logon, caso o usuário não possua unidade default cadastrada, será exigido então que ele informe em qual unidade ele deseja obter acesso naquela sessão de trabalho. Caso contrário, será assumida a unidade default.

"nnn" - Para indicar que o usuário tem acesso somente a uma unidade específica.

No momento do logon, não será exigido que o usuário informe a unidade na qual deseja trabalhar, porque a unidade especificada em seu cadastramento será sempre assumida, tendo em vista ser a única unidade que o usuário poderá acessar.

"000" - Para indicar que o usuário não tem vínculo a nenhuma unidade.

Neste caso, se o usuário solicitar uma transação de um aplicativo que trata as informações de acordo com a unidade na qual o usuário efetuou o logon, então, nesta transação, não lhe será permitido manipular ou consultar nenhuma informação.

No momento do logon, não lhe será exigida a informação da unidade na qual deseja trabalhar, pois será assumida a unidade "000".

Caso o código da unidade não seja informado, será assumido, internamente ao Sistema de Segurança, unidade "000".

Deve existir uma coerência na informação das unidades autorizadas de um usuário, da seguinte forma:

Se um usuário pode ter acesso às informações de todas as empresas (999), conseqüentemente, não pode ser autorizado a um estabelecimento específico (nnnn) e/ou unidade específica (nnn);

Se um usuário pode ter acesso às informações de todos os estabelecimentos (9999), conseqüentemente, não pode ser autorizado a uma unidade específica (nnn), independentemente da empresa ser específica (nnn) ou genérica (999);

Se um usuário não tem acesso a nenhuma empresa (000), então também não poderá ter acesso a nenhum estabelecimento (0000) e unidade (000);

Se um usuário não tem acesso a nenhum estabelecimento (0000), então também não poderá obter acesso a nenhuma unidade (000);

As demais combinações das unidades autorizadas são válidas.

"T" - Para trocar as unidades autorizadas do usuário e as unidades de suas habilitações, caso satisfeitos os testes anteriormente relacionados.

Pode assumir os seguintes valores:

"S" - Para confirmar a operação;

"N" - Para cancelar a operação.

Exclusão de usuários que não efetuaram logon no período

Através desta operação, é possível excluir os usuários que não efetuaram logon nos últimos dias. Esta quantidade de dias, utilizada para pesquisar os usuários a serem excluídos, deverá ser informada no momento do disparo da operação.

Somente um administrador de segurança poderá proceder esta exclusão de usuários da base de dados do Sistema de Segurança.

Os usuários que não efetuaram logon durante os últimos dias serão anteriormente exibidos ao administrador de segurança, e este poderá então selecionar, informando "S" na linha referente a cada usuário, aqueles que realmente deverão ser excluídos, ou poderá selecionar, informando "N", aqueles que deverão permanecer cadastrados na base de dados do Sistema de Segurança.

Também serão exibidos, para possível exclusão, aqueles usuários que ainda não efetuaram nenhum logon e cuja data de cadastramento seja inferior à data calculada a partir do número de dias informado.

Para excluir usuários que não efetuaram logon no período, deve-se utilizar o menu de operação "UTIUSU", operação "EXL".

Transformação de usuário em perfil

Esta operação propicia a alteração do tipo do usuário, permitindo que um usuário já cadastrado (tipo = "U") se transforme em perfil modelo (tipo = "P").

O usuário a ser transformado não pode estar vinculado a nenhum perfil.

Somente o administrador de segurança poderá efetuar tal transformação.

Para transformar um usuário em perfil modelo, deve-se utilizar o menu de operação "UTIUSU", operação "TRA", e informar o número ou código do usuário.

CAMPOS DA TELA

Serão apresentados todos os dados cadastrais do usuário, informados no momento de seu cadastramento, para que se possa certificar do usuário a ser transformado.

Pode assumir os seguintes valores:

"S" - Para confirmar a transformação do usuário em perfil;

"N" - Para cancelar a operação.
