Banco de dados

Conceituação

Bancos de dados são conjuntos de estruturas onde os dados dos sistemas aplicativos ficam armazenados. Estes dados podem permanecer distribuídos em vários arquivos, para os quais são permitidos acessos simultaneos por diferentes aplicações.

Os acessos a estes arquivos são controlados por um SGBD (Sistema Gerenciador de Banco de Dados), portanto, os programas aplicativos não têm acesso direto a eles, podendo ser atualizados somente através deste gerenciador.

O SADS suporta a construção de bancos de dados para o SGBD DMSII - Data Management System II - da Unisys. Portanto, o banco de dados deve ser definido no repositório de dados do SADS de acordo com as particularidades desse software.

Maiores informações relativas ao DMSII poderão ser obtidas no "DMSII - Data and Structure Definition Language (DASDL) - Programming Reference Manual" da Unisys.

Padronização

O padrão de codificação adotado pelo SADS para banco de dados é o seguinte:

"sssBxx" onde:

"sss" é o código alfabético que identifica o sistema;

"B" é a constante de estrutura DMSII,

"xx" é o identificador alfanumérico do bando de dados.

Como o padrão de codificação de banco de dados é semelhante ao de dataset ou remap, no momento da definição do banco no repositório de dados, o código informado só será válido se não tiver sido anteriormente utilizado para identificar um dataset ou remap.

Utilização

Um banco de dados pode ser utilizado por um conjunto de programas aplicativos com a finalidade de registrar e manter informações dos sistemas.

O banco de dados, após ter sido definido no repositório de dados do SADS, pode ter seu fonte DASDL (Data and Structure Definition Language) gerado. Também poderá ser referenciado em bases de dados e em hosts.

Cadastramento

O processo de definição de um banco de dados só é possível a partir do cadastramento do sistema e dos datasets e remaps que irão compô-lo no repositório de dados do SADS.

Não é necessário que cada sistema tenha os seus bancos de dados específicos. Um banco de dados pode ser compartilhado por vários sistemas e um sistema pode ter acesso a mais de um banco de dados. Porém, por recomendação do DMSII, somente um banco de dados pode ser aberto como update por um programa .

Roteiro

As funções relacionadas ao cadastramento de um banco de dados são as seguintes :

Definição de banco de dados;

Cadastramento de parâmetros e defaults globais ;

Cadastramento de opções globais;

Cadastramento de softwares de banco;

Cadastramento de audit e control file;

Cadastramento de itens globais;

Cadastramento de especificações físicas globais;

Geração de fonte DASDL de banco.

Definição de banco de dados

CAMPOS DA TELA

BANCO - Código do banco de dados no padrão SADS (sssBxx).

NOME - Descrição resumida do banco de dados.

O nome tem o objetivo de identificar, de maneira resumida, a natureza das informações que serão nele armazenadas.

CODIGO EXTERNO - Código com o qual o banco de dados será referido externamente ao SADS.

Este código pode ser utilizado quando a instalação adotar um padrão de codificação diferente do padrão SADS.

É também utilizado para definir, no SADS, bancos de dados já existentes na instalação que não foram gerados pelo SADS, mas que serão tratados e/ou atualizados por programas por ele gerados.

Caso este campo seja informado, será utilizado em todas as gerações do SADS, isto é, o código externo substituirá o código no padrão SADS tanto no fonte DASDL do banco quanto nas libraries de declaração de bancos em que ele estiver referenciado.

TIPO - Classificação a ser adotada para o banco de dados.

Um banco de dados pode ter os seguintes tipos:

"L" (Lógico) - Um banco de dados lógico é tratado no SADS como um banco composto exclusivamente por remaps, que são visões lógicas de datasets;

"F" (Físico) - Um banco de dados físico é tratado no SADS como um banco composto exclusivamente por datasets.

Após a definição do banco, seu tipo não mais poderá ser alterado. Caso exista esta necessidade, o banco terá de ser excluído e incluído novamente. Vale lembrar que um banco de dados somente poderá ser excluído se não estiver relacionado a hosts ou bases de dados.

BANCO FISICO RELACIONADO - Código do banco de dados físico relacionado ao banco de dados lógico a ser definido (sssBxx)

Para definir um banco de dados lógico, é necessário informar o banco físico a que ele está relacionado, que, por sua vez, tem de estar cadastrado no repositório de dados do SADS.

AMBIENTE - Ambiente em que o banco será utilizado.

Os valores possíveis são os seguintes:

"T" (Teste);

"P" (Produção).

O banco de dados, quando definido para ser utilizado em ambiente de produção, deverá estar associado a um banco de teste já cadastrado. Este relacionamento será observado no momento da geração da library da base de dados, onde o banco gerado na library será aquele compatível com o ambiente para o qual ela foi gerada.

Após a definição do banco, seu ambiente não mais poderá ser alterado. Caso exista esta necessidade, o banco terá de ser excluído e incluído novamente. Vale lembrar que um banco de dados somente poderá ser excluído se não estiver relacionado a hosts ou bases de dados.

BANCO DE TESTE RELACIONADO - Código do banco de dados em teste relacionado ao banco de dados em produção a ser definido (sssBxn).

O banco de dados, quando definido para ser utilizado em ambiente de produção, deverá estar associado a um banco de teste já existente. Este relacionamento será observado no momento da geração da library da base de dados, onde o banco gerado na library será aquele compatível com o ambiente para o qual ela foi gerada.

O banco de teste informado tem de estar previamente definido no repositório de dados.

Um banco de dados de teste poderá estar relacionado a vários bancos de produção, porém, um banco de dados de produção somente poderá estar relacionado a um único banco de teste.

CODIGO DO RESTART DATASET - Código do restart dataset a ser utilizado pelo banco de dados, no padrão SADS (SBGBnn).

O restart dataset padrão utilizado em gerações do SADS é o "SBGB01". Podem ser utilizados também os códigos "SBGB03", "SBGB04". "SBGB05", "SBGB06" OU "SBGB07".

Para utilização de restart dataset com código diferente de "SBGB01", a tabela de opções de compilação, SBPM5018, deve ser utilizada, informando-se as opções "SADSRDS03" para uso do "SBGB03", "SADSRDS04" para uso do "SBGB04", "SADSRDS05" para uso do "SBGB05","SADSRDS06" para uso do "SBGB06" ou "SADSRDS07" para uso do "SBGB07". Detalhes a respeito da tabela SBPM5018 estão descritos na seção V, "Parametrizações do SADS", neste manual.

DATASETS - Código(s) do(s) dataset(s) ou remap(s) relacionado(s) ao banco de dados. Estes elementos devem ser informados no padrão SADS (sssBxn).

Na definição de bancos de dados físicos deve-se informar somente datasets. Nas definições de bancos de dados lógicos, somente remaps.

Os datasets ou remaps devem estar previamente definidos no repositório de dados do SADS para que possam ser relacionados ao banco.

Na definição de um banco de dados deve-se indicar pelo menos um dataset ou remap.

Não é necessário informar o restart dataset no campo DATASETS, pois ele já foi especificado no campo CODIGO DO RESTART DATASET.

CONTINUA(S/N) - Indicador de continuidade de operação.

Os valores possíveis são os seguintes:

"S" - Para prosseguir a definição de banco de dados;

"N" - Para encerrar a operação.

OPERACAO - Operação efetuada em relação ao banco de dados.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de parâmetros e defaults globais

Os parâmetros globais são informações necessárias para controlar as características das rotinas de acesso ao banco de dados em tempo de compilação e de execução.

Defaults globais são valores assumidos nas estruturas dos atributos de bancos de dados que não apresentarem declarações explícitas de algum valor para os mesmos.

Um acompanhamento periódico, feito por administradores de bancos dados, para ajustes dos valores declarados, geralmente garante uma boa performance e qualidade do banco de dados.

As informações relativas aos parâmetros e defaults globais no repositório de dados do SADS são utilizadas especificamente para geração do fonte DASDL do banco de dados.

CAMPOS DA TELA

BANCO - Código do banco de dados no padrão SADS (sssBxx).

DEFAULTS GLOBAIS:

INCORPORAR DEFAULTS DA INSTALACAO - Indicador de incorporação de defaults globais definidos para os bancos da instalação.

Os valores possíveis são os seguintes:

"S" - Para assumir os valores pré-definidos para os defaults globais que não forem
especificados para o banco de dados;

"N" - Para desconsiderar os valores pré-definidos para os defaults globais.

A pré-definição de defaults globais deve ser especificada na tabela SBPM5050. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY GERACAO DASDL - Código de library, no padrão SADS (sssLxx), que contém as especificações de defaults globais de bancos de dados que não estejam sendo contempladas na versão corrente do SADS.

BUFFERS - Número de buffers que o sistema irá alocar para a estrutura, mais o número de buffers adicionais que o sistema deverá alocar para cada programa que utilizá-la.

Os valores defaults são os seguintes: 

 System    Buffers         = 1
 Random    user    Buffers = 1
 Serial    user    Buffers = 0 se Reblock    = FALSE
                               2 se Reblock    = TRUE

O valor máximo permitido é 254 + 254 ou apenas 254.

CHECKSUM - Indicador de utilização da cláusula checksum.

Os valores possíveis são os seguintes:

"S" - Para utilização do checksum;

"N" - Para eliminar o uso do checksum (default do DMSII).

O checksum é utilizado para controlar erros de I/O, através da comparação de valores calculados na gravação e leitura de blocos de dados, podendo ser declarado em nível de estrutura.

SECURITYGUARD -Usercode, nome e localização física do arquivo guard, caso este seja utilizado.

O arquivo guard controla o acesso ao banco de dados pelos aplicativos. Um arquivo guard somente deverá ser definido se realmente tiver de ser utilizado, pois a definição aleatória destes arquivos pode causar "SECURITY ERROR" nos acessos ao banco de dados.

PACK - Disco onde os arquivos do banco de dados ficarão residentes.

REBLOCK - Indicador de utilização de tamanhos de blocos diferentes numa mesma estrutura.

Os valores possíveis são os seguintes:

"S" - Para reblocagem;

"N" - Sem reblocagem (default do DMSII).

O reblock é utilizado para manter, simultaneamente, dois tamanhos diferentes de blocos para uma mesma estrutura, com a finalidade de otimizar os acessos randômicos e seriais.

REBLOCKFACTOR - Fator que dimensiona os blocos quando estiverem sendo usados dois tamanhos diferentes. O maior bloco, para acessos seriais, é calculado como resultado da multiplicação do tamanho do bloco pelo valor reblockfactor.

Este campo deve ser informado apenas se o campo REBLOCK tiver sido definido com "S".

O valor máximo que ele pode assumir é 60 e o default é 1.

INITIALVALUE - Valores iniciais que os itens assumem na criação de registros.

Os valores iniciais válidos são apresentados na tabela a seguir:

Formato dos itens

Valores iniciais válidos

ALPHA

"B" (brancos)

"H"(high-values)

"L" (low-values)

BOOLEAN

"V" (verdadeiro)

"F" (falso)

REAL

"0" (zeros)

"H"(high-values)

"L" (low-values)

NUMBER

"0" (zeros)

"H"(high-values)

"L" (low-values)

PARAMETROS GLOBAIS:

INCORPORAR DEFAULTS DA INSTALACAO - Indicador de incorporação de defaults para parâmetros globais.

Os valores possíveis são os seguintes:

"S" - Para assumir os valores pré-definidos para os parâmetros globais que não forem especificados para o banco de dados;

"N" - Para desconsiderar os valores pré-definidos para os parâmetros globais.

A pré-definição de parâmetros globais deve ser especificada na tabela SBPM5050. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY GERACAO DASDL - Código de library, informado no padrão SADS (sssLxx), que contém as especificações de parâmetros globais de bancos de dados que não estejam sendo contempladas na versão corrente do SADS.

CONTROLPOINT - Número de syncpoints entre atualizações físicas do banco de dados.

Só se aplica a bancos auditados.

O valor máximo permitido é 4095 e o default DMSII é 2.

SYNCPOINT - Número máximo de transações que podem ocorrer entre gravações no audit.

Só se aplica a bancos auditados.

O valor máximo permitido é 4095 e o default DMSII é 100.

SYNCWAIT - Tempo, em segundos, de espera do programa por syncpoint.

Para utilizaço do SYNCWAIT, as opções INDEPENDENTTRANS e REAPLYCOMPLETED devem estar setadas.

ALLOWEDCORE - Quantidade de memória a ser utilizada para buffers do banco. O valor máximo permitido é 16.000.000 words e o default DMSII é 50.000 words.

O valor informado não poderá ser menor que o Resident Limit.

RESIDENT LIMIT - Número máximo de words para ocupação pelo banco.

OVERLAYGOAL - Percentual do alowedcore por minuto.

O overlaygoal controla a taxa de overlay de buffers para disco.

O valor máximo permitido é 100% do allowedcore por minuto e o default DMSII é 5% do allowedcore por minuto. Se for informado o valor "0", OVERLAYGOAL = 0 será gerado no fonte DASDL. Caso o valor não seja informado, o comando não será gerado.

OPERACAO - Operação efetuada em relação aos defaults/ parâmetros globais.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de opções globais

As opções globais controlam as características das rotinas de acesso ao banco de dados, em tempo de compilação e execução.

As informações relativas a opções globais no repositório de dados do SADS são utilizadas para geração do fonte DASDL do banco de dados.

CAMPOS DA TELA

BANCO - Código do banco de dados no padrão SADS (sssBxx).

INCORPORAR DEFAULTS DA INSTALACAO - Indicador de incorporação de defaults para opções globais.

Os valores possíveis são os seguintes:

"S" - Para assumir os valores pré-definidos para as opções globais que não forem especificados para o banco de dados;

"N" - Para desconsiderar os valores pré-definidos para as opções globais.

A pré-definição de defaults globais deve ser especificada na tabela SBPM5060. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY PARA GERACAO DASDL - Código de library, no padrão SADS (sssLxx), que contém as especificações de opções globais de bancos de dados que não estejam sendo contempladas na versão corrente do SADS.

AUDIT - Deve-se informar "S" para utilização de audit para o banco de dados. Caso o campo não seja informado, será assumido o valor default, isto é, que o banco de dados não será "auditado".

O audit proporciona a recuperação automática do banco de dados após falhas de hardware ou software. O banco de dados, quando auditado, deve conter a declaração da trilha de audit e do restart dataset.

ADDRESSCHECK - Deve-se informar "S" para utilização da opção addresscheck no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco de dados não utilizará esta opção.

Se informado "S", serão gerados controles de erros de I/O através da comparação de valores calculados na gravação e leitura de blocos de dados. Não pode ser declarada em nível individual de estruturas.

BACKOUTTOSPT - Deve-se informar "S" para utilização da opção backouttospt no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Se informado "S", em caso de recuperação, o banco de dados será retornado incondicionalmente a um ponto de integridade.

INDEPENDENTTRANS - Deve-se informar "S" para utilização da opção independenttrans no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Se informado "S", as transações ficarão independentes, de forma a permitir aborts de transações únicas, eliminar exceções e recuperar programas que não sofreram aborts.

KEYCOMPARE - Deve-se informar "S" para utilização da opção keycompare no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Se informado "S", serão geradas as opções de comparação da chave do dataset com as chaves dos sets e/ou subsets automáticos.

NO OPEN INITIALIZE - Deve-se informar "S" para utilização da opção no open initialize no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Se informado "S", não será permitida a utilização do comando "OPEN INITIALIZE" para o banco.

RDSSTORE - Deve-se informar "S" para utilização da opção rdsstore no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

A informação deste campo garante que, nas operações de begin e end transaction, o restart dataset será atualizado.

REAPLYCOMPLETED - Deve-se informar "S" para utilização da opção reaplycompleted no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Se informado "S", em caso de halt/load recovery, todas as transações completas serão reaplicadas. Esta informação exige a utilização da opção INDEPENDENTTRANS.

RESTART TRANSACTIONS ONLY - Deve-se informar "S" para utilização da opção restart transactions only no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

O "S" determina que só as informações para transações com término anormal, durante o estado de transação, serão armazenadas no restart dataset.

STATISTICS - Deve-se informar "S" para utilização da opção statistics no banco de dados. Caso este campo não seja informado, será assumido o valor default, isto é, que o banco não utilizará esta opção.

Esta opção faz com que as informações referentes à performance do banco sejam armazenadas durante a sua execução.

OPERACAO - Operação efetuada em relação às opções globais.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de softwares de banco

Esta função permite o cadastramento dos nomes e da localização dos softwares do DMSII, accessroutines, recovery, datarecovery, reconstruct e dmsupport.

As informações relativas aos softwares do banco de dados no repositório de dados do SADS são utilizadas para geração do fonte DASDL do mesmo.

O title especificado para a dmsupport é utilizado pelos programas gerados no SADS para que, em caso de cancelamentos em acessos e atualizações de bancos de dados, seja apresentado um display do nome da estrutura na qual aconteceu a exceção.

CAMPOS DA TELA

BANCO - Código do banco no padrão SADS (sssBxx).

INCORPORAR DEFAULTS DA INSTALACAO - Indicador de incorporação de titles defaults para softwares de bancos de dados.

Os valores possíveis são os seguintes:

"S" - Para assumir os titles pré-definidos para os softwares que não forem especificados para o banco de dados;

"N" - Para desconsiderar os titles pré-definidos para os softwares.

A pré-definição de titles de softwares deve ser especificada na tabela SBPM5070. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY PARA GERACAO DE DASDL - Código de library, no padrão SADS (sssLxn), que contém as especificações de titles de softwares de bancos de dados que não estejam sendo contempladas na versão corrente do SADS.

ACCESSROUTINES - Código das rotinas que executam o gerenciamento lógico e físico do banco de dados e que permitem o acesso concorrente ao banco por vários usuários.

Os comandos executados em programas, referentes a bancos de dados, são traduzidos em mensagens para as accessroutines. Assim, o acesso e a atualização dos bancos de dados somente podem ser executados através delas.

A rotina de acesso default do DMSII é a SYSTEM/ACCESSROUTINES.

DATARECOVERY - Código da rotina usada em procedimentos de recuperação que, através da leitura da trilha de audit e de sua atualização, auxiliam o reconstruct na recuperação de rows.

A rotina default do DMSII para recuperação é a SYSTEM/DMDATARECOVERY.

DMSUPPORT - Código da rotina, com funções específicas do DMSII, utilizada para atender necessidades especiais dos usuários através de seus entry points.

O title da dmsupport, por ser específico para cada banco de dados, não pode ser definido previamente. Portanto, sua informação é obrigatória.

O title padrão para a dmsupport é DMSUPPORT/código do Banco de Dados.

RECONSTRUCT - Código da rotina utilizada para recuperação de rows durante o processo de recuperação.

O title do reconstruct, por ser específico para cada banco de dados, não pode ser previamente definido. Portanto, sua informação é obrigatória.

O title padrão para o reconstruct é RECONSTRUCT/código do Banco de Dados.

RECOVERY - Código da rotina executada durante uma recuperação em audit e control file, rebuild ou rollback.

A rotina default do DMSII é a SYSTEM/DMRECOVERY.

OPERACAO - Operação efetuada em relação aos softwares de banco.

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de audit e control file

A audit trail ,ou trilha de auditoria, contém registros de controle e cópia das atualizações efetuadas no banco de dados.

É utilizada quando houver necessidade de recuperação do banco, pois, além da cópia das atualizações efetuadas, fornece informações de reinício dos programas sempre que necessário.

O control file contém informações sobre estruturas e parâmetros, responsáveis por garantir a compatibilidade entre as estruturas do banco de dados e os programas que as utilizam, e de controle para recuperação.

As informações relativas a audit e control file no repositório de dados do SADS são utilizadas para geração do fonte DASDL do banco de dados.

CAMPOS DA TELA

BANCO - Código do banco no padrão SADS (sssBxx).

AUDIT TRAIL:

INCORPORAR DEFAULTS INSTALACAO - Indicador de incorporação de defaults para a audit trail de bancos de dados.

Os valores possíveis são os seguintes:

"S" - Para assumir os valores pré-definidos para a audit trail que não forem especificados para o banco de dados;

"N" - Para desconsiderar os valores pré-definidos para a audit trail.

A pré-definição de valores para a audit trail deve ser especificada na tabela SBPM5080. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY PARA INCORPORAR DASDL - Código de library, no padrão SADS (sssLxx), que contém as especificações da audit trail do banco de dados que não estejam sendo contempladas na versão corrente do SADS.

AREAS - Número máximo de áreas em disco para uso da audit trail.

O valor default do DMSII é 100.

AREASIZE/*UNIDADE - Tamanho de cada área da audit trail, na unidade especificada.

As unidades possíveis são os seguintes:

"R" (Records) - Para que o tamanho seja expresso em número de registros;

"S" (Segments) - Para que o tamanho seja expresso em quantidade de segmentos;

"Y" (bYtes) - Para que o tamanho seja expresso em número de bytes;

"W" (Words) - Para que o tamanho seja expresso em número de palavras;

"B" (Blocks) - Para que o tamanho seja expresso em número de blocos.

O default do SADS para a unidade do campo AREASIZE é Blocks.

O valor default do DMSII é 100 blocks.

BLOCKSIZE/*UNIDADE - Tamanho dos blocos físicos da audit trail, na unidade especificada.

As unidades possíveis são os seguintes:

"R" (Records) - Para que o tamanho seja expresso em número de registros;

"S" (Segments) - Para que o tamanho seja expresso em quantidade de segmentos;

"Y" (bYtes) - Para que o tamanho seja expresso em número de bytes;

"W" (Words) - Para que o tamanho seja expresso em número de palavras;

"B" (Blocks) - Para que o tamanho seja expresso em número de blocos.

O default do SADS para a unidade do campo BLOCKSIZE é Words.

O valor default do DMSII é 900 words.

CHECKSUM(S/N) - Indicador de utilização da cláusula checksum.

Os valores possíveis são os seguintes:

"S" - Para utilização do checksum;

"N" - Para eliminar o uso do checksum (default do MSII).

O checksum é utilizado para controlar erros de I/O através da comparação de valores calculados na gravação e leitura de blocos de dados, podendo ser declarado em nível de estrutura.

PACK PRIMARIO - Local físico (família) onde o audit primário ficará residente.

PACK SECUNDARIO - Local físico (família) onde o audit secundário ficará residente.

O pack secundário é idêntico ao primário e é utilizado para garantir a recuperação do banco.

O default do DMSII é não possuir audit secundário.

 FITA (TIPO) - Tipo da fita magnética.

Os tipos possíveis são os seguintes:

"TAPE" ;

"TAPE7";

"TAPE9";

"PETAPE";

"TAPEPE";

"GCRTAPE";

"TAPEGCR".

Os tipos possíveis são aqueles relacionados no DMSII DASDL Program Reference Manual, da Unisys.

 DENSIDADE - Densidade da fita magnética.

Os tipos possíveis são os seguintes:

"BPI200";

"BPI556";

"BPI800";

"BPI1250";

"BPI1600";

"BPI6250";

"BPI11000";

"BPI38000".

Só é possível informar a densidade para as fitas definidas com o tipo "TAPE".

As densidades possíveis são aqueles relacionadas no DMSII DASDL Program Reference Manual, da Unisys.

NUM. COPIAS - Número de cópias do arquivo de audit, quando copiado para fita, verificado e removido do meio original.

Para que o arquivo possa ser copiado para fita, verificado e removido, o valor no campo no campo VERIFY deve ser "N".

 

VERIFY(S/N) - Indicador de verificação do arquivo de audit, sem que este seja copiado para fita.

Os valores possíveis são os seguintes:

"S" - Para verificar se o arquivo de audit está correto, sem copiá-lo para fita;

"N" - Para não impedir que o arquivo seja copiado para fita e verificado.

Se for informado o valor "S" no campo VERIFY, não será permitida a informação de número de cópias.

UPDATE EOF - Frequência com que um ponteiro de end of file é atualizado em disco, em número de blocos.

O default do DMSII é 100 blocks.

ALTERNATE IS TAPE - Indicador de utilização da fita magnética como meio físico alternativo para gravação de audit.

Os valores possíveis são os seguintes:

"S" - Para gravação do audit em fita magnética, quando a área em disco for insuficiente;

"N" - Para gravação de audits exclusivamente em disco.

O default do DMSII é a não utilização de fitas magnéticas.

JOB COPIA/VERIFICACAO - Nome do job a ser utilizado na cópia de audit primário para fita.

COPIA DE AUDIT SECUNDARIO - Indicador de cópia do audit secundário para fita, que pode assumir os seguintes valores:

"S" - Para que a cópia do audit secundário para a fita seja efetuada.

Neste caso o pack secundário deve ter sido informado.

"N" - Para que a cópia do audit secundário para a fita não seja efetuada.

O default para este campo é "N".

JOB COPIA/VERIF. AUDIT SECUND. - Nome do job a ser utilizado na cópia de audit secundário para fita.

Caso o nome do job para audit secundário não seja informado, e o indicador de cópia de audit secundário estiver com o valor "S", será utilizado o job informado para o audit primário.

Pode assumir os seguintes valores:

"N" - Para desconsiderar valores pré-definidos para control file.

USERCODE - Usercode sob o qual o control file ficará residente.

SECURITYGUARD - Nome do arquivo a ser utilizado pelo SYSTEM/GUARDFILE, caso ele seja utilizado.

OPERACAO - Operação efetuada em relação aos audits e control file .

As operações possíveis são as seguintes:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

Cadastramento de itens globais

Os itens globais formam um conjunto de atributos ou campos associados ao código do banco de dados.

Maiores informações poderão ser obtidas no "DMSII - Data and Structure Definition Language (DASDL) - Programming Reference Manual" da Unisys.

Através desta operação é permitido definir a estrutura dos itens globais, com as especificações dos atributos e campos que a compõem, seus relacionamentos e suas particularidades.

No contexto de itens globais, entenda-se atributo como atributo propriamente dito, campo ou sinônimo.

CAMPOS DA TELA

BANCO - Código do banco de dados de acordo com a padronização adotada pelo SADS (sssBxx).

Descrição resumida do banco de dados.

SEQ - Número sequencial que indica o posicionamento do atributo dentro da estrutura de itens globais do banco de dados.

Este campo já vem informado, sendo o primeiro número igual a "0010" e os demais acrescidos de 10.

Para incluir um atributo entre dois já incluídos, deve-se preencher uma linha que estiver em branco e alterar o número sequencial desta linha para um número entre os já informados. O SADS reorganizará automaticamente a estrutura. Na próxima operação, os atributos já estarão na nova ordem.

ATRIBUTO - Mnemônico do atributo que fará parte da estrutura dos itens globais do banco de dados.

NIVEL - Nível de estruturação do atributo, análogo ao nível de área de work dos programas Cobol.

O nível inicial é o 10 , se for necessário criar agrupamentos de atributos, deve-se usar múltiplos de 5, a partir de 15, até 45.

NDEF - Nível de definição do atributo.

Os valores possíveis são os seguintes:

"S" (Sistema) - Para atributo definido no sistema informado no campo SIS ou para atributo do mesmo sistema do banco de dados;

"G" (Geral) - Para atributo genérico do SADS;

Caso NDEF não seja informado, o atributo será interpretado como um item grupo na estrutura dos itens globais do banco de dados.

SIS - Código do sistema do atributo.
Este campo deve ser informado somente se NDEF = "S". Se o atributo estiver definido no mesmo sistema do banco de dados em questão, não é necessário informá-lo.

OCCURS - Número de ocorrências do atributo para criação de grupos repetitivos dentro da estrutura.

Para criar um grupo repetitivo, é necessário definir um item grupo, pois os itens elementares não aceitam o preenchimento do campo OCCURS.

CLAUSULA - Tipo de relacionamento para o atributo.

Os valores possíveis são os seguintes:

"C" (Comentário) - Indica que o atributo funciona como uma redefinição do anterior.

Como o DMSII não permite redefinição, apenas será gerado um comentário no fonte DASDL do banco de dados. Esta cláusula exige que o atributo anterior existente na estrutura do registro seja informado no campo COMPLEMENTO.

"D" (Cláusula Depending On)

No campo COMPLEMENTO, deve-se informar o atributo numérico pertencente à estrutura que está sendo definida e para o qual a cláusula depending on será gerada.

"E" (Nome externo) - Indica que o nome colocado no campo COMPLEMENTO deverá ser assumido na geração do DASDL das estruturas, em detrimento do nome gerado no padrão SADS, facilitando, desta forma, a integração com bancos de dados já existentes;

"I" (Initialvalue) - Indica que o atributo terá o valor inicial informado no campo COMPLEMENTO;

Em uma estrutura de banco de dados, um atributo pode possuir várias cláusulas, bastando para isto repetir a informação do atributo, com uma outra cláusula.

COMPLEMENTO - Este campo tem sua utilização dependente da informação do campo CLAUSULA, conforme explicações anteriores.

CONTINUA (S/N) - Indicador de continuidade de operação.

Pode assumir os seguintes valores:

"S" - Para prosseguir na operação que está sendo efetuada;

"N" - Para encerrar a operação que está sendo efetuada.

Se uma tela não for suficiente para a informação de todos os atributos da estrutura do dataset, ao informar "S" no campo CONTINUA, uma nova tela será mostrada para informação de outros atributos.

Especificações físicas globais são características físicas relativas ao banco de dados.

Maiores informações poderão ser obtidas no "DMSII - Data and Structure Definition Language (DASDL) - Programming Reference Manual" da Unisys.

Para cadastrar parâmetros físicos para o banco de dados, é necessário que o mesmo esteja definido no repositório de dados do SADS.

CAMPOS DA TELA

BANCO - Código do banco de dados para o qual serão especificados os parâmetros físicos (sssBxx).

INCORPORAR DEFAULTS DA INSTALACAO - Indicador de incorporação de defaults para o bancos de dados.

Os valores possíveis são os seguintes:

"S" - Para assumir os valores pré-definidos para o banco de dados que não forem especificados;

"N" - Para desconsiderar os valores pré-definidos para o banco de dados.

A pré-definição de valores para as especificações físicas globais do banco de dados deve ser especificada na tabela SBPM5085. Detalhes sobre esta tabela estão descritos na Seção V, "Parametrizações do SADS", neste manual.

LIBRARY GERACAO DASDL - Código de library, no padrão SADS (sssLxx), que contém as especificações físicas globais do banco de dados que não estejam sendo contempladas na versão corrente do SADS.

CHECKSUM - Indicador de utilização de checksum.

Os valores possíveis são os seguintes:

"S" - Para o uso de checksum;

"N" - Para eliminar o uso de checksum (default DMSII).

O checksum é utilizado para controlar erros de I/O, através da comparação de valores calculados na gravação e leitura de blocos de dados, podendo ser declarado em nível de estrutura.

KIND - Tipo do periférico a ser utilizado pelo banco de dados.

DISKPACK - Indicador do disco a ser utilizado pelo banco de dados.

PACKNAME - Nome do disco a ser utilizado pelo banco de dados.

PACK - Nome do disco a ser utilizado pelo banco de dados.

O disco default do DMSII é o Disk.

GUARDFILE - Indicador de uso de guardfile.

O arquivo guard controla o acesso ao banco de dados pelos aplicativos. Um arquivo guard somente deverá ser definido se realmente tiver de ser utilizado, pois a definição aleatória destes arquivos pode causar "SECURITY ERROR" nos acessos ao banco de dados.

Geração de fonte DASDL de banco

A DASDL (Data and Structure Definition Language) é a linguagem utilizada para descrição e construção de bancos de dados DMSII.

A compilação de um fonte DASDL gera o description e os outros softwares do banco que tornam os programas que o referenciam disponíveis para compilação. Para que as estruturas do banco sejam criadas, é necessário executar o processo de inicialização (initialize) e, caso contrário, o de atualizaço (update). Porém, é possível gerar um fonte DASDL sem a informação do objetivo da compilação (initialize ou update).

Para gerar o fonte DASDL do banco de dados, é necessário que ele, os titles da Dmsupport e do Reconstruct tenham sido definidos no repositório de dados do SADS, conforme descrito no tópico "Softwares de banco", neste mesmo item.

Caso seja desejável, pode-se parametrizar que, antes da geração de um novo fonte DASDL, o antigo seja salvo. Detalhes de como se parametrizar a salva de fontes DASL estão descritos na Seção V, "Parametrizações do SADS", item "Bancos", deste manual.

Maiores informações poderão ser obtidas no "DMSII - Data and Structure Definition Language (DASDL) - Programming Reference Manual" da Unisys.

CAMPOS DA TELA

Banco - Código do banco de dados que terá seu fonte DASDL gerado (sssBxx).

Objetivo da geração - Processo a ser executado sobre as estruturas do banco, que pode ser:

"I" (Initialize) - Para inicialização das estruturas do banco; ou

"U" (Update) - Para atualização das estruturas do banco.

Caso não seja informado, o objetivo da geração não será composto no fonte DASDL.

Ambiente - Ambiente no qual a library será gerada.

Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Se alguma das incorporações de defaults da instalação foi selecionada, na definição do banco de dados, o acesso às tabelas de parametrização de banco de dados será feito no ambiente indicado na geração do fonte DASDL. Detalhes a este respeito estão descritos na seção V, "Parametrizações do SADS", neste manual.

Será feito o acesso às tabelas de parametrização que contêm as opções de compilação a nível de instalação e a nível específico do banco de dados, no ambiente indicado na geração do fonte DASDL. Detalhes a este respeito estão descritos na seção V, "Parametrizações do SADS", neste manual.

Se a library for gerada para teste, as informações de teste serão obtidas neste mesmo ambiente, caso contrário, serão obtidas no ambiente de produção.

De acordo com ambiente informado, o código do fonte DASDL será gerado com o seguinte padrão: 

        Para o Banco de Dados - XYZB01
        O fonte DASDL será: 
XY/Z/BF/01        - Para produção;
XY/Z/BF/01T       - Para teste.

Usercode - Usercode de análise para geração da library.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Para usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS".
