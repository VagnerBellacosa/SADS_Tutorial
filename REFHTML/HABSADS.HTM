Habilitações de usuários SADS

Introdução

Alguns módulos do SADS validam a habilitação de elemento dos usuários que solicitam suas transações. Daí a necessidade de se cadastrar adequadamente as habilitações de elementos aos usuários, para que os mesmos consigam trabalhar satisfatoriamente em todo o SADS.

Os módulos do SADS que validam habilitações de elementos são:

SADS/SCP - Sistema de Planejamento e Controle de Projetos;

SADS/SDD - Repositório de Dados;

SADS/SPO - Sistema de Controle de Processamento;

SADS/SS - Sistema de Segurança;

SADS/SST - Sistema Gerenciador de Tabelas.

Como exemplo, na duplicação de um elemento ou na criação de um elemento a partir de outro já existente, o SADS testa se o usuário que disparou a transação está habilitado, em nível de elemento, a trabalhar com o sistema ao qual o elemento a ser criado pertence.

É importante lembrar ser imprescindível que também sejam habilitados corretamente aos usuários todos os aplicativos e transações das telas do SADS com as quais eles desejam trabalhar. Desta forma, para cada título de tela da navegação que um usuário irá utilizar, deverão ter-lhe sido previamente habilitados:

O aplicativo da tela de navegação;

O aplicativo do menu de opção, disparado por tal tela de navegação; e

As transações do menu de opção, referentes à operação tratada na tela de navegação.

Todas as habilitações a serem definidas para um usuário, conforme descrito no item "Usuários e Habilitações", nesta seção, podem ser cadastradas de forma hierarquizada. Isto significa que pode-se habilitar um usuário a um elemento específico ou a um grupo de elementos que iniciam com o argumento informado. A mesma possibilidade de hierarquização é válida para os aplicativos.

Habilitação para definição de contexto

Para definir um determinado contexto, com seus menus, rotas e regras, o usuário deverá ter habilitação de elemento para o mesmo no Sistema de Segurança. O nome da lista de elementos na qual o contexto deverá ser habilitado ao usuário é a "CONTEXT".

Esta lista deverá conter, como elementos, os códigos de todos os contextos com os quais o usuário irá trabalhar, com o devido tipo de habilitação (só consultar, consultar e atualizar ou não trabalhar).

Habilitação para exclusão de testes de segurança

Através da tabela SBPM5000, o responsável pelo suporte SADS pode definir que somente usuários autorizados possam eliminar testes de segurança que tenham sido cadastrados para serem disparados antes da execução das transações.

A partir desta parametrização, os usuários poderão então ser impedidos de:

Informar o valor "N" nos campos "Segur", "Consistencia Menu" e "Consistencia Transacao" no cadastramento de uma transação no menu;

Excluir um teste "SEG" ou "SGA" de um atributo na consistência; ou

Excluir um atributo da consistência que possua teste "SEG" ou "SGA".

Se a parametrização houver sido feita, somente os usuários autorizados ao elemento "S" na lista de elementos "MENUSEG", poderão:

Informar "N" nos campos "Segur", "Consistencia Menu" e "Consistencia Transacao" na tela de cadastramento de menu;

Excluir o teste "SEG" ou "SGA" de um atributo na consistência; ou

Excluir um atributo que possua tais testes.

Se a parametrização na tabela não houver sido feita, qualquer usuário poderá excluir qualquer um destes testes de segurança.

Maiores detalhes sobre esta parametrização encontram-se descritos na Seção V, "Parametrizações do SADS", item "Segurança", neste manual.

Habilitação para trabalhar com um usercode

A lista de elementos "SEGUSER" deverá conter todos os usercodes nos quais o usuário estará autorizado a trabalhar, como por exemplo: gerar elementos do SADS (programas, libraries e sub-rotinas), disparar compilações e binder, copiar fonte ou objetos de elementos, etc.

No momento do disparo destas operações, será então verificado se o usercode informado pelo usuário pertence à sua lista de elementos "SEGUSER". Caso negativo, a operação não poderá ser por ele processada no referido usercode.

Na inclusão do usercode default de um usuário, se o responsável pela inclusão possuir nível de acesso igual a 98 ou 99, o usercode por ele informado será gravado na lista de elementos "SEGUSER" do usuário a ser incluído.

Por outro lado, se o responsável pela inclusão do usercode default do usuário for o próprio usuário, então será feita a consistência da existência do usercode default na lista de elementos "SEGUSER", que deverá obrigatoriamente existir.

Habilitações para o SADS/SCP - Sistema de Planejamento e Controle de Projetos

As habilitações de elemento do usuário no SADS/SCP deverão determinar:

A sigla de funcionário com a qual o usuário foi identificado no SCP e com a qual ele deverá trabalhar neste sistema; e

Os códigos de projetos com os quais o usuário está relacionado e que, devido a isto, poderão ser por ele manipulados no SADS/SCP.

A sigla do funcionário, para diferenciar dos códigos dos projetos, deve ser informada precedida pelo caracter arroba ("@").

Habilitações para o SADS/SS - Sistema de Segurança

As habilitações de elemento para o SADS/SS deverão ser cadastradas para um usuário de nível de acesso geral igual a 98 - subadministrador de Segurança, a fim de determinar ou restringir seu poder de controle no Sistema de Segurança.

No Sistema de Segurança, os únicos usuários com permissão de cadastrar habilitações ou efetuar funções de controle são o administrador de Segurança, nível de acesso igual a 99, ou um usuário por ele devidamente habilitado, que deve possuir nível de acesso igual a 98.

O administrador de Segurança pode cadastrar quaisquer habilitações de aplicativos, transações e elementos para os usuários e efetuar todas as funções de controle. Já o subadministrador de Segurança somente poderá efetuar estas operações caso suas habilitações próprias o permitam, da seguinte forma:

Para que possa efetuar manutenção no cadastro de um segundo usuário, ele deverá possuir todos os aplicativos que estiverem habilitados ao usuário, habilitados como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar);

Para que possa habilitar elementos a um outro usuário, ele já deve possuir igual habilitação de elemento;

Para que possa habilitar um determinado aplicativo a um segundo usuário, o subadministrador de Segurança deverá possuir tal aplicativo habilitado como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar);

Caso o usuário a ser por ele habilitado possua também nível de acesso 98, então este tipo de habilitação deve ser "9" (Habilitar outro usuário 98).

Só um administrador de Segurança possui autorização para especificar "9" no tipo de habilitação e, assim mesmo, somente para um usuário 98 e apenas quando a lista de elementos for a "SBT" ou "SBT01".

Como exemplo, para autorizar um subadministrador de Segurança a habilitar um segundo usuário a:

Atualizar e consultar os sistemas "SDF";

Apenas consultar o sistema "DET" e

Atualizar o aplicativo "GLH04".

Ele deve possuir as seguintes habilitações de elementos:

Para que possa habilitar uma transação de um aplicativo a um segundo usuário, o subadministrador de Segurança deverá possuir tal aplicativo habilitado como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar).

Caso o subadministrador também esteja habilitado ao aplicativo, ele deve então possuir habilitação à mesma transação do aplicativo, para que possa habilitá-la a outro usuário.

Para que possa efetuar funções de controle sobre um aplicativo ou transação, ele deverá possuir tal aplicativo/transação habilitado como elemento, na lista de elementos "SBT" ou "SBT01", com tipo de habilitação igual a "2" (Atualizar) ou "9" (Habilitar a outro usuário 98).
