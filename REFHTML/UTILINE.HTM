Utilitários on-line

O SADS dispõe de certas funções on-line que apoiam o desenvolvimento e a manutenção dos sistemas aplicativos. Estas funções são as seguintes:

Geração de elementos do COMS;

Geração e liberação de família;

Duplicação de elemento;

Duplicação de registro;

Ressequenciamento de estrutura;

Descrição de elemento;

Importação de descrição de elemento;

Exportação de descrição de elemento;

Consulta a lista de cancelamentos on-line;

Consulta a transação cancelada;

Integração ET-SADS x SADS;

Relacionamento "Função X Transações";

Importação de edit;

Exportação de edit;

Cancelamento de transferência de tela e consistência para a máquina de produção;

Paginadores.

Cada uma destas funções encontra-se explicada detalhadamente a seguir.

Geração de elementos do COMS

Elementos do COMS são informações básicas, utilizadas pelo mesmo, para gerenciamento das transações on-line.

No momento da geração de elementos do COMS, será criado um arquivo fonte no usercode especificado pelo usuário que será posteriormente carregado através do Utility do COMS.

Argumentos da geração de elementos do COMS

Para que a geração de elementos do COMS possa ser utilizada, os seguintes argumentos devem ser informados no menu de operação "GERCOM":

CAMPOS DA TELA

Ambiente - Ambiente para o qual as especificações de elementos do COMS serão geradas.

Pode assumir os seguintes valores:

"T" - Para gerar o arquivo de elementos do COMS com características de sistema em teste;

"P" - Para gerar o arquivo de elementos do COMS com características de sistema em produção.

Janela - Nome da janela (window) na qual o sistema será executado.

Será apresentada uma tela de processamento, com as informações digitadas no menu de operação, para que sejam preenchidos os seguintes campos:

CAMPOS DA TELA

Prioridade - Número associado à execução do host, que determinará a prioridade deste em relação a outros programas na utilização dos recursos da máquina.

Pack do User de Execucao - Nome do disco onde ficarão residentes os arquivos aos quais o sistema poderá executar o acesso e os que ele irá gerar.

otherwise - Nome do disco em que o sistema encontrará os arquivos necessários para execução, que não estiverem residentes no disco primário.

Os valores possíveis são os seguintes:

"S" - Para que o código do programa e da agenda sejam gerados com o nome da janela como sufixo;

"N" - Para que o código do programa e da agenda sejam gerados sem o nome da janela como sufixo.

Janela/Host como INSTALLATION_DATA - Indicador de geração de INSTALLATION_DATA.

Os valores possíveis são os seguintes:

"J" - Será gerado um INSTALLATION_DATA com o mesmo nome da janela;

"H" - Será gerado um INSTALLATION_DATA com o mesmo nome do host;

Caso este campo não seja informado, não será gerado um INSTALLATION_DATA.

Agenda com TRANSACTION_MODE - Indicador de solicitação de transações em uma mesma janela.

Pode assumir os seguintes valores:

"S" - Quando cada transação de uma janela deve ser finalizada para que outra transação seja solicitada;

"N" - Quando for possível solicitar uma transação em uma janela sem que a solicitada anteriormente já tenha terminado.

Se nada for informado, será assumido "N".

Diferencia TRANCODE em teste - Indicação que permite a geração de trancodes diferenciados para o ambiente de teste, possibilitando que um sistema em teste e outro em produção possam ser executados simultaneamente em uma mesma agenda.

Esta diferenciação resume-se na utilização das letras "T" e "M" para trancodes, primários e secundários, em teste. Neste caso, os programas on-line do sistema têm de ser compilados com a opção de compilação "SADSDESENV".

Cumpre ressaltar que, o módulo de Navegação e o COOPERATOR não funcionam com trancodes em teste diferenciados.

Cria/associa DATABASE - Cria o elemento DATABASE e associa este código ao elemento program do COMS, correspondente ao host, que deve utilizar este banco como update.. Este campo deve ser informado caso se utilize recuperação sincronizada do COMS com o DMSII.

Trancodes de navegacao processados pelo host na janela - Trancodes utilizados em menus de escolha ou operação que sejam tratados por este host.

Detalhes sobre navegação podem ser encontrados na seção III, "Funções de Apoio", item "Navegação", neste manual.

Geração e liberação de família

O SADS possui uma função encarregada por gerar libraries , programas e sub-rotinas, liberar telas e consistências para testes e liberar libraries, telas e consistências para o ambiente de produção.

Esta função consiste em executar o acesso aos elementos de um sistema que forem indicados, gerar suas libraries ou liberá-los de acordo com a necessidade e o estado em que os mesmos se encontrarem.

Serão enviadas mensagens informando sobre o início das gerações e/ou liberações, o resultado de cada operação e se o término da geração e/ou liberação foi efetuado com sucesso. No caso de ocorrência de erro em alguma operação, ao final do processamento, é encadeada uma consulta paginada com os erros ocorridos durante este processamento de famílias, sendo salvas para que possam ser consultadas posteriormente através do menu "GERLIBFAM". A cada novo processamento de família, as mensagens de erro de execuções anteriores são excluídas deste arquivo.

Através desta função podem ser efetuadas:

Geração de libraries de área conversacional, área de trabalho, arquivo, área de comunicação com sub-rotina, DASDL, library de procedure, registro de uso geral, relatório, tabela, tela e tela de explicação para os ambientes de teste e produção;

Geração de programas e sub-rotinas;

Liberação de consistências e telas para o ambiente de teste;

Liberação de consistências, libraries e telas para o ambiente de produção.

Só serão processadas as liberações para os elementos da família que se encontrarem em estado compatível com as solicitações. Por exemplo: se forem solicitadas liberações de telas para o ambiente de teste, somente aquelas que estiverem em desenvolvimento ou manutenção serão liberadas.

As gerações também, em via geral, só são necessárias para os elementos da família que se encontrarem em estado compatível com as solicitações. Porém, para facilitar a geração das libraries de todos os elementos de uma família, pode-se optar por gerar todas as libraries, independentemente do estado de seus elementos, através do campo GERAR TODAS AS LIBRARIES (S/N).

Caso seja informado o código de um atributo (SISTEMA/ATRIBUTO), só serão gerados ou liberados os elementos que contenham o atributo em sua estrutura e estejam com status de "a gerar" ou "a liberar", respectivamente.

SISTEMA - Código do sistema do atributo pertencente aos elementos a serem gerados/liberados, no padrão SADS. (Opcional)

Para usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS", neste manual.

BANCO - Código do banco de dados, no padrão SADS (sssBxx), a ser utilizado para geração de parâmetros físicos de todos os datasets pertencentes a este, inclusive do restart dataset.

ELEMENTO - Família de elementos a ser gerada/liberada, no padrão SADS.

Duplicação de elemento

O SADS oferece uma função responsável por duplicar as definições e estruturas dos elementos cadastrados no repositório de dados do SADS.

Os elementos que podem ser duplicados são: arquivo, dataset, consistência, library, registro de uso geral, relatório, tabela e tela.

A duplicação de programas, sub-rotinas e módulos on-line e batch encontra-se descrita na Seção II, "Elementos do repositório de dados do SADS", no item "Programa", tópico "Duplicação de programa/módulo" e no item "Sub-rotina", tópico "Duplicação de sub-rotina/módulo".

A duplicação somente poderá ocorrer entre elementos do mesmo tipo se o elemento original e o sistema do elemento destino estiverem cadastrados e se o elemento a ser gerado ainda não estiver cadastrado. A única exceção desta regra é a duplicação de dataset, onde o dataset a ser gerado pode já estar cadastrado no repositório de dados, desde que possua o mesmo código externo do dataset original.

Somente elementos completos podem ser duplicados nesta função, não sendo possível duplicar isoladamente registros de elementos ou linhas de relatórios. Para duplicar isoladamente registros de elementos, consultar o item de duplicação de registro, neste mesmo capítulo.

Ao duplicar um elemento, todas as características de cadastramento do original serão herdadas pelo elemento duplicata, bem como descrição, estruturas, desenhos de linhas, validades de consistência etc. As únicas exceções são para telas e tabelas, que possuem funções de duplicação de desenho e de parâmetros, respectivamente.

Argumentos da duplicação de elemento

Para duplicar as definições e estruturas dos elementos cadastrados no repositório de dados do SADS, alguns argumentos devem ser informados no menu de operação DUPELE, quais sejam:

Elemento original - Código do elemento que será duplicado, no padrão SADS.

Os elementos que podem ser duplicados são: arquivo, base de dados, dataset, consistência, library, registro de uso geral, relatório, sub-rotina, tabela e tela.

Elemento novo - Código do elemento que será criado, no padrão SADS.

O elemento novo deve ser do mesmo tipo do elemento duplicado.

Duplicação de registro

O SADS oferece uma função responsável por duplicar registros das estruturas de dados cadastradas no repositório de dados do SADS.

Os elementos que podem ter seus registros duplicados são: arquivo, dataset, library, registro de uso geral e área de comunicação de sub-rotina.

A duplicação somente poderá ocorrer entre registros de elementos do mesmo tipo se o registro original e o elemento destino estiverem cadastrados e se o registro novo ainda não estiver cadastrado, com exceção do elemento dataset, que, ao ser duplicado, exige o registro 00. Porém se o registro a ser duplicado for o 00, o dataset, se não existente, será criado.

Ao duplicar um registro, todas as características de cadastramento do original serão herdadas pelo registro duplicado.

Argumentos da duplicação de registro

Para duplicar registro de estruturas dos elementos cadastrados no repositório de dados do SADS, alguns argumentos devem ser informados no menu de operação DUPREG, quais sejam:

Registro original - Código do registro que será duplicado, no padrão SADS.

Os elementos que podem ter seus registros duplicados são: arquivo, dataset, library, registro de uso geral e área de comunicação de sub-rotina.

Registro novo - Código do registro que será criado, no padrão SADS.

O registro novo deve ser de um elemento do mesmo tipo do elemento duplicado.

Ressequenciamento de estrutura

O SADS oferece uma função responsável por ressequenciar a estrutura informada. A primeira sequência ficará com o valor "0010" e as demais, acrescidas de 10.

Para isso, é necessário que a estrutura do elemento a ser ressequenciada já esteja cadastrada no repositório de dados do SADS.

Argumentos do ressequenciamento de estrutura

Para ressequenciar estrutura, deve-se utilizar o menu de operação RESEST, cujo campo deve ser preenchido conforme indicado:

Estrutura - Código da estrutura que se deseja ressequenciar.

Deve-se informar um código de estrutura, conforme especificado a seguir:

Área de comunicação de sub-rotina: "sssSxxnn";

Área de trabalho: "sssLxnmm", onde "mm" é o número do registro a ser ressequenciado;

Área de work de programa batch: "sssPxx";

Área de work de programa on-line: "sssNxx";

 Área de work de sub-rotina: "sssSxx";

Chave secundária de entidade: "sssEaaaa",
onde o último "a" é a letra que indica a chave secundária da entidade a ser ressequenciada;

Chave secundária de tabela: "sssMnnnna",
onde "a" é a letra que indica a chave secundária de tabela a ser ressequenciada;

Consistência: "sssCnnmm" ou "sssQxnmm".
As validades definidas para a consistência também serão ressequenciadas;

Entidade: "sssEaaa";

Linha de relatório: "sssRxn ll" ou "sssRxnnll",
onde "ll" é o número da linha a ser ressequenciada;

Registro de arquivo: "sssAxnmm",
onde "mm" é o número do registro a ser ressequenciado;

Registro de dataset: "ssBxxmm",
onde "mm" é o número do registro a ser ressequenciado;

Registro de uso geral: "sssGxnmm",
onde "mm" é o número do registro a ser ressequenciado;

Sets, subsets, access : "sssBxxa",
onde "a" é a letra que indica o set, subset ou access a ser ressequenciado;

Tabela: "sssMnnnn";

Tela: "sssTnnmm".

Descrição de elemento

O SADS permite que se cadastre a descrição de qualquer tipo de elemento - dataset, tabela, entidade, tela, etc - anteriormente definido no seu repositório de dados.

Estas descrições tem diversas utilidades, quais sejam:

Composição do manual da aplicação (manual do sistema, do usuário ou por comando);

Explicação das telas de menus de escolha e operação (através do acionamento do help pela tecla SPFY);

Explicação mais detalhada de um atributo, cujo erro foi detectado pela consistência, através do acionamento do comando "HELP".

Cumpre ressaltar que, para os elementos entidade externa, fluxo de dados, depósito de dados, função/processo e sistema, o cadastramento da descrição é feito simultaneamente às suas definições.

Argumentos da descrição de elemento

Para descrever um elemento, deve-se efetuar o desvio para o menu de escolha referente ao elemento que se quer descrever e nele selecionar a função apropriada para esta operação (geralmente "DESC").

No menu de operação apresentado, os campos que sempre devem ser preenchidos são os seguintes:

Operacao - Operação a ser efetuada em relação à descrição.

Pode assumir os seguintes valores:

"INC" (Inclusão);

"ALT" (Alteração);

"EXC" (Exclusão);

"CON" (Consulta).

<Elemento> - Código do elemento a ser descrito.

Importação de descrição de elemento

A descrição de qualquer elemento pode ser feita utilizando os recursos de edição do Cande e posteriormente importada para o repositório de dados do SADS.

Na operação de importação de descrição de um elemento qualquer, exceto atributo, campo, sinônimo e domínio, o arquivo texto deve estar no padrão "(USERCODE)TEXTO/sssexxxxxx", onde::

  (USERCODE) - Nome do usercode sob o qual se encontra;

 TEXTO - Palavra constante;

 sssexxxxxx - Nome do usercode sob o qual se encontra.

Na operação de importação de descrição de um atributo, campo, sinônimo ou domínio, o arquivo texto deve estar no padrão "(USERCODE)TEXTO/sss/eeetttqqqqq", onde:

 (USERCODE) - Nome do usercode sob o qual se encontra;

 TEXTO - Palavra constante;

 sss - Código alfabético que identifica o sistema no qual o atributo, campo, sinônimo ou domínio está definido;

 eeetttqqqqq - Mnemônico do atributo, campo, sinônimo ou domínio, cuja descrição será importada.

O arquivo texto deve ser do tipo SEQDATA e tem, necessariamente, de estar residente em algum usercode.

Para criar um arquivo deste tipo no Cande, deve-se executar o seguinte comando:

"make <<nome do arquivo>> SEQ", onde:

 make - Comando de criação de arquivo no Cande;

 <<nome do arquivo>> - Nome do arquivo nos padrões descritos anteriormente;

 SEQ - Palavra constante que identifica o tipo do arquivo

Como o tamanho da linha de descrição no repositório de dados do SADS comporta 78 caracteres e nos arquivos SEQDATA dispõe-se de apenas 72, pode-se utilizar 6 caracteres da próxima linha do arquivo para completar os 78 disponíveis no SADS. Para isto, deve-se informar "@" na coluna 9 da linha onde a informação será complementada.

Argumentos da importação de descrição de elemento

Para importar descrição de elemento, deve-se utilizar o menu de operação IMPDES, cujos campos devem ser preenchidos conforme indicado:

Elemento - Código do elemento no padrão SADS.

Para importar a descrição de um sistema, deve-se informar seu código no campo Elemento.

Para importar descrição de atributos, campos, sinônimos e domínios, deve-se informar, também, um código de sistema.

Mnemonico - Mnemônico de atributo, campo, sinônimo ou domínio.

Este campo somente deverá ser informado para importar a descrição de um atributo, campo, sinônimo ou domínio.

Usercode - Usercode onde está residente o arquivo texto que será importado para o SADS.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Exportação de descrição de elemento

A partir do cadastramento da descrição de um elemento, atributo, campo, sinônimo ou domínio, pode-se gerar um arquivo texto, tipo SEQDATA, no Cande, através de sua exportação.

Esta função possibilita executar manutenções na descrição de elementos, utilizando os recursos de edição do Cande.

Quando a exportação for de um elemento qualquer, exceto atributo, campo, sinônimo e domínio, o arquivo gerado terá o seguinte padrão:

"(USERCODE)TEXTO/sssexxxxxx", onde:

 (USERCODE) - Nome do usercode sob o qual se encontra o arquivo;

 TEXTO - Palavra constante;

 sssexxxxxx - Código do elemento cuja descrição foi exportada (padrão SADS).

Quando a exportação for da descrição de um atributo, campo, sinônimo ou domínio, o arquivo terá o padrão "USERCODE)TEXTO/sss/eeetttqqqqq", onde:

 (USERCODE) - Nome do usercode sob o qual se encontra o arquivo;

 TEXTO - Palavra constatne;

 sss - Código do sistema (padrão SADS) no qual o atributo, campo, sinônimo ou domínio está definido;

 eeetttqqqqq - Mnemônico atributo cuja descrição foi exportada.

Argumentos da exportação de descrição

Para exportar uma descrição de elemento, deve-se utilizar o menu de operação EXPDES, cujos campos devem ser preenchidos conforme indicado:

Elemento - Código do elemento que será exportado, no padrão SADS.

Para exportar a descrição de um sistema, deve-se informar apenas o seu código. Para exportar a descrição de atributos, campos, sinônimos ou domínios, deve-se informar, também, o código do sistema.

Mnemonico - Mnemônico de atributo, campo, sinônimo ou domínio.

Este campo somente deverá ser informado se a exportação for da descrição de um atributo, campo, sinônimo ou domínio.

Usercode - Usercode sob o qual será gerado o arquivo texto.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Caso uma das linhas da descrição do elemento cadastradas no repositório de dados do SADS tenha mais de 72 caracteres, ao efetuar a exportação para o Cande, os excedentes serão deslocados para a linha seguinte, antecedidos pelo caractere "@", na coluna 9. Consulta a lista de cancelamentos on-line

O SADS oferece uma função encarregada de listar os cancelamentos de programas on-line desenvolvidos no SADS.

A tela de processamento apresentada é do tipo consulta paginada e encontra-se descrita na Seção I, "Requisitos e conceitos para utilização do SADS", no item "Paginador de saída de dados".

Argumentos da consulta

Para consultar a lista de cancelamentos, deve-se utilizar o menu de operação LISCAN, cujos campos devem ser preenchidos conforme indicado:

Programa Coms - Código do programa cadastrado no Utility do COMS, cuja transação foi cancelada (opcional).

Data - Data em que ocorreu o cancelamento no formato DDMMAA (opcional).

Hora - Hora em que ocorreu o cancelamento no formato HHMM (opcional).

Informações exibidas na consulta

Os cancelamentos a serem apresentados dependem dos argumentos que foram informados, da seguinte forma:

Se forem informados os três argumentos (Programa Coms, Data e Hora), a consulta mostrará o cancelamento informado, se houver, e os cancelamentos ocorridos posteriormente.

Se for informado apenas o argumento Programa Coms, a consulta mostrará os cancelamentos ocorridos no programa, em ordem cronológica.

Se for informado apenas o argumento Data, a consulta mostrará os cancelamentos ocorridos a partir da data informada.

Se forem informados os argumentos Programa Coms e Data, serão mostrados os cancelamentos no programa informado, a partir da data informada.

Se forem informados os argumentos Data e Hora, serão mostrados os cancelamentos ocorridos na data e hora informadas, e também os cancelamentos ocorridos posteriormente.

Se nenhum argumento for informado, serão mostrados todos os cancelamentos de todos os programas.

Caso não tenha ocorrido nenhum cancelamento, será enviada a seguinte mensagem: "NENHUM CANCELAMENTO PARA O(S) ARGUMENTO(S)"

Para cada um dos cancelamentos on-line, serão apresentadas as seguintes informações:

PROGRAMA NO COMS - Nome do programa no COMS.

USUARIO - Número do usuário no cadastro do Sistema de Segurança.

ESTACAO - Estação que disparou a transação.

DATA - Data em que ocorreu o cancelamento.

HORA - Hora em que ocorreu o cancelamento.

MENSAGEM - Mensagem (truncada) enviada pela estação que provocou o cancelamento.

Integração ET-SADS x SADS

A integração entre o ET-SADS e o SADS tem como objetivo permitir uma complementação entre as fases do projeto de um determinado sistema desenvolvidas no ET-SADS com aquelas desenvolvidas no SADS.

Para executar a integração SADS x ET-SADS, determinados passos no ET-SADS deverão ser realizados, conforme descrito no "Guia de Integração com SADS", e outros no SADS, conforme descritos nos tópicos "Carga do sistema do ET-SADS para o SADS" e "Extração do sistema do SADS para o ET-SADS", a seguir.

Carga do sistema do ET-SADS para o SADS

Através desta função, o SADS torna possível carregar para seu repositório de dados informações relativas a um determinado sistema que tenham sido cadastradas no ET-SADS.

Para efetuar a carga do sistema no SADS, é necessário que os arquivos que contêm os dados do sistema tenham sido transferidos do ET-SADS (micro) para o mainframe.

Argumentos da carga

Para carregar os arquivos do ET-SADS para o SADS, deve-se utilizar o menu de operação CARSAD, cujos campos devem ser preenchidos conforme indicado:

SISTEMA - Código do sistema a ser carregado, no padrão SADS.

FILA PARA EXECUCAO - Fila para execução do job de carga, que não pode possuir limites.

USERCODE/PASSWORD - Usercode e password sob o qual será executado o job de carga.

ACCESSCODE/PASSWORD - Acesscode e accesscodepassword sob o qual será executado o job de carga.

CHARGECODE - Chargecode sob o qual será executado o job de carga.

Informações exibidas

Será apresentada a tela de processamento para integração SADS X ET-SADS.

Nesta tela são apresentados dados do job de carga e os arquivos que foram gerados pelo ET-SADS e deverão ser carregados.

SISTEMA - Código do sistema a ser carregado, no padrão SADS (sss).

FILA P/ EXECUCAO - Número da fila para execução do job de carga.

VERSAO DA INTEGRACAO - Número da versão do SADS, seguido do número da versão do ET-SADS, separados por um
ponto ( . ).

ARQUIVOS - Número sequencial correspondente ao dataset do SADS, seguido do seu nome abreviado.

DATA - Data em que foi extraído o arquivo do ET-SADS.

HORA - Hora em que ocorreu a extração do arquivo do ET-SADS.

NUM.REGISTROS - Número de registros que o arquivo possui.

CONFIRME(S/N) - Indicador de confirmação de operação.

Pode assumir os seguintes valores:

"S" - Para confirmar a carga dos arquivos;

"N" - Para cancelar a carga dos arquivos.

Na execução da carga do sistema, será feito um merge entre os arquivos carregados e os datasets do repositório de dados do SADS.

Extração do sistema do SADS para o ET-SADS

Através desta função, o SADS possibilita extrair os elementos cadastrados no repositório de dados do SADS, de um determinado sistema, para torná-los disponíveis para utilização no ET-SADS.

Somente serão extraídos os elementos que são tratados pelo ET-SADS, quais sejam: sistema, entidades, atributos/sinônimos, domínios, generalizações, relacionamentos, sub-rotinas, processos/funções, depósitos de dados, entidades externas, fluxos de dados, além das descrições, expressões e validades dos mesmos.

Para executar esta função, é necessário que o sistema e seus elementos estejam cadastrados no repositório de dados do SADS.

Argumentos da extração

Para extrair os arquivos para o ET-SADS, deve-se utilizar o menu de operação EXTSAD, cujos campos devem ser preenchidos conforme indicado:

FILA PARA EXECUCAO - Número da fila para execução do job de extração.

ACCESSCODE/PASSWORD - Acesscode e accesscodepassword sob o qual será executado o job de extração.

CHARGECODE - Chargecode sob o qual será executado o job de extração.

Importação de edit

O SADS possibilita importar arquivos de comandos em Cobol para que se tornem comandos EDIT. É necessário que o arquivo a ser importado possua o nome no padrão "EDIT/ sssennmmm/xxxxxxxxx", onde:

"sssenn" é o código do programa

"mmm" é o número do módulo ao qual o EDIT pertence

"xxxxxxxxx" é o nome do EDIT.

Com a operação de importação, torna-se possível utilizar as facilidades de edição do Cande.

Sempre quando for efetuada uma importação de comando EDIT do Cande, o programa que teve o EDIT importado ficará marcado, podendo ser compilado somente após ser regerado. Isto para garantir que os comandos do EDIT sejam devidamente gerados no programa após sua importação para o SADS.

Para importar um arquivo para se tornar um comando EDIT, basta transmitir os argumentos do menu de operação IMPEDI, cujos campos devem ser preenchidos conforme indicado:

Programa - Código do programa ao qual o EDIT pertence;

Modulo - Código do módulo do programa ao qual o EDIT pertence;

Edit - Nome do comando EDIT;

Usercode - Usercode onde reside o arquivo a ser importado.

O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

Exportação de edit

O SADS possibilita exportar comandos de módulo EDIT para o Cande. Ao efetuar uma exportação de um EDIT para o Cande, será gerado um arquivo de nome "EDIT/ sssennmmm/xxxxxxxxx", onde:

"sssenn" é o código do programa

"mmm" é o número do módulo ao qual o EDIT pertence

"xxxxxxxxx" é o nome do EDIT.

Este arquivo pode então ser alterado e depois importado pelo SADS, permitindo, com estas operações, que sejam utilizadas as facilidades de edição do Cande.

Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

EXPORTAR MESMO QUE JA EXISTA O EDIT? (S/N) - Indicador de sobreposição de arquivos no Cande, na operação de esportação.

"N" - Para que seja não seja gerado o arquivo no Cande contendo os comandos do EDIT caso já exista um arquivo neste usercode com mesmo nome.

Paginadores

O SADS possui módulos que facilitam a programação dos sistemas que utilizam múltiplas páginas em aplicações on-line. São eles:

Paginador de entrada de dados e

Paginador de saída de dados.

Paginador de entrada de dados

O paginador de entrada de dados facilita a programação de aplicações on-line de entrada de dados que utilizam múltiplas telas.

Elimina a necessidade de codificação e economiza recursos de processamento e memória. Provê o armazenamento de diversas páginas de entrada de dados e flexibilidade de acesso às mesmas.

Suas principais características são:

Facilidades de navegação entre as páginas;

Consistência efetuada a cada página transmitida, com possibilidade do programa aplicativo consistir as páginas armazenadas, de uma só vez, contra cadastro;

Manutenção da última transação paginada, em nível de estação;

Paginação de telas de layouts diferentes, com independência de seus formatos, sendo aplicável a transações de inclusão e alteração;

Possibilidade de restrição do número máximo de páginas a serem tratadas pelo usuário;

Utilização por programas escritos em diferentes linguagens.

Argumentos para o paginador de entrada

A última atualização de dados que utilizou o paginador de entrada - atualização paginada - estará sempre disponível na mesma estação onde foi apresentada para posterior alteração. Deste modo, após uma atualização paginada ter sido exibida, ela fica armazenada pelo SADS e o usuário pode, a qualquer momento, reapresentá-la na mesma estação, sem que seja necessário disparar a transação que a originou.

Para então trabalhar com a última atualização paginada apresentada na estação, basta transmitir o menu de operação PAGENT, cujo único campo é o seguinte:

Pagina - Número da primeira página a ser utilizada, que já vem preenchida com "01".

Informações exibidas pelo paginador de entrada

Os campos específicos para a atualização paginada são:

PAGINA - Número da página corrente;

PROXIMA PAGINA - Número da página desejada, quando houver necessidade de se ver uma tela fora de sequência.

Pode assumir os seguintes valores:

"-" ou "--" - Para voltar à página anterior;

"UL" - Para chegar à última página;

"99" - Para encerrar a paginação e efetivar o processamento.

Se este campo não for informado, será apresentado a próxima página na sequência que foi programada.

O processamento dos dados preenchidos nas páginas somente será efetuado quando for transmitido "99" no campo PROXIMA PAGINA.

Os demais campos apresentados na atualização paginada são específicos da transação efetuada e estão de acordo com o programa aplicativo que trata a transação.

Paginador de saída de dados

O paginador de saída facilita a programação dos sistemas que utilizam consultas de múltiplas páginas em aplicações on-line.

Elimina a necessidade de codificação e economiza recursos de processamento e memória. Provê o armazenamento de diversas páginas de consultas e flexibilidade de acesso às mesmas.

Suas principais características são:

Preparação de consultas através de programas on-line, com arquivamento feito por sub-rotina e possibilidade de utilização de telas de layouts diferentes;

Paginação das consultas, independente de seus formatos;

Utilização por programas escritos em diferentes linguagens;

Possibilidade de impressão remota de páginas selecionadas;

Facilidades de navegação entre as páginas;

Manutenção da última transação paginada, em nível de estação.

Argumentos para o paginador de saída

A última consulta que utilizou o paginador de saída - consulta paginada - estará sempre disponível na mesma estação onde foi apresentada, para posterior visualização ou impressão. Deste modo, após uma consulta paginada ter sido exibida, ela fica armazenada pelo SADS e o usuário pode, a qualquer momento, reapresentá-la na mesma estação ou imprimí-la, sem que seja necessário disparar a transação que a originou.

Para então visualizar ou imprimir a última consulta paginada apresentada na estação, deve-se utilizar o menu de operação PAGSAI, cujos campos devem ser preenchidos conforme indicado:

Operacao - Operação a ser efetuada em relação à última consulta paginada, que pode ser:

"CON" (Consulta) - Para que seja reapresentada a última consulta paginada efetuada na estação; ou

"IMP" (Impressão) - Para que seja impressa a última consulta paginada efetuada na estação.

Primeira pagina - Número da primeira página a ser impressa. (Opcional)

Caso não seja informada, a impressão irá iniciar da página 1 da consulta.

Ultima pagina - Número da última página a ser impressa. (Opcional)

Caso não seja informada, a impressão irá finalizar na última página da consulta.

Impressora - Código da impressora onde a consulta paginada será impressa.(Opcional)

Pode assumir os seguintes valores:

"<Código da impressora remota>" - Para impressão da consulta na impressora remota informada (esta impressora deve estar cadastrada na rede de terminais);

"PRINTER" - Para impressão da consulta na impressora central, no formulário de 132 posições;

"PRINxx" - Para impressão da consulta na impressora central em formulário especial, onde "xx" é um formid definido na instalação.

Caso a impressora não seja informada, será assumida a impressora default do usuário.

Detalhes sobre o cadastramento da impressora default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e Habilitações", neste manual.

Informações exibidas pelo paginador de saída

Os campos específicos da consulta paginada são:

PAGINA - Número da página corrente;

PROXIMA PAGINA - Número da próxima página que será apresentada. Quando houver necessidade de se ver uma tela fora de sequência, deve-se informar o número da página desejada.

Pode assumir os seguintes valores:

"-" ou "--" - Para voltar à página anterior;

"UL" - Para chegar à última página;

"99" - Para encerrar a paginaço.

Se este campo não for informado, será apresentada a próxima página na sequência que foi programada.

No rodapé da página será exibida a palavra "CONTINUA" ou "FIM", significando, respectivamente:

"CONTINUA" - Existem mais páginas para serem exibidas;

"FIM" - Não existem mais páginas para serem exibidas.

Os demais campos apresentados pela consulta paginada são específicos da transação efetuada e são compostos pelo próprio programa aplicativo que trata a transação.

Desvio de transações através do paginador de saída de dados

O paginador de consulta - STP03 - permite que transações sejam encadeadas após uma consulta paginada, não necessitando assim retornar ao menu para disparar uma próxima transação.

Deste modo, após uma consulta paginada pode-se encadear uma outra transação, quer seja consulta ou não, de um menu qualquer, a partir de dados informados pelo usuário.

Para que este encadeamento seja possível, é necessário que a tela de consulta paginada e o programa encarregado pelo seu processamento sejam adaptados.

A tela de consulta paginada deve ser adaptada de acordo com as seguintes orientações:

 No desenho e na estrutura da tela de consulta devem ser incluídos, logo após o campo "TLAESTAC", os campos - argumentos - necessários ao disparo da próxima transação.

 Nem todos os campos do menu referente à próxima transação necessitam ser incluídos, somente aqueles que devem ser informados pelo usuário. Os demais poderão ser compostos pelo próprio programa de consulta.

No programa encarregado pela consulta paginada devem ser efetuadas as seguintes adaptações:

 Devem ser criados, em sua WORKING-STORAGE SECTION, campos encarregados por simular a tela menu da transação a ser encadeada.

 Devem ser incluídos, em sua PROCEDURE DIVISION, antes do comando que desvia o processamento para a sub-rotina de paginação, os comandos que determinam e complementam o encadeamento a ser efetuado.

 O primeiro FILLER determina, entre as arrobas, o número de caracteres do primeiro argumento. No exemplo, "04" é o tamanho do código do empregado - "DSRS".

 Sempre quando o campo da tela menu for um argumento da tela de consulta paginada, deve-se informar apenas seu tamanho, delimitado por arrobas, no campo da área de work do programa de consulta paginada.

 O segundo FILLER tem função similar ao primeiro e determina, entre as arrobas, o número de caracteres do segundo argumento. No exemplo, "03" é o tamanho do código do dependente a ser informado.

 O terceiro FILLER determina, entre as chaves, o número e o tipo do caractere a ser passado como argumento. No exemplo, serão preenchidos 17 (dezessete) brancos, correspondentes ao campo IMPRESSORA.

 O campo WSS-OPCAO tem o valor "84", que indicará o encadeamento da opção "84" do menu "PED03" - Consulta a dependente.

 O quarto FILLER - "?" - indica o fim do argumento.

Generalizando, os seguintes padrões são adotados para definição dos campos na Working-Storage Section:

 A sequência de caracteres @nn@ indica que o dado será utilizado como argumento da tela e "nn" é o tamanho deste argumento;

 {nnx} - As chaves indicam repetição de caracteres, onde "x" é o caractere a ser reptido e "nn" é a quantidade em que o mesmo será repetido;

 ? - A interrogação indica o fim do argumento.

Como um exemplo genérico, tem-se:

01  WSS-STPS04-DTTELA.

    05 FILLER

PIC X(04) VA "@20@".

(Indica que 20 bytes serão montados a partir de informação do usuário na tela de consulta paginada)

    05 FILLER

 

PIC X(06) VA "010190".

(Indica que será movida a constante "010190")

    05 WSS-DATA

 

PIC X(06).

(Indica que este campo será composto internamente ao programa de consulta paginada)

    05 FILLER

 

PIC X(05) VA "{10*}".

(Indica que serão montados 10 caracteres * - 10 asteriscos)

    05 WSS-OPCAO

 

PIC X(02) VA "01".

(Indica que será movida a constante "01")

    05 FILLER

 

PIC X(01) VALUE "?".

(Indica fim do argumento)

Uma outra forma de propiciar o encadeamento é compor o campo WSS-STPS04-DTTELA com todos os campos transmissíveis da tela menu. Para isto, deve-se informar um campo FILLER de tamanho 4 (quatro) com valor "@00@" como o primeiro do item grupo, conforme exemplo:

01  WSS-STPS04-DTTELA.

   05 FILLER

PIC X(04) VALUE "@00@".

   05 WSS-ARGUMENTOS

PIC X(26).

   05 FILLER

PIC X(01) VALUE "?".

No exemplo anterior, WSS-ARGUMENTOS deve ter o tamanho total dos argumentos da tela menu do programa a ser encadeado, que é de 26 (vinte e seis) bytes.

    1. Na PROCEDURE-DIVISION

Na PROCEDURE-DIVISION do programa, antes do primeiro comando que desvia o processamento para a sub-rotina de paginação - PERFORM 89000-00-PAGINAR -, deve-se acrescentar os seguintes comandos:

 Composição dos campos que não são constantes em WSS-STPS04-DTTELA, caso existam;

 Composição do campo de opção, se ele não for uma constante;

 Composição do código de transação a ser encadeado, da seguinte forma: MOVE "sssnn.mm" TO STPS04-DTTRANSACAO;

 Movimentação do campo da área de Work para a área de comunicação do paginador: MOVE WSS-STPS04-DTTELA TO STPS04-DTTELA.

No exemplo de encadeamento já exibido, teríamos as seguintes composições:

MOVE "PED03.84"

    TO STPS04-DTTRANSACAO.

MOVE WSS-STPS04-DTTELA

    TO STPS04-DTTELA.

A respeito destes comandos, tem-se as seguintes considerações:

 Foi considerada a primeira forma de composição da área de work do programa explicada anteriormente;

 O primeiro comando "MOVE" irá compor o trancode do menu cuja transação se deseja encadear.

 É obrigatório informar este trancode.

 O segundo comando "MOVE" irá montar a simulação da tela menu.

 É também obrigatório incluir este comando.

Um terceiro comando "MOVE" poderia ter sido incluído para compor o campo WSS-OPCAO, ao invés de ter sido composto através da área de work, da seguinte forma:

MOVE "84"

TO WSS-OPCAO

 

Este comando também seria incluido antes da chamada da sub-rotina de paginação.

Deve-se, então, observar que os argumentos do menu podem ser compostos internamente ao programa de consulta paginada.
