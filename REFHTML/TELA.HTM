Tela

Conceituação

Tela é um elemento responsável por captar e apresentar dados, ou seja, é um elemento utilizado para promover a interface entre usuários e sistemas aplicativos.

Cada tela possui um desenho, apresentado ao usuário sempre que este solicitar sua transação correspondente, e uma estrutura composta de atributos e/ou campos, elementos variáveis no processamento das transações.

Tipos de telas

Existem três tipos diferentes de telas para o SADS:

Telas que compõem as transações do sistema aplicativo, identificadas no seu código pela constante "T" (sssTnnmm);

Telas para explicação de argumentos de uma tela menu, identificadas no seu código pela constante "E" (sssEnnmm);

Telas utilizadas para informação de parâmetros para jobs do Sistema de Controle de Processamento (SCAP/SPO), identificadas no seu código pela constante "J" (sssJxnmm).

As telas que fazem parte do aplicativo, identificadas pela letra "T", são as de menu e as de processamento das opções dos menus.

As telas para explicação de argumento, identificadas pela letra "E", são utilizadas para esclarecer como os argumentos dos menus de opção devem ser informados para o processamento de cada uma de suas opções.

Essas telas, disponíveis para serem utilizadas apenas em menus de opção, não precisam ser desenhadas para sistemas cuja navegação seja feita através de menus de escolha e de operação, método assim construído visando, por si só, possibilitar um tratamento diferenciado para cada transação.

Para cada uma das opções constantes no menu pode ser cadastrada uma tela de explicação de argumento.

A seguir, são apresentados exemplos de como codificar uma tela de explicação de argumento:

Sistema SXY, menu 01, opção 45: tela de explicação "SXYE0145";

Sistema XYZ, menu 05, opção 04: tela de explicação "XYZE0504".

Para tornar uma tela de explicação disponível para utilização pelo usuário final, basta desenhá-la e liberá-la. Não é necessário defini-la ou gerar sua library Cobol.

Após a sua liberação, a tela de explicação será exibida ao usuário final quando este, ao apresentar dúvidas quanto ao preenchimento dos argumentos, informar o número da opção nos dois primeiros bytes disponíveis da tela menu e transmiti-la com a opção "97".

As telas utilizadas para informar parâmetros para jobs do Sistema de Controle de Processamento (SPO), identificadas pela letra "J" (sssJxnmm), devem ser criadas sempre que na preparação de um serviço (job) houver necessidade desta informação para a execução do mesmo.

Os jobs que possuem parâmetros devem estar relacionados às telas para captação dos mesmos, como por exemplo:

Sistema FGH, job "FGHJD1", tela de parâmetro "FGHJ0100";

Sistema JFD, job "JFDJ08", tela de parâmetro "JFDJ0403".

Não é necessário que o nome do job seja o mesmo ou esteja indicado no código da tela de parâmetro a ele relacionado. Basta que, nos jobs fontes do SPO, seja indicado qual é o código da tela de parâmetro a ele relacionada.

Estados de telas

O estado de uma tela identifica, para cada um de seus desenhos, se ela pode ser utilizada em algum ambiente de processamento (teste ou produção), se ainda está em desenvolvimento ou em manutenção.

Assim sendo, uma única tela possui desenhos específicos nos estados de desenvolvimento, manutenção, teste e produção.

O estado de desenvolvimento é exclusivo, ou seja, uma tela "em desenvolvimento" possui apenas uma cópia do desenho. Portanto, ela ainda não pode ser utilizada pelos programas aplicativos nos ambientes de teste e produção.

Quando uma tela "em desenvolvimento" é liberada para teste, ela deixa de existir naquele ambiente para existir apenas "em teste". A partir deste momento, qualquer alteração efetuada no seu desenho acarretará na criação de outra versão "em manutenção" que, até ser liberada, não interferirá nas versões que se encontram "em teste" ou "em produção".

Portanto, uma tela pode ter, simultaneamente, até três versões diferentes do seu desenho, ou seja, "em produção", "em teste" e "em manutenção".

A seguir, são apresentados detalhes sobre cada um dos estados de uma tela:

Desenvolvimento - Estado inicial da tela. Neste estado, a tela não se encontra disponível para os programas aplicativos nos ambientes de teste e produção.

Teste - Estado resultante da liberação para teste de uma tela em manutenção ou em desenvolvimento. Neste estado, a tela encontra-se disponível para acesso pelos programas aplicativos executáveis no ambiente de teste. Porém, caso algum programa no ambiente de produção tente obter acesso a uma tela que esteja liberada para teste mas não esteja liberada para produção, o acesso será para a versão em teste.

Produção - Estado resultante da liberação para produção de uma tela em teste. Neste estado, a tela encontra-se disponível apenas para os programas aplicativos executáveis no ambiente de produção.

Manutenção - Estado intermediário de uma tela anteriormente em teste, ou seja, é o estado resultante da alteração de uma tela já liberada. A versão "em manutenção" não fica disponível para acesso pelos programas aplicativos, podendo sofrer alterações sem que estas reflitam nos ambientes operacionais de teste e de produção.

Padronização

No SADS, existe um padrão de codificação específico utilizado para cada tipo de tela no cadastramento do seu desenho:

- Padrão para telas de menu e de processamento:

"sssTnnmm" onde:

"sss" é o código alfabético que identifica o sistema;

"T" é a constante de tela;

"nn" é o código numérico que identifica o menu e

"mm" é o código numérico que identifica a tela

O código numérico que identifica a tela deve ser o mesmo do número da opção do menu ao qual ela se encontra relacionada, pois o processo de geração de programas on-line pressupõe esta compatibilidade.

Exemplo da formação de um código de tela:

  A partir do Sistema: XYZ,
              Menu : 01 e
              Opção : 05,
  tem-se a Tela   : XYZT0105.

- Padrão para telas de explicação de argumento:

"sssEnnmm" onde:

"sss" é o código alfabético que identifica o sistema;

"E" é a constante de tela de explicação de argumento;

"nn" é o código numérico que identifica o menu e

"mm" é o código numérico que identifica a opção a ser explicada.

- Padrão para telas para captação de parâmetros de job:

"sssJxnmm" onde:

"sss" é o código alfabético que identifica o sistema;

"T" é a constante de tela de parâmentro de job e

"xnmm" é o identificador da tela.

Utilização

As telas são utilizadas, de modo geral, como interface entre usuários e transações on-line para captação e apresentação de informações em terminais de vídeo.

Cadastramento

No SADS, a definição e o desenho da tela são cadastrados separadamente. Para definir ou desenhar uma tela no SADS, é necessário que o sistema tenha sido anteriormente cadastrado.

Roteiro

Para cadastrar uma tela no SADS e torná-la disponível para utilização, é necessário executar as seguintes funções:

Desenho de tela;

Definição de tela;

Cadastramento de estrutura de tela;

Geração de library de tela;

Geração de consistência de tela;

Liberação de desenho de tela;

Liberação de library de tela, e

Liberação de consistência de tela.

Para a tela de parâmetro de job SXYJ0100, o trancode será SXY01J00. Este trancode não será utilizado pelo COMS para endereçar uma transação, pois, a tela, no momento de sua apresentação, sempre virá precedida do trancode SPO01.43.

Tratamento da navegação - Caso seja utilizado um campo de ação para tratamento da navegação, este campo, com 69 posições, input e opcional, deverá ser incluído, em menus ou telas de processamento que não utilizem o paginador, após o trancode, ou em telas paginadas, após o campo TLAPAGNUM - sugere-se o campo TLAACAO. Caso este campo seja utilizado no desenho de telas de processamento, deve-se informar "S" no campo "acao", no cadastramento do menu, descrito no item "Menu", sub-item "Definição de menu", nesta seção. As seguintes situações podem ocorrer em função do uso de um campo de ação em telas de processamento:

  • Se o campo "acao", na definição de menu, tiver sido informado, e no desenho e estrutura da tela não existir o campo TLAACAO, o host processará a transação como se as primeiras 69 posições preenchidas na tela fossem o campo de ação;

  • Se o campo "acao", na definição de menu, não tiver sido informado, e no desenho e estrutura da tela existir o campo TLAACAO, o host processará a transação independente do conteúdo do campo de ação;

  • Se o campo "acao", na definição de menu, tiver sido preenchido, e no desenho e estrutura da tela também existir o campo TLAACAO, o host processará a transação interpretando o conteúdo do campo de ação.
    •

    Nas telas de processamento e menus dos sistemas, é possível desenhar um campo de output com 18 (dezoito) ou 20 (vinte) posições alfanuméricas na primeira linha, para que sejam apresentadas automaticamente a data e hora de sua exibição no terminal, sendo que deve ser utilizado o atributo TLADTH ou TLADTHC, respectivamente, no cadastramento de sua estrutura.

    Nas telas menus, é possível que sejam preenchidos automaticamente, pelo próprio host, os campos sigla da empresa e ambiente nos quais foi efetuado o logon. Para isso, é necessário que eles sejam desenhados, na primeira ou segunda linha da tela, sem o caracter chaves ("{") anteriormente e sem necessidade de cadastramento na estrutura da tela. A sigla da empresa deve possuir 3 (três) posições alfanuméricas e o ambiente, 8 (oito) posições alfanuméricas. O host deve ter sido compilado após a implantação da versão 3.2.661 do SADS para que tal preenchimento possa ser efetuado.

    Nas telas menus, o SADS disponibiliza também o preenchimento automático de determinados campos de output. Para utilizar esta facilidade, devem ser observados os seguintes procedimentos:

  • É necessário desenhar, em qualquer posição da tela menu, uma sequência de caracteres especiais para identificação dos campos de output, sem colocá-los precedidos por chaves ("{").

  • Náo é necessário cadastrar um atributo correspondente à estes caracteres na estrutura da tela.

  • Estes campos podem estar separados, em qualquer posição da tela.

  • O caractere especial utilizado para identificação dos campos de output não pode ser utilizado para qualquer outra finalidade nos desenhos de tela.

  • Qualquer sequência menor do que 3 (três) caracteres não será composta, sendo que o número de caracteres especiais deve ser múltiplo de 3 (três).

  • O caractere especial que irá definir tais campos de output, o conteúdo a ser preenchido nestes campos e a data a partir da qual o conteúdo deve ser utilizado no preenchimento devem ser parametrizados na tabela SBPM5022. Detalhes a este respeito estão descritos na seção V, "Parametrizações do SADS", neste manual.

  • O host que irá apresentar os menus com campos de output especiais deve ter sido compilado após a implantação da versão 3.2.661 do SADS.
    •

    Nas telas de consulta que serão utilizadas por programas de consulta paginada, é necessário desenhar, após o campo de data/hora, se existente, os seguintes campos:

  • Um campo com 2 (duas) posições numéricas para exibir o número da página corrente - sugere-se o campo TLAPAGNUM;

  • Opcionalmente, após o campo TLAPAGNUM, pode-se desenhar um campo "ACAO" para tratamento da navegação, com 69 posições, input e opcional. Sugere-se o campo TLAACAO;

  • Um campo com 2 (duas) posições numéricas para se informar o número da próxima página a ser apresentada na consulta - sugere-se o campo TLAPAGPROX;
    •

    Deve-se salientar que, em uma tela a ser utilizada em um programa de consulta paginada, os únicos campos de entrada de dados (input) permitidos são: o trancode, o campo para o número da página corrente, o campo para o número da próxima página e, opcionalmente, os campos de ação - TLAACAO e de transmissão - TLAESTAC.

    Nas telas de atualização paginada, é necessário desenhar, após o campo de data/hora, se existente, os seguintes campos:

  • Um campo com 8 (oito) posições alfanuméricas para o código da transação de atualização paginada que será processada - sugere-se o campo TRACOD;

  • Um campo com 2 (duas) posições numéricas para exibir o número da página corrente - sugere-se o campo TLAPAGNUM;

  • Um campo com 2 (duas) posições numéricas para se informar o número da próxima página a ser atualizada - sugere-se o campo TLAPAGPROX;

  • Opcionalmente, após o campo TLAPAGNUM, pode-se ter um campo "ACAO" para tratamento da navegação, com 69 posições, input e opcional - sugere-se o campo TLAACAO.
    •

    Deve-se salientar que, em uma tela de atualização paginada, não pode existir campos de entrada de dados (input) entre o campo do código da transação - TRACOD - e o campo do número da página corrente - TLAPAGNUM.

    Nas telas de seleção de dados, é necessário desenhar, antes de cada item selecionável, um campo de input alfanumérico de 1 (uma) posição - TLAITESEL. Este campo servirá para o usuário indicar ou não a seleção do respectivo item.

    Caso a seleção dos dados seja apresentada em mais de uma página - Seleção com paginação (sem utilizar o paginador de entrada), deve-se também definir um campo de input alfanumérico de 1 (uma) posição - TLACONTPAG, para que o usuário possa indicar o seu desejo de continuar a sua seleção de dados. Este campo pode assumir os valores: "S" (Continua na próxima página), "R" (Retorna à pagina anterior) ou "N" (Não continua a paginação, encadeando a próxima transação, se houver, ou retornando ao menu).

    Nas telas de consulta utilizadas em programas com lógica de consulta com encadeamento, caso os itens sejam apresentados em mais de uma página - Consulta encadeada com paginação (sem utilizar o paginador de entrada), deve-se também definir um campo alfanumérico de 1 (uma) posição - TLACONTPAG, para que o usuário possa indicar o seu desejo de continuar a sua consulta aos dados. Este campo pode assumir os valores: "S" (Continua na próxima página), "R" (Retorna à pagina anterior) ou "N" (Não continua a paginação, encadeando a próxima transação, se houver, ou retornando ao menu).

    O desenho de uma tela tem um variado número de utilizações, quais sejam:

  • Pelos programas aplicativos, para coletar ou exibir dados;

  • Pela navegação, para a composição dos contextos;

  • Pela tela, no processo de geração de sua library Cobol;

  • Pelos hosts, para apresentação das telas de explicação de argumento;

  • Pelo sistema de controle de processamento SCAP/SPO, para captação de parâmetros para jobs.

    CAMPOS A SALTAR - Número de campos desprotegidos a serem saltados pelo cursor no momento do envio da tela para o terminal. Os campos a saltar são contados a partir do trancode, inclusive. Os campos protegidos transmissíveis não são contados para posicionamento do cursor.

    <Trancode da tela> - Código de transação derivado do código da tela a ser desenhada. Este é o único campo obrigatório no desenho de uma tela, tendo que ocupar o primeiro dos seus campos e iniciar na primeira ou segunda posição da mesma. No momento em que uma inclusão de tela é solicitada, o editor já coloca o trancode na posição inicial da mesma, ficando o seu atributo de vídeo livre para alterações.

    O restante da tela é uma área livre, com 23 linhas de 80 colunas cada, disponível para a definição dos campos e valores constantes que irão compor o desenho da mesma.Uma ou duas linhas do final da tela devem ser reservadas para as mensagens que serão enviadas pelos aplicativos do sistema. O mesmo número de linhas tem de ser reservado para todas as telas que forem tratadas pelo mesmo host, tendo, ainda, de ser especificado na definição deste host.

    Definição de tela

    A definição de uma tela no repositório de dados do SADS consiste na identificação do seu nome e do tipo de processamento ao qual ela se destina.

    Somente telas de menu e de processamento devem ser definidas no repositório de dados do SADS. Esta definição não é permitida para telas de explicação de argumento.

    A definição de uma tela é utilizada pelo SADS na geração da sua library Cobol e na consistência dos recursos definidos para os programas aplicativos, onde é necessário que os programas on-line tenham ao menos uma tela de tipo compatível com a lógica do programa.

    Para definir uma tela, é necessário que seu sistema já tenha sido definido no repositório de dados do SADS.

    CAMPOS DA TELA

    Tela - Código da tela no padrão SADS (sssTnnmm).

    A padronização utilizada para as telas encontra-se descrita no tópico "Padronização", explicado anteriormente, neste mesmo item.

    Nome - Descrição resumida da tela.

    O nome tem o objetivo de identificar e explicar, de maneira reduzida, a que se refere a tela.

    Tipo - Tipo de processamento a que se destina a tela.

    Os tipos de telas possíveis, de acordo com o processamento a que ela se destina, são os seguintes:

    "A" (Atualização simples);

    "P" (Atualização paginada);

    "C" (Consulta);

    "M" (Menu);

    "S" (Seleção).

    Operacao - Operação efetuada em relação à definição da tela.

    As operações possíveis são as seguintes:

    "INC" (Inclusão);

    "ALT" (Alteração);

    "EXC" (Exclusão);

    "CON" (Consulta).

    Cadastramento de estrutura de tela

    A estrutura é o conjunto de atributos e/ou campos que constituem as informações variáveis da tela, ou seja, os campos de entrada de dados (input) e os campos de saída de dados (output).

    A estrutura de dados de uma tela tem um variado número de utilizações, quais sejam:

    -Geração de sua library Cobol;

  • Movimentação de campos, na geração de programas;
  • Geração de estrutura de consistência geral;
  • Conversão de estrutura, através das regras, no módulo de navegação;
  • Geração de layouts de telas de sistemas cooperativos, transação esta que evita recompilações desnecessárias de programas quando os desenhos das telas são simplesmente modificados sem haver inclusão ou exclusão de campos e sem que eles tenham suas posições invertidas na tela.
    •

    No momento da inclusão da estrutura de uma tela, alguns campos já vêm preenchidos, de acordo com o tipo da tela, a saber:

  • Para as telas com tipo "A" (atualização), "C" (consulta), "M"(menu) e "S" (seleção), já vem preenchidos os seguintes campos: TLATRS (código da transação) e TLADTH (Data dd/mm/aa e hora hh:mm:ss da transação);
  • Para as telas com tipo "P" (atualização paginada) já vem preenchidos os seguintes campos: TLATRS (código da transação do paginador), TLADTH (Data dd/mm/aa e hora hh:mm:ss da transação), TRACOD (código da transação) e TLAPAGNUM (número da página).
    •

    Caso o desenho da tela não possua data/hora, o campo TLADTH deve ser eliminado de sua estrutura.

    Caso a data possua ano com 4 posições (dd/mm/aaaa), o campo TLADTH deve ser substituído pelo campo TLADTHC (dd/mm/aaaa hh:mm:ss) em sua estrutura.

    Para efetuar o cadastramento da estrutura de uma tela, é necessário que a tela e todos os atributos e campos que irão compor a sua estrutura tenham sido definidos no repositório de dados.

    CAMPOS DA TELA

    Tela - Código da tela no padrão SADS (sssTnnmm).

    Para cada um dos atributos e campos que compõem a estrutura da tela, tem-se as seguintes informações:

    Seq - Número para posicionamento do atributo ou campo na estrutura da tela.

    O primeiro número é "0010" e os demais, acrescidos de 10.

    Para incluir um atributo ou campo entre dois outros já incluídos, é necessário preencher uma linha que esteja em branco e alterar o número seqüencial desta linha para um número entre os já informados. O SADS reorganizará automaticamente a estrutura. Na próxima operação, os atributos ou campos já estarão na nova ordem.

    Atributo - Mnemônico do atributo ou campo.

    Para telas de sistemas não cooperativos que possuam constantes repetitivas, ou seja, dentro de ocorrências, é necessário definir, na posição relativa à constante, o atributo TLACTE como se fosse um campo do grupo de ocorrências.

    Existem atributos e campos, definidos no repositório de dados do SADS para uso genérico dos sistemas aplicativos - definidos no sistema "SBC" - e que são geralmente utilizados em telas, quais sejam:.

    EMPSIGLA - Atributo para apresentação da sigla da empresa na qual o usuário efetuou o logon;

    TLAACAO - Campo para tratamento das ações da navegação;

    TLACONFIRME - Campo para confirmação de transação;

    TLACONT - Campo para indicação de continuação;

    TLACTE - Campo para indicação de constante em ocorrências de tela;

    TLADET - Campo para composição de literais genéricos;

    TLADTH - Campo para edição da data e hora da apresentação da tela (dd/mm/aa hh:mm:ss);

    TLADTHC - Campo para edição da data e hora da apresentação da tela (dd/mm/aaaa hh:mm:ss);

    TLAESCOLHA - Campo para tratamento de escolhas em menus de navegação;

    TLAESTAC - Campo de posicionamento do cursor para transmissão no final da tela;

    TLAETX - Campo de posicionamento do cursor para transmissão do trancode;

    TLAIMP - Campo para informação de impressora para programas de emissão remota;

    TLAIMPRE - Campo para informação de impressora para programas de emissão remota;

    TLAMSG - Campo para apresentação de mensagem no corpo da tela;

    TLAOPCAO - Campo para indicação de opção em menus;

    TLAOPER - Campo para indicação de operação;

    TLAPAGNUM - Número da página corrente em telas paginadas;

    TLAPAGOPER - Código da operação para telas paginadas;

    TLAPAGPROX - Campo para indicação da próxima página em telas paginadas;

    TLATOT - Campo para apresentação de totais;

    TLATRS - Campo para apresentação do trancode (código de transação).

    Nivel - Nível de estruturação do atributo ou campo (análogo ao nível do Cobol).

    O nível inicial é 10. Se for necessário criar agrupamentos de atributos/campos, pode-se usar múltiplos de 5 (15, 20, 25, ...) de forma análoga ao Cobol.

    Ndef - Nível de definição do atributo ou campo no repositório de dados.

    Podem ser informados os seguintes valores:

    "S" - Se o atributo está definido no sistema informado no campo SIS ou no mesmo sistema da tela;

    "G" - Se o atributo está definido no SADS para utilização genérica. A relação dos atributos genéricos geralmente utilizados encontra-se na explicação do campo ATRIBUTO, descrito anteriormente.

    Se este campo não for informado, será assumido pelo gerador de library de tela que ele é um item grupo.

    Sis - Código do sistema ao qual o atributo ou campo pertence.

    O sistema só deve ser informado se o campo NDEF estiver preenchido com "S" e o atributo ou campo pertencer a um sistema diferente do sistema da tela.

    Input - Identificador de atributo ou campo de entrada.

    Podem ser informados os seguintes valores:

    "S" - Para atributo ou campo de entrada e saída (input e output);

    "A" - Para atributo ou campo de entrada e saída (input e output) que inicia um grupo opcional, na consistência da tela;

    "F" - Para atributo ou campo de entrada e saída (input e output) que finaliza um grupo opcional, na consistência da tela.

    Se este campo não for informado, o SADS assumirá que o atributo ou campo é somente de saída (output).

    Opc - Identificador de opcionalidade de preenchimento do atributo ou campo, no momento de entrada de dados.

    Podem ser informados os seguintes valores:

    "S" - Opcional;

    "Z" - Opcional. Assume valor 0 (zero), caso o campo não seja informado pelo usuário (válido somente para atributos ou campos numéricos);

    Se este campo não for informado, o SADS assumirá que é obrigatória a informação do atributo ou campo no momento da entrada de dados na tela.

    Tam - Tamanho do atributo ou campo no desenho da tela.

    O tamanho do atributo ou campo somente deverá ser informado se, na tela, estiver diferente daquele definido para o atributo ou campo no repositório de dados.

    Este campo não pode ser informado, caso tenha sido parametrizado na tabela SBPM5000 que os atributos/campos não podem ter seus tamanhos redefinidos nas estruturas nas quais são incluídos. Somente se o atributo for uma variável definida - Sintaxe = "VD" - é que este campo poderá ser informado, caso a parametrização tenha sido cadastrada.

    Maiores detalhes a respeito desta parametrização, deve-se consultar a Seção V, "Parametrizações do SADS", item "Atributo", subitem "Redefinição em estruturas".

    Sin - Critério para geração de sinal do atributo ou campo na library da tela.

    Podem ser informados os seguintes valores:

    "S" - Atributo ou campo numérico sinalizado (sem decimais);

    "N" - Atributo ou campo sem sinal e decimais;

    "A" - Atributo ou campo com sinal e decimais;

    "D" - Atributo ou campo com decimais (sem sinal);

    "B" - Atributo ou campo com BLANK WHEN ZERO.

    Este campo somente deverá ser informado quando for necessário utilizar sinalização para o atributo na library da tela, ou a quantidade de casas decimais necessária na tela for diferente daquela informada na definição do atributo ou campo no repositório de dados.

    Se for informado "D" ou "A", a quantidade de decimais deve ser informada no campo DEC.

    Dec - Número de casas decimais do atributo ou campo.

    Este campo somente deverá ser informado se a quantidade de decimais da tela for diferente da que está definida para o atributo ou campo no repositório de dados. Neste caso, deve-se informar "D" ou "A" no campo SIN.

    Occurs - Número de ocorrências de um item de grupo.

    Não pode ser utilizado em itens elementares.

    Continua (S/N) - Indicador de continuidade de operação.

    As possibilidades de preenchimento deste campo são as seguintes:

    "S" - Nova tela será aberta para prosseguir a operação com a estrutura da tela;

    "N" - Última tela da operação com a estrutura da tela.

    Operacao - Operação efetuada em relação à estrutura da tela.

    As operações possíveis são as seguintes:

    "INC" (Inclusão);

    "ALT" (Alteração);

    "EXC" (Exclusão);

    "CON" (Consulta).

    Geração de library de tela

    A geração de library de tela compreende a criacão de um arquivo em linguagem Cobol com os atributos e campos que a compõem.

    As libraries são geradas para telas de processamento e de menus de opção. Portanto, as telas para explicação de argumento, parâmetros de jobs, menus de escolha e menus de operação não possuem library.

    A library de uma tela é gerada a partir do seu desenho, da sua definição, da sua estrutura de dados e da definição dos atributos e campos que a compõem. Depende, ainda, se seu sistema é do tipo cooperativo ou não, isto é, se utiliza ou não o software COOPERATOR da MSA-INFOR.

    Sua codificação é derivada do código da tela, como no exemplo a seguir:

    Tela : SXYT0123
    Library : SX/Y/TL/01/23

    Após a sua geração, a library será utilizada pelos compiladores Cobol, devendo ser inserida na WORKING-STORAGE SECTION do programa através do comando COPY, conforme o exemplo a seguir:

    COPY "SX/Y/TL/01/23".

    A library, geralmente, é formada por duas estruturas básicas que se redefinem mutuamente, são elas:

  • Input - Onde estão declarados os campos transmissíveis da tela, isto é, os campos utilizados para entrada de dados.

    Os prefixos dos campos que compõem a área de input são formados do seguinte modo:

    Tela : SXYT0123
    Prefixo de     input : YT23I-

    Portanto, o prefixo é formado pela terceira letra do sistema (subsistema) somada à constante "T" (tela), ao número da tela e à constante "I" (input).

    Exemplo da área de input:

    05   YT23I-IN.
        10    YT23I-TLATRS.
            15    YT23I-TLATRS-RED PIC 9(08).
        10    YT23I-FUNCOD.
            15    YT23I-FUNCOD-RED PIC 9(05).
        10    YT23I-FUNNOM.
            15    YT23I-FUNNOM-RED PIC X(40).
        10 YT23I-FUNDATNASC.
            15    YT23I-FUNDATNASC-RED PIC 9(06).

    Os campos com até 23 bytes são sempre alfanuméricos com redefinição numérica.

  • Output - Onde encontram-se declarados todos os campos da tela, de entrada e de saída, com suas pictures de edição.

    Os prefixos dos campos que compõem a área de output são formados do seguinte modo:

    Tela : SXYT0124
    Prefixo de     output : YT24O-

    Portanto, o prefixo é formado pela terceira letra do sistema (subsistema) somada à constante "T" (tela), ao número da tela e à constante "O" (output).

    Exemplo da área de output para sistemas não cooperativos :
       05    YT24O-OUT.
        10    FILLER PIC X(05).
        10    YT24O-TLATRS
           15    YT24O-TLATRS-RED PIC 9(08).
        10    FILLER PIC X(50).
        10    YT24O-TLADTH.
           15    YT24O-TLADTH-RED PIC 9(18).
        10    FILLER PIC X(05).
        10    YT24O-FUNCOD.
           15    YT24O-FUNCOD-RED PIC 9(05).
        10    FILLER PIC X(05).
        10    YT24O-FUNNOM.
           15 YT24O-FUNNOM-RED PIC X(40).
        10    FILLER PIC X(05).
        10   YT24O-FUNDATNASC.
           15    YT24O-FUNDATNASC-RED PIC 99/99/99.

    Os campos com até 23 bytes são sempre alfanuméricos com redefinição numérica.

    Os "FILLER" que aparecem na library são relativos a caracteres constantes e/ou de posicionamento.

    Exemplo da área de output para sistemas cooperativos :
       05    YT24O-OUT.
         10    YT24O-TLATRS.
           15   YT24O-TLATRS-RED PIC 9(08).
         10    YT24O-TLADTH.
           15   YT24O-TLADTH-RED PIC 9(18).
         10    YT24O-FUNCOD.
           15   YT24O-FUNCOD-RED PIC 9(05).
         10    YT24O-FUNNOM.
           15   YT24O-FUNNOM-RED PIC X(40).
         10    YT24O-FUNDATNASC.
           15 YT24O-FUNDATNASC-RED PIC 99/99/99.

    Os campos com até 23 bytes são sempre alfanuméricos com redefinição numérica.

    Para telas de entrada paginada (para programas que utilizam o módulo de paginação de entrada), é gerada, ainda, a seguinte library:

  • Work - Onde encontram-se declarados os campos transmissíveis da tela (campos de input) para utilização na WORKING-STORAGE SECTION dos programas de atualização paginada.

    Os prefixos dos campos que compõem a área de work são formados do seguinte modo:

    Tela : SXYT0125
    Prefixo de     work : YT25W-

    Portanto, o prefixo é formado pela terceira letra do sistema (subsistema) adicionada à constante "T" (tela), ao número da tela e à constante "W" (work).

    Exemplo de área de work :
       05    YT25W-WORK.
          10   YT25W-TLATRS.
            15   YT25W-TLATRS-RED PIC 9(08).
          10   YT25W-TLAPAGNUM.
            15   YT25W-TLAPAGNUM-RED PIC 9(18).
          10   YT25W-FUNCOD.
            15   YT25W-FUNCOD-RED PIC 9(05).
          10   YT25W-FUNNOM.
            15   YT25W-FUNNOM-RED PIC X(40).
          10   YT25W-FUNDATNASC.
            15   YT25W-FUNDATNASC-RED PIC 9(06).

    Os campos com até 23 bytes são sempre alfanuméricos com redefinição numérica.

    A library da tela é gerada no usercode e disco definidos para o seu usercode de análise e de acordo com o ambiente escolhido.

    O usercode deverá estar cadastrado na Tabela de Usercodes do SADS - SBPM5002. Para maiores informações, consulte Seção V, "Parametrizações do SADS", neste manual.

    A geração de library de tela não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu GERTEL, descritos a seguir.

    •

    CAMPOS DA TELA

    Tela - Código da tela no padrão SADS (sssTnnmm).

    Ambiente - Ambiente em que a library será gerada.

    Podem ser informados os seguintes ambientes:

    "T" (teste) - Para gerar a library para o ambiente de teste;

    "P" (produção) - Para gerar a library para o ambiente de produção.

    Só é possível gerar uma library para produção, quando esta não existir em teste, caso contrário deve ser feita a sua liberação.

    Caso o ambiente não seja informado, será assumido o ambiente default do usuário. Detalhes sobre o cadastramento do ambiente default do usuário podem ser obtidos na seção VI, "Sistema de Segurança", item "Usuários e habilitações".

    Usercode - Usercode de análise.

    O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

    Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

    Para os usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes as libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, cuja manutenção é feita pelo responsável pelo SADS na sua instalação. Esta tabela encontra-se descrita na seção V, "Parametrizações do SADS", neste manual.

    Neste momento, o SADS compatibiliza as informações constantes no repositório de dados com as informações do desenho da tela, como por exemplo, pelo tamanho de cada um de seus campos. Caso haja incompatibilidade entre o desenho e o repositório de dados, uma mensagem é enviada notificando o erro ocorrido, caso contrário, a mensagem apenas confirma o sucesso da operação.

    Geração de consistência de tela

    A consistência geral de uma tela pode ser cadastrada manualmente ou gerada automaticamente a partir da estrutura da tela.

    Para efetuar a geração da consistência de uma tela, é necessário que a sua estrutura já esteja cadastrada no repositório de dados.

    Detalhes a respeito da consistência de uma tela estão descritos no item "Consistência geral", nesta seção.

    Liberação de desenho de tela

    A liberação é o processo que torna os elementos das telas disponíveis para utilização pelos programas aplicativos num determinado ambiente.

    A liberação de desenho de tela consiste em tornar disponível um desenho novo ou alguma manutenção para os programas que a utilizam no ambiente a que ela se destina.

    Portanto, a liberação pode ocorrer tanto para um novo desenho de tela quanto para alguma manutenção efetuada em um desenho já liberado.

    A liberação do desenho da tela pode comportar-se de duas formas distintas, dependendo do ambiente a que se destina:

  • Teste -A partir do desenho com ESTADO = "D" ("em desenvolvimento") ou "M" ("em manutenção"), é criada ou substituída, se for o caso, a cópia com ESTADO = "T" ("em teste"), sendo então excluído o desenho original ("em manutenção" ou "em desenvolvimento");
  • Produção - A partir do desenho com ESTADO = "T" ("em teste"), é criada ou substituída, se for o caso, a cópia com ESTADO = "P" ("em produção"), sendo mantido o desenho original ("em teste").

    A transação de liberação de tela para produção dispara, automaticamente, a liberação de consistência para produção, que, por sua vez, somente será liberada se tiver status igual a "a liberar para produção".

    A liberação de telas para produção torna-as disponíveis para utilização em ambiente de produção no mesmo computador onde foram liberadas. Para torná-las disponíveis para outro computador, é necessário executar o job SD/D/JB/70 no computador de origem, copiar os arquivos SS/S/AF/70 e SS/S/AF/72 para o computador de produção e nele executar o job SD/D/JB/72.

    Neste job, como as consistências liberadas para produção são também extraídas, o arquivo SS/C/AF/71 deve ser copiado para possibilitar a compatibilização das consistências em relação às suas telas, caso estas também tenham de ser liberadas para o computador de produção.

    •

    Se alguma tela ou consistência tiver sido liberada indevidamente para produção e se houver um computador diferente para processamento dos sistemas em produção, sua transferência para a máquina de produção pode ser cancelada através do menu CANTRAPRO. Detalhes a este respeito estão descritos na seção III, "Funções de Apoio", item "Utilitários on-line", neste manual.

    A liberação de desenho de tela não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu "LIBDESTEL", conforme descrito a seguir.

    CAMPOS DA TELA

    Tela - Código da tela no padrão SADS (ssstnnmm).

    Ambiente - Ambiente para o qual o desenho da tela será liberado.

    Podem ser informados os seguintes ambientes:

    "T" (Teste) - Para liberar o desenho da tela para o ambiente de teste;

    "P" (Produção) - Para liberar o desenho da tela para o ambiente de produção.

    Liberação de library de tela

    A liberação de library para produção consiste em gerar uma cópia da mesma no usercode de produção, excluindo-a do usercode de teste.

    A liberação de library de tela para produção não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu "LIBLIBTEL", conforme descrito a seguir.

    CAMPOS DA TELA

    Tela - Código da tela no padrão SADS (sssTnnmm).

    Usercode - Usercode de análise.

    O usercode a ser informado deve estar habilitado ao usuário, como elemento, em sua lista de elementos "SEGUSER", no Sistema de Segurança. Detalhes sobre este cadastramento estão descritos na Seção VI, "Sistema de Segurança", item "Habilitações de usuários SADS".

    Caso o usercode não seja informado, será assumido o usercode default do usuário. Detalhes sobre o cadastramento do usercode default do usuário podem ser obtidos na Seção VI, "Sistema de Segurança", item "Usuários e habilitações".

    Para usercodes de análise, são definidos os ambientes (usercode e disco) onde ficarão residentes suas libraries de teste e produção, sendo possível que as libraries de produção fiquem sem usercodes. Estas definições estão na tabela SBPM5002, que se encontra descrita na Seção V, "Parametrizações do SADS".

    Se na definição do usercode de análise na tabela SBPM5002 estiver indicado que as libraries de produção ficarão sem usercode, ao liberá-las, a versão de teste será removida. Esta tabela encontra-se descrita na Seção V, "Parametrizações do SADS", neste manual.

    Liberação de consistência de tela

    A liberação de consistência de tela visa tornar disponível uma consistência nova ou alguma manutenção efetuada na consistência da tela para o ambiente onde será utilizada.

    A liberação de consistência da tela pode se comportar de duas formas diferentes, dependendo do ambiente a que se destina:

  • Teste - A consistência liberada estará disponível somente para os programas que foram compilados para o ambiente de teste.

  • Produção - A consistência liberada estará disponível somente para os programas que foram compilados para o ambiente de produção.

    A liberação para produção torna a consistência disponível para utilização no ambiente de produção no mesmo computador onde ela foi liberada. Para torná-la disponível em outro computador, é necessário executar o job SD/D/JB/70 no computador de origem, copiar o arquivo SS/C/AF/71 para o computador de produção e nele executar o job SD/D/JB/72.

    Neste job, como as telas liberadas para produção são também extraídas, os arquivos SS/S/AF/70 e SS/S/AF/72 devem ser copiados para possibilitar a compatibilização entre as telas e as consistências, caso estas também tenham de ser liberadas para o computador de produção.

    • Se alguma tela ou consistência tiver sido liberada indevidamente para produção e se houver um computador diferente para processamento dos sistemas em produção, sua transferência para a máquina de produção pode ser cancelada através do menu CANTRAPRO. Detalhes a esse respeito estão descritos na seção III, "Funções de Apoio", neste manual.

    A liberação de consistência de tela não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu "LIBCONTEL" ou "LIBCON", conforme descrito a seguir.

    CAMPOS DA TELA

    Consistencia - Código da consistência no padrão SADS (sssCnnmm).

    Ambiente - Ambiente para o qual a consistência será liberada.

    Podem ser informados os seguintes ambientes:

    "T" (Teste) - Para liberar a consistência para o ambiente de teste;

    "P" (Produção) - Para liberar a consistência para o ambiente de produção.

    Não é possível liberar uma consistência de tela em manutenção diretamente para o ambiente de produção. Para isto, é necessário efetuar antes a sua liberação para teste.

    Funções de apoio

    Específicas de telas

    Para manipulação do desenho de uma tela, estão disponíveis as seguintes funções de apoio:

  • Duplicação de desenho da tela;

  • Lista de telas desenhadas;
    •

    A seguir, são apresentados detalhes sobre cada uma dessas funções:

    Duplicação de desenho de tela

    A duplicação de desenho de tela consiste na cópia do desenho de uma tela com outro código, sendo este incluído com ESTADO = "D" (em desenvolvimento).

    A duplicação de tela é utilizada quando existe a necessidade de se criar um desenho de tela similar a um já existente. Neste caso, basta duplicar o desenho já existente e processar a alteração da tela recém-criada.

    Para efetuar a duplicação de desenho de tela, é necessário que o desenho da tela a ser duplicado já esteja cadastrado no repositório de dados.

    Para disparar a duplicação, deve-se primeiramente transmitir os argumentos do menu DESTEL, informando "DUP" no campo "Operacao".

    CAMPOS DA TELA

    NOME - Código da tela cujo desenho será duplicado.

    Este código deve ser informado no padrão SADS (sssTnnmm, sssEnnmm ou sssJxnmm).

    ESTADO - Estado da tela cujo desenho será duplicado.

    Podem ser informados os seguintes valores:

    "D" (Desenvolvimento);

    "M" (Manutenção);

    "P" (Produção);

    "T" (Teste).

    DUPLICAR COM O NOME DE - Código da nova tela que será criada.

    Este código deve ser informado no padrão SADS (sssTnnmm, sssEnnmm ou sssJxnmm).

    Lista de telas desenhadas

    A lista de telas cadastradas é uma consulta paginada que, a partir de um determinado argumento, exibe a relação de telas cadastradas num estado específico.

    Este argumento pode ser um código de tela ou parte dele, devendo ter pelo menos quatro caracteres, ou seja, o sistema mais a constante de tela "T", "E" ou "J".

    A consulta de telas desenhadas não está associada a nenhuma tela de processamento. Portanto, para efetuá-la, basta transmitir os argumentos do menu "DESTEL", informando "LIS".
